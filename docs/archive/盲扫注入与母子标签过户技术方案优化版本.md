# 盲扫注入与母子标签过户技术方案（优化版本）

## 重新理解：母卡的三重价值

### 1. 使用权与归属权分离（核心安全逻辑）

| 概念 | 说明 |
| --- | --- |
| 使用权 | 拿着实物 = 可以用、可以展示 |
| 归属权 | 拿着母卡 = 可以过户、可以处置 |

**场景示例：** 把一块表借给朋友戴（使用权转移），但朋友无法卖掉（归属权仍在我手里），因为母卡在我这里。

### 2. 品牌交付仪式感（商业价值）

> 高端品牌 = 实物 + 证书 + 母卡

- 劳力士 = 表 + 保卡 + 证书
- 茅台 = 酒 + 防伪标 + 溯源码
- 我们的方案 = 实物 + 子标签 + 母卡（数字主权证书）

### 3. 品牌控制力延伸（战略价值）

> 母卡 = 品牌对资产的长期控制权

- 品牌可通过母卡追踪资产流转
- 品牌可在二级市场分成（过户必须用母卡）
- 品牌可提供"母卡托管服务"（增值收入）

---

## 核心思路

> **母卡必须存在，但不增加用户负担。**

## 一、母卡形态优化（降低成本 + 提升体验）

### 当前方案的问题

每个商品 = 1 个物理母卡（NFC 芯片），成本：2 个芯片 × 1000 商品 = 2000 个芯片

### 方案 A：虚拟母卡（默认）

品牌激活时自动生成虚拟母卡：

```http
POST /assets/activate
{
  "asset_id": "asset_001",
  "product_id": "product_001",
  "brand_id": "brand_001"
}
```

**后端逻辑：**

1. 派生子标签密钥：`K_chip_child`
2. 派生虚拟母卡密钥：`K_chip_mother`（存在数据库，不写入物理芯片）
3. 生成母卡凭证：QR Code / 数字证书
4. 绑定关系：`child_uid ↔ virtual_mother_id`

**虚拟母卡的三种形态：**

| 形态 | 说明 |
| --- | --- |
| 二维码 | 印在包装盒内，扫码后显示"数字主权证书"，含母卡 UID、品牌签名、资产 ID |
| App 内数字卡片 | 用户首次验真后自动导入，显示为"资产授权卡"，过户时调用，无需物理 NFC |
| 加密文件（高端版） | 品牌方生成加密 PDF，含母卡私钥、品牌签名，用户下载到手机，过户时解密使用 |

**优势：** 成本从 2N 降到 N、用户无需携带物理母卡、密码学强度相同

### 方案 B：物理母卡（高端可选）

| 商品价值 | 母卡形态 |
| --- | --- |
| < 5,000 RMB | 虚拟母卡（二维码） |
| 5,000 - 50,000 RMB | 虚拟母卡（App 数字卡） |
| > 50,000 RMB | 物理母卡（NFC 芯片 + 精美卡片） |

**物理母卡仪式感设计：**

- 精美卡套（类似信用卡黑卡）
- 烫金 Logo + 资产编号
- NFC 芯片内嵌
- 随商品附赠或单独寄送（增强仪式感）

## 二、过户流程优化（保留母卡，简化操作）

> 核心设计：母卡必须存在，但用户无感知

### 场景 1：日常使用（无需母卡）

用户 B 扫描子标签 → 显示"该资产归属于用户 A"

**可以：** 查看资产信息、展示给别人、验证真伪
**不可以：** 过户、修改资产信息、申请售后

async function transfer() {

## 三、母卡遗失处理

### 方案 A：虚拟母卡（可恢复）

```typescript
async function recoverVirtualMotherCard() {
  // 1. 用户登录新手机
  // 2. 扫描子标签验证身份
  const childTag = await scanNFC()
  // 3. 生物识别
  const biometric = await authenticateBiometric()
  // 4. 后端验证身份后，重新下发虚拟母卡
  const virtualMotherCard = await api.post('/assets/recover-mother-card', {
    asset_id: assetId,
    child_tag: childTag,
    biometric_token: biometric.token
  })
  // 5. 保存到本地
  await saveToLocalStorage('mother_card_' + assetId, virtualMotherCard)
}
```

### 方案 B：物理母卡（人工审核）

1. 提交人工审核申请（购买凭证、身份证、手持子标签照片）
2. 审核员审核（24-48 小时）
3. 审核通过后，品牌方重新签发虚拟母卡
4. 品牌方可收取"母卡补发费"（增值服务）

## 四、品牌交付仪式感设计


### 低价值商品（< 5,000 RMB）

- **包装内容：** 商品本体（贴子标签）+ 包装盒内印刷二维码（虚拟母卡）+ 说明卡片
- **用户体验：** 收到商品 → 扫描包装盒内二维码 → 自动导入虚拟母卡到 App → 完成激活

### 中价值商品（5,000 - 50,000 RMB）

- **包装内容：** 商品本体（贴子标签）+ 精美卡片（印刷虚拟母卡二维码 + 品牌 Logo）+ 说明书
- **用户体验：** 收到商品 → 扫描卡片二维码 → 自动导入虚拟母卡到 App → 卡片可收藏（仪式感）

### 高价值商品（> 50,000 RMB）

- **包装内容：** 商品本体（贴子标签）+ 物理母卡（NFC 芯片 + 精美卡套）+ 证书（烫金 + 品牌签名）+ 说明书

- **母卡外观：**
  - 黑色金属卡片（类似信用卡黑卡）
  - 烫金品牌 Logo + 激光雕刻资产编号
  - NFC 芯片内嵌
  - 背面印刷 "Asset Sovereignty Card"
  - 独立卡套（皮质/金属）+ 品牌封蜡

- **用户体验：** 收到商品 → 收到独立寄送的母卡（增强仪式感）→ 首次验真时扫描母卡绑定 → 母卡永久保存（类似劳力士保卡）

## 六、安全性分析：虚拟母卡 vs 物理母卡

> 物理母卡的安全锚点是"硬件"：每次过户验证实实在在的 AES-128 CMAC 动态签名，即使黑客拿到用户手机和账号，没有物理卡绝对过不了户。

> 虚拟母卡的安全锚点是"账号身份"：密钥 `K_chip_mother` 存在数据库或 App 本地存储中，防伪和防重放的压力从芯片硬件转移到了 WebAuthn（生物识别）和 `go-iam` 的 Token 防伪上。

**结论：** 对于低中单价商品，用账号级安全置换 45% 硬件成本和极简体验，是极度聪明的商业决策；而把硬件级绝对防御留给 5 万以上高端资产，完美踩中了客户的心理预期。

## 七、数据库设计

### 推荐方案：统一表 + 类型字段 + 分层校验

**设计原则：**

1. 协议层必须对母卡形态无感知
2. 数据库设计必须支持形态演进
3. 安全校验逻辑必须分层

### 母卡凭证表（Authority Devices）

```sql
CREATE TABLE authority_devices (
    authority_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    authority_uid VARCHAR(32) NOT NULL UNIQUE,

    -- 母卡类型
    authority_type VARCHAR(32) NOT NULL CHECK (authority_type IN (
        'PHYSICAL_NFC',      -- 物理 NFC 芯片
        'VIRTUAL_QR',        -- 虚拟母卡（二维码）
        'VIRTUAL_APP',       -- 虚拟母卡（App 数字卡）
        'VIRTUAL_BIOMETRIC'  -- 虚拟母卡（纯生物识别，未来扩展）
    )),

    brand_id VARCHAR(64) NOT NULL,

    status VARCHAR(32) NOT NULL DEFAULT 'Active' CHECK (status IN (
        'Active', 'Suspended', 'Revoked', 'Replaced'
    )),

    key_epoch INTEGER NOT NULL DEFAULT 0,

    -- 物理卡专属字段（虚拟卡为 NULL）
    physical_chip_uid VARCHAR(14),
    last_known_ctr INTEGER,

    -- 虚拟卡专属字段（物理卡为 NULL）
    virtual_credential_hash VARCHAR(128),
    bound_user_id VARCHAR(64),

    -- 审计字段
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    created_by VARCHAR(64) NOT NULL,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    revoked_at TIMESTAMPTZ,
    revoked_reason TEXT,

    CONSTRAINT fk_brand FOREIGN KEY (brand_id) REFERENCES brands(brand_id),
    CONSTRAINT fk_user FOREIGN KEY (bound_user_id) REFERENCES users(user_id)
);

CREATE INDEX idx_authority_uid ON authority_devices(authority_uid);
CREATE INDEX idx_authority_brand ON authority_devices(brand_id);
CREATE INDEX idx_authority_user ON authority_devices(bound_user_id);
CREATE INDEX idx_authority_type ON authority_devices(authority_type);
```

## 九、分层校验逻辑

### `rc-core/src/authority_verification.rs`

```rust
/// 母卡授权证明（前端提交）
pub enum AuthorityProof {
    /// 物理卡证明（NFC 动态签名）
    Physical { uid: String, ctr: u32, cmac: Vec<u8> },
    /// 虚拟卡证明（账号身份 + 生物识别）
    Virtual { credential_token: String, biometric_token: String, user_id: String },
}

/// 母卡校验器（协议核心）
impl<K: Kms> AuthorityVerifier<K> {
    /// 根据母卡类型分发校验逻辑
    pub fn verify_authority(
        &self,
        authority: &AuthorityDevice,
        proof: &AuthorityProof,
    ) -> Result<AuthorityVerificationResult> {
        // 1. 检查母卡状态
        if authority.status != AuthorityStatus::Active {
            return Ok(AuthorityVerificationResult::AuthorityRevoked);
        }
        // 2. 根据类型分发
        match (&authority.authority_type, proof) {
            // 物理卡：硬件 CMAC 校验
            (AuthorityType::PhysicalNfc, AuthorityProof::Physical { uid, ctr, cmac }) => {
                self.verify_physical_authority(authority, uid, *ctr, cmac)
            }
            // 虚拟卡：账号身份校验
            (AuthorityType::VirtualQr | AuthorityType::VirtualApp,
             AuthorityProof::Virtual { credential_token, biometric_token, user_id }) => {
                self.verify_virtual_authority(authority, credential_token, biometric_token, user_id)
            }
            _ => Ok(AuthorityVerificationResult::InvalidProof),
        }
    }
}
```

**物理卡校验流程：**

1. UID 必须匹配
2. CTR 防重放（必须大于上次记录）
3. 派生母卡密钥
4. 验证 CMAC（硬件级安全）

**虚拟卡校验流程：**

1. 用户 ID 必须匹配绑定用户
2. 验证虚拟凭证 Token（Hash 比对）
3. 验证生物识别 Token（JWT 校验）
