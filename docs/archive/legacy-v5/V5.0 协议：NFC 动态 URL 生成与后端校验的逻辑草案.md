# 《V5.0 协议：NFC 动态 URL 生成与后端校验逻辑草案》

## 一、 硬件层：NTAG 424 DNA 配置

为了实现“免 App 验证”，芯片必须配置为 **NDEF 消息模式**，并将动态参数嵌入 URL 路径中。

### 1. 动态 URL 结构

每枚芯片在出厂“点睛”时，会被写入一个包含加密镜像的 URL：

`https://v5.auth/verify?uid=04A1B2C3D4E5F6&ctr=000005&cmac=8A3F...`

- **`uid`**: 芯片的全球唯一 7 字节序列号（明文）。
    
- **`ctr`**: 芯片内部的 3 字节 NFC 计数器。**每感应一次，数值 +1**。
    
- **`cmac`**: 核心安全码。由芯片使用内部密钥对 `uid` 和 `ctr` 进行 AES-128 计算生成的动态密文。
    

---

## 二、 后端层：动态校验逻辑 (The Zero-Trust Logic)

当用户手机感应并跳转至 H5 时，后端服务器需执行以下“三连检”：

### 1. 密钥匹配 (Key Mapping)

- **动作**：根据 URL 中的 `uid`，从数据库中提取该芯片对应的 **$K_{auth}$**（认证密钥）。
    
- **逻辑**：每一枚芯片的密钥都是唯一的。即使一枚芯片被破解，也不会威胁到整个协议库。
    

### 2. 计数器校验 (Anti-Replay)

- **动作**：对比 URL 中的 `ctr` 与数据库中记录的该芯片 `Last_CTR`。
    
- **判定**：
    
    - 如果 `ctr` $\le$ `Last_CTR`：判定为**重放攻击**（链接被录制或截图），直接显示“验证失效”。
        
    - 如果 `ctr` $>$ `Last_CTR`：更新数据库，进入下一步。
        

### 3. 算法验签 (CMAC Verification)

- **动作**：后端服务器使用相同的算法和 $K_{auth}$，对传入的 `uid` 和 `ctr` 重新进行一次 AES 计算。
    
- **判定**：
    
    - 计算结果与 URL 中的 `cmac` 完全一致 $\rightarrow$ **官方正品**。
        
    - 不一致 $\rightarrow$ **伪造芯片**。

## 三、 业务层：H5 状态机转换

一旦后端校验通过，H5 页面根据资产的生命周期进入不同的显示状态：

|**数据库状态**|**前端 H5 显示**|**交互逻辑**|
|---|---|---|
|**`IN_FACTORY`**|**[未激活]**|提示：该商品尚未出库，请联系品牌方。|
|**`ACTIVATED`**|**[正品认证] + [多维行价]**|显示资产详情，并弹出：**“确认交易？点击确权”**。|
|**`TRANSFERRED`**|**[已确权] + [持有人证书]**|显示当前合法持有人的部分脱敏信息（如：Mr. Zhang）。|

## 四、 核心安全防御：防“暴力破解”与“克隆”

1. **分级密钥体系**：
    
    - **$K_{meta}$**: 用于加密 UID（可选）。
        
    - **$K_{auth}$**: 用于生成动态 CMAC（核心）。
        
    - **$K_{file}$**: 用于读取私有资产数据（进阶）。
        
2. **暴力拆解保护**：
    
    如果后端检测到某个 `uid` 在短时间内出现大量 `ctr` 异常跳变，系统自动将该 `uid` 标记为 **`COMPROMISED` (已失陷)**，并向品牌方发出预警。
