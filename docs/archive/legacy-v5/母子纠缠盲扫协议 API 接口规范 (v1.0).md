**文档版本：** v1.0

**状态：** 草案 (Draft)

**安全等级：** 绝密 (Top Secret)

**核心原则：** 物理生产黑盒化、资产定义中心化。

---

## 一、 全局逻辑对象定义

在调用接口前，必须明确以下三个核心 ID 的定义：

1. **`C-UID` (Child UID)：** 子标签物理唯一标识，代表“物品实物”。
    
2. **`M-UID` (Mother UID)：** 母标签物理唯一标识，代表“所有权/定义权”。
    
3. **`SKU-ID` (Stock Keeping Unit)：** 品牌方定义的数字资产身份，代表“商品灵魂”。
    

---

## 二、 阶段一：工厂端 - 盲扫入库协议

该接口由工厂生产线调用。**禁止**在该阶段传入任何品牌信息。

### 1. 盲扫录入接口 (`/v1/factory/blind-log`)

- **调用方：** 工厂端授权设备
    
- **请求方式：** `POST`
    
- **请求参数：**
    



```JSON
{
  "factory_auth_key": "STRING", // 工厂设备授权令牌
  "c_uid_sun": "STRING",       // 子标签生成的动态加密SUN消息
  "timestamp": "LONG",         // 扫描时间戳
  "batch_no": "STRING"         // 工厂内部生产批次（可选）
}
```

- **平台逻辑：**
    
    1. 校验 `factory_auth_key` 合法性。
        
    2. 解密 `c_uid_sun` 提取 `C-UID`。
        
    3. 检查该 `C-UID` 是否已存在。
        
    4. **状态变更：** 将该标签标记为 `FACTORY_LOGGED` (已盲扫)。
        
- **返回参数：**
    



```JSON
{
  "code": 200,
  "msg": "LOG_SUCCESS", // 严禁返回商品名称
  "data": {
    "trace_id": "UUID"  // 仅返回流水号供工厂记录
  }
}
```

## 三、 阶段二：品牌总部 - 纠缠与定义协议

该接口由品牌方总部在商品上架前调用，执行“点睛”操作。

### 1. 资产纠缠激活接口 (`/v1/brand/entangle-active`)

- **调用方：** 品牌方总部/门店授权设备
    
- **请求方式：** `POST`
    
- **请求参数：**
    



```JSON
{
  "brand_auth_token": "STRING", // 品牌方高权限Token
  "m_uid_sun": "STRING",       // 母标签动态加密消息
  "c_uid_sun": "STRING",       // 子标签动态加密消息
  "target_sku_id": "STRING"    // 品牌方定义的 SKU ID（如：DIOR-2027-BAG-01）
}
```

- **平台逻辑：**
    
    1. 验证 `C-UID` 是否处于 `FACTORY_LOGGED` 状态（确保来源合法）。
        
    2. 验证 `M-UID` 是否为该品牌下发的合法母卡。
        
    3. **强绑定：** 在数据库中建立 `M-UID` <---> `C-UID` <---> `SKU-ID` 的三位一体映射。
        
    4. **状态变更：** 更新为 `ACTIVATED` (已激活)。
        
- **返回参数：**
    



```JSON
{
  "code": 200,
  "msg": "IDENTITY_GRANTED",
  "data": {
    "activation_time": "ISO8601",
    "asset_status": "LIVE"
  }
}
```

## 四、 阶段三：消费者端 - 动态解析协议

当终端用户扫描时，系统根据标签状态返回不同的内容。

### 1. 动态重定向接口 (`/v1/resolve/{sun_msg}`)

- **调用方：** 消费者移动端浏览器
    
- **请求方式：** `GET`
    
- **解析逻辑：**
    
    - **IF `Status == FACTORY_LOGGED`：** 物理件合法但未授权。
        
        - _返回：_ 品牌方预设的“生产中/认证中”占位页。
            
    - **IF `Status == ACTIVATED`：**
        
        - _返回：_ 302 重定向至品牌方 SKU 详情页，并携带加密的 `Verify_Token`。
            
    - **IF `Status == TAMPERED`：**
        
        - _返回：_ 红色报警页（物理防伪破损）。

## 五、 安全审计规则 (Security Audit)

1. **Anti-Replay (防重放)：** 每次调用必须校验 SUN 消息中的计数器（Counter），若计数值小于或等于历史值，判定为克隆攻击。
    
2. **Geo-Fencing (地理围栏)：** 盲扫接口必须记录工厂 IP 范围。若盲扫出现在非授权经纬度，系统自动标记为 `SUSPICIOUS`。
    
3. **Entropy Check (熵值校验)：** 若某一母卡在短时间内尝试纠缠大量不同的子标，立即封锁该 `M-UID`。

## 一、 全链路加密传输架构

我们要实现的是：**即便网关被监听，数据包被截获，攻击者也无法解密内容，更无法伪造指令。**

### 1. 通讯层：双向认证 (mTLS)

- **协议：** 工厂/总部设备与平台之间不使用简单的 HTTP，而是强制使用 **双向 TLS (Mutual TLS)**。
    
- **机制：** 每个授权终端（读写器）内置一个独一无二的设备证书。只有证书合法的设备，服务器才会握手。
    
- **价值：** 彻底杜绝非法设备模拟 API 调用。
    

### 2. 数据层：有效载荷加密 (Payload Encryption)

- **策略：** 采用 **AES-256-GCM** 对 API 的 `body` 进行二次加密。
    
- **过程：** 设备在发送前，利用**设备私钥**对数据进行签名，并利用**平台公钥**加密。平台收到后，通过 HSM 进行硬件解密。
    

---

## 二、 核心加密逻辑：KMS 与芯片的“密文交谈”

芯片与云端的通讯从不传输 UID 明文，而是传输**动态密文**。

### 1. 密钥派生算法 (Diversified Keys)

我们绝不在数据库存储每一颗芯片的原始密钥（Master Key），而是存储派生规则：

- **算法：** $K_{tag} = CMAC(K_{master}, UID \parallel System\_ID)$
    
- **逻辑：** 即使某一颗芯片被物理暴力拆解、密钥被提取，攻击者也无法推导出其他芯片或系统的总密钥。
    

### 2. SUN (Secure Unique NFC) 消息解构

芯片每次被扫码，都会在内部利用 AES-128 产生一段动态变化的密文。

$$URL = [https://v.com/resolve?picc_data=E(K](https://v.com/resolve?picc_data=E(K)_{sun}, UID \parallel Counter)$$

- **动态计数器 (Counter)：** 每次扫码自动加 1。
    
- **防重放逻辑：** 云端 KMS 会记录每个 UID 的当前计数器值。如果收到的密文解密后，Counter 值小于或等于历史值，系统判定为**克隆重放攻击**，直接封禁。

## 三、 绝密级 API 交互流程（修订版）

### 1. 工厂盲扫（全加密态）

JSON

```
{
  "header": {
    "device_sn": "FAC-778899",
    "signature": "BASE64_HMAC_SHA256" // 证明请求未被篡改
  },
  "payload": "ENCRYPTED_BLOB_VIA_SM2_OR_RSA" // 包含加密的 SUN 消息和流水号
}
```

### 2. 总部纠缠（双重密文对冲）

总部设备在执行纠缠时，平台会启动 **原子化事务 (Atomic Transaction)**：

1. **解密 Parent：** 确认母卡合法性。
    
2. **解密 Child：** 确认子标来源。
    
3. **内存对冲：** 在 HSM 的**安全内存 enclave** 中完成绑定逻辑，数据不落地。
    
4. **加密入库：** 绑定关系以加密形式存入数据库。
    

---

## 四、 硬件级的安全部署建议

1. **HSM (Hardware Security Module)：** * 所有的 Master Key 必须存储在物理硬件加密机中，任何人类（包括你和我）都无法通过控制台查看明文密钥。
    
2. **TEE (Trusted Execution Environment)：** * 云端解析引擎运行在信创环境或受信任执行环境（如 Intel SGX）。即使服务器操作系统被 Root，攻击者也无法读取内存中的解密逻辑。
    
3. **阅后即焚记录：** * 盲扫过程中的原始密文在验证成功后，仅保留哈希值用于审计，原始密文立即物理粉碎。
