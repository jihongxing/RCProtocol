首先要建立的是**“数字协议层”**。没有协议，硬件只是散沙。这套协议的核心在于：**物理 UID 与数字身份的完全解耦，直到母卡介入才完成“点睛”之击。**

---

## 阶段一：定义核心协议 (The V5.0 Protocol)

我们需要定义一套三端协同的 API 规范，确保数据在流转中始终保持黑盒状态。

### 1. 物理层标识协议 (Tag ID Protocol)

- **UID 规则：** 平台统一采用全局唯一标识符，芯片端通过 NXP 的 **SUN (Secure Unique NFC)** 功能，将 UID 与动态计数器加密为不可伪造的 URL。
    
- **黑盒特征：** 芯片预写入时，不包含任何关于品牌、价格、SKU 的明文信息。
    

### 2. 工厂端“盲扫入库”协议 (Blind-Entry API)

工厂设备只需向平台发送以下极简数据包：

- `Action`: `FACTORY_CHECK_IN`
    
- `Device_ID`: 工厂授权读写器 ID
    
- `Chip_Data`: 芯片传回的加密 SUN 密文（包含 UID）
    
- **平台返回：** `{ "status": "SUCCESS", "timestamp": "2026-03-11..." }`（**禁止返回任何商品名称**）。
    

### 3. 总部端“纠缠激活”协议 (Entanglement & Identity Granting API)

这是最具权力的接口，必须双因子验证：

- `Action`: `BONDING_AND_ACTIVATION`
    
- `Parent_Data`: 母卡加密密文
    
- `Child_Data`: 物品子标加密密文
    
- `Target_SKU_ID`: 品牌方指定的秘密 SKU 编号
    
- **逻辑：** 平台校验子标是否曾由合法工厂盲扫 $\rightarrow$ 校验母标是否由总部持有 $\rightarrow$ 执行纠缠绑定 $\rightarrow$ 激活防伪与获客链接。

## 阶段二：平台架构搭建 (The Platform)

平台不只是一个数据库，而是一个**“异步验证机”**。

### 1. KMS 密钥管理中心

- **物理隔离：** 存储所有芯片的 Master Key。
    
- **派生逻辑：** 根据不同的订单任务，动态派生临时写入密钥给工厂。
    

### 2. 状态机调度引擎 (State Machine)

每一个标签在平台内部拥有五个严格的生命周期状态：

1. **PRE_MINTED (已生产)：** 标签已出厂，但尚未进入品牌供应链。
    
2. **FACTORY_LOGGED (已盲扫)：** 工厂已植入，物理来源合法，但无数字身份。
    
3. **BONDED (纠缠中)：** 母子标签已完成物理匹配，正在等待品牌发布指令。
    
4. **ACTIVATED (已激活)：** 商品已上架，扫码可看到品牌内容。
    
5. **TAMPERED (已破坏)：** 子标回路断裂，该资产在数字世界标记为“物理报废”。
    

### 3. 解耦的解析器 (Resolver)

- 当消费者扫码时，解析器根据状态机返回不同的前端界面（黑盒、正品、或报警）。

## 阶段三：品牌方管理后台 (Brand Dashboard)

这是品牌方行使“定义权”的工具。

- **SKU 库维护：** 品牌方在发布前，先在后台建立“秘密文件夹”，上传素材，但不向外界公开。
    
- **激活工具流：** 品牌仓库管理员通过该后台，将成堆的“工厂盲扫件”与“母卡”进行批量纠缠。

## 阶段四：首个 MVP 任务清单 (First Action List)

为了落地“地基”，我们需要在两周内完成以下动作：

1. **模拟黑盒接口：** 开发一个不带 UI 的后端 API，仅用于接收 UID 录入。
    
2. **母子配对演示：** 编写一段逻辑：当数据库同时检测到 `A (Child)` 和 `B (Parent)` 时，自动将状态由 `Blind` 更新为 `Live`。
    
3. **动态重定向测试：**
    
    - 状态 A (未激活) $\rightarrow$ 页面显示：“商品认证中...”
        
    - 状态 B (已激活) $\rightarrow$ 页面显示：“Dior 2027 绝密款 + 购买链接”
