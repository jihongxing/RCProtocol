## 1. 背景与设计目标

### 1.1 背景

当前二奢确权系统需要保证以下几点：

- **品牌方完全控制资产发行权**，杜绝代工或渠道乱发资产
    
- **平台可高效管理资产生命周期和二手流转状态**
    
- **保证资产数字化确权的安全与可审计性**
    

传统单一母卡模式或软件私钥单独管理模式存在局限：

| 模式   | 优点          | 局限                 |
| ---- | ----------- | ------------------ |
| 硬件母卡 | 品牌绝对控制、直观   | 扩展困难、管理复杂、离职/丢失风险高 |
| 软件私钥 | 高效自动化、可远程操作 | 品牌可能顾虑资产发行权被平台掌控   |

### 1.2 设计目标

1. **确保品牌方掌握资产发行权**
    
2. **允许平台管理日常资产状态操作**
    
3. **保证系统可扩展、可工业化落地**
    
4. **提供安全、可审计、透明的操作机制**
    

---

## 2. 核心设计

### 2.1 系统结构

#### 2.1.1 硬件母卡（Root Mother Card）

- **持有者**：品牌总部
    
- **功能**：
    
    - 唯一可生成新资产或母卡的签名权
        
    - 激活子标签资产
        
- **特点**：
    
    - 物理控制，无法被平台或第三方篡改
        
    - 保留对资产发行的绝对控制权
        
- **适用场景**：
    
    - 新款水桶包或限量款生产激活
        
    - 区域母卡生成或品牌总部分发母卡
        

#### 2.1.2 软件私钥（Delegated Operational Key）

- **持有者**：平台
    
- **功能**：
    
    - 管理已激活资产状态
        
        - 激活 / 冻结 / 转移 / 异常标记
            
    - 支持自动化操作与远程控制
        
- **限制**：
    
    - **无法生成未激活资产**
        
    - **无法激活或伪造创世母卡签名**
        
- **适用场景**：
    
    - 二手交易平台操作
        
    - 用户资产状态管理
        
    - 日常系统维护与批量处理
        

---

### 2.2 子母标签逻辑

|标签类型|功能|控制方|
|---|---|---|
|子标签|资产唯一标识|与物理资产绑定|
|母卡|权属激活控制|硬件母卡/软件私钥分权体系|

流程示意：

生产方生产资产 → 扫描子标签生成资产ID → 品牌创世母卡签名激活 → 平台软件私钥管理状态 → 用户交易/转移 → 状态可审计/冻结

---

### 2.3 权力分层设计原则

1. **发行权归品牌**：创世母卡物理掌握，任何新资产激活都需要其签名
    
2. **日常操作由平台**：软件私钥仅管理已激活资产状态
    
3. **审计透明**：所有操作有签名与时间戳，可随时由品牌方验证
    
4. **扩展性**：软件私钥支持高并发、远程操作、自动化系统（物流、仓储、线上商城）
    

---

## 3. 风险分析与对策

|风险|原因|对策|
|---|---|---|
|软件私钥被攻破|平台控制密钥|限制私钥权限，仅能操作已激活资产；引入多因素认证|
|创世母卡丢失或损坏|物理卡管理|采用多重备份母卡，严格保管流程|
|离职员工滥用权限|母卡或私钥|软件私钥可撤销，权限可定期刷新；创世母卡操作受审计|
|非法资产激活|系统漏洞|所有激活必须有创世母卡签名；系统进行签名验证|
|自动化操作失误|系统 bug 或异常|流程前置验证；操作日志记录；异常报警机制|

# 5.子母标签 + 创世母卡 + 软件私钥分权模型

```markdown
品牌总部（持有创世母卡）
│
├─ 权限：
│    ├─ 唯一可生成新资产/母卡
│    ├─ 唯一可激活子标签资产
│    └─ 可撤销/冻结区域母卡权限
│
└─ 下放权限给区域/门店（可选硬件或软件）
     │
     ├─ 区域母卡 / 区域软件私钥（品牌可管理）
     │    ├─ 管理区域范围内资产状态（已激活资产）
     │    ├─ 无法生成新资产
     │    └─ 可协助审计与异常处理
     │
     └─ 门店软件私钥（平台管理）
          ├─ 日常管理已激活资产状态：
          │     ├─ 激活/转移/冻结
          │     ├─ 异常标记（如违约、频率异常）
          │     └─ 可用于物流、二手交易等自动化操作
          ├─ 无法生成新资产
          └─ 所有操作记录签名 & 时间戳，品牌可审计

子标签（绑定物理资产，如水桶包）
│
├─ 唯一标识物理资产
├─ 被创世母卡签名激活后，状态可由软件私钥操作
└─ 资产状态：
      ├─ #已激活
      ├─ #冻结
      ├─ #转移中
      └─ #异常
```

           ▼

# 核心设计逻辑总结

1. **创世母卡** = 绝对资产发行权
    
2. **软件私钥** = 日常操作权（仅限已激活资产）
    
3. **子标签** = 资产唯一标识
    
4. **权力分离** = 品牌保持控制权，平台可高效管理与工业化落地
    
5. **安全与可审计** = 所有状态变更有签名与时间戳，防止滥用

我非常认可这种“混合动力”模式。物理母卡提供的是**“物理上的最终裁决权（Physical Root of Trust）”**，而软件私钥提供的是**“工业级的发行效率”**。

关于你担心的“品牌方不可能只有一张母卡”以及“管理复杂度”问题，我们必须对母卡的级别进行**“权力的原子化拆解”**。

### 一、 创世母卡体系：RC-Hierarchy (主权树) 设计

为了解决扩展性与安全性的矛盾，我建议参考 **PKI (公钥基础设施) 证书链** 的逻辑，设计一套“三级卡权架构”：

|**级别**|**名称**|**物理形式**|**权限范围**|**存放场景**|
|---|---|---|---|---|
|**L0**|**Genesis Root (创世根卡)**|极少数定制卡（多签）|签发/吊销下级 Master 权限|品牌全球总部保险柜 (Cold)|
|**L1**|**Master Key (区域/品类大班卡)**|高阶加密 NFC 卡|负责特定区域（如 APAC）或特定品类（如皮具）的资产激活|区域运营中心 (Warm)|
|**L2**|**Operational Tag (工位点睛卡)**|工业级读写设备/嵌入式模块|仅具备“单次资产绑定”权限，无权签发密钥|生产线/工厂工位 (Hot)|

---

### 二、 如何解决你提到的“局限性”？

#### 1. 针对“管理复杂与丢失风险”：**M-of-N 多签机制**

- **逻辑**：创世根卡不应该只有一张。我们可以设定为：品牌方持有 3 张根卡，任何重大的密钥重置或权限变更需要“至少 2 张根卡”同时靠近读写器才能生效。
    
- **效果**：哪怕一名高管离职或弄丢一张卡，帝国的根基依然稳固。
    

#### 2. 针对“平台掌控发行权”的顾虑：**分层密钥派生 (KDF)**

- **方案**：根密钥（Root Secret）的派生过程在品牌方的物理根卡中完成，`rc-kms` 仅持有经过品牌方授权后的**“派生临时私钥”**。
    
- **效果**：平台（你）只提供技术流水线，而不持有品牌方的“母本”。品牌方随时可以利用物理根卡单方面关闭平台权限。
    

---

### 三、 核心技术补丁：`rc-core` 状态机扩展

为了支持这套分级体系，我们在 `rc-core` 中需要对母卡（Master Card）的属性进行定义：

Rust

```
pub struct MasterCard {
    pub level: u8,            // 0: Root, 1: Regional, 2: Operational
    pub scope: Scope,         // 权限作用域：全品类 vs 单品类
    pub expiration: u64,      // 权限有效期
    pub multi_sig_threshold: u8, // 多签阈值
}
```

老钱们非常迷恋这种**“权力的级差”**。

- **想象一下**：爱马仕的全球 CEO 手里握着 L0 根卡，这代表着品牌的灵魂；各工厂厂长拿着 L1 大班卡；而流水线工人操作的是 L2 设备。
