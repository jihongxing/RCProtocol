## 一、逻辑关联 vs 物理纠缠

RC-Protocol 中"母子纠缠"不是云端数据库里两行 ID 的简单对应关系。如果仅靠数据库映射，那只能称为"关联"。我们称之为"纠缠"，是因为子母标签在算法层级和认证链路上构成了不可分割的密码学整体。

| 维度 | 逻辑关联（普通设计） | 物理纠缠（RC-Protocol） |
|------|----------------------|------------------------|
| 存储位置 | 仅云端数据库 | 云端 + 芯片 EEPROM |
| 验证前提 | 必须联网 | 支持离线密钥挑战 |
| 攻击成本 | 数据库篡改即失效 | 需同时克隆两枚硬件级别私钥（几乎不可能） |
| 绑定深度 | ID 对 ID | 密钥对密钥（Key-to-Key） |
| 断裂检测 | 依赖云端查询 | 芯片内部 50ms 内自主判定 |

核心洞察：物理纠缠的本质是**密钥派生链的交叉依赖**——子标签的密钥中混入了母标签的特征，母标签的认证流程中嵌入了子标签的状态。两者在密码学意义上"同根生"，任何一方被替换都会导致整条认证链路断裂。

## 二、三层技术锚定

### 2.1 第一层：密钥派生纠缠 (Diversified Key Entanglement)

子标签与母标签的密钥不是随机生成的，而是通过同一个 `Brand_Key` 以及彼此的关联特征派生出来的。

**纠缠派生公式**：

```
Brand_Key = HMAC-SHA256(Root_Key, Brand_ID ‖ System_ID)

# 母标签持有资产根密钥
K_root_asset = HMAC-SHA256(Brand_Key, M_UID ‖ Asset_Epoch)

# 子标签的芯片密钥，混入母标签 UID 的摘要
K_chip = HMAC-SHA256(Brand_Key, C_UID ‖ Epoch ‖ SHA-256(M_UID)[..8])[..16]

# 蜜标密钥（256-bit HMAC key）
K_honey = HMAC-SHA256(Brand_Key, Serial_BE)
```

**关键设计**：

- `K_chip` 的派生输入中包含 `SHA-256(M_UID)[..8]`（母标签 UID 的前 8 字节摘要）
- 这意味着子标签的密钥与特定母标签在数学上绑定
- 如果试图用"母标签 A"去处置"子标签 B"（即便两者属于同一品牌），派生出的会话密钥将无法通过校验

**技术后果**：

| 场景 | 结果 |
|------|------|
| 正确母标签 + 正确子标签 | 会话密钥匹配，认证通过 |
| 错误母标签 + 正确子标签 | 派生密钥不同，CMAC 校验失败 |
| 正确母标签 + 克隆子标签 | PUF 物理指纹不匹配，握手失败 |
| 两者均被克隆 | 计数器（CTR）回溯检测触发熔断 |

> **代码权威**：密钥派生实现以 `rc-core/src/crypto/kdf.rs` 为准。

### 2.2 第二层：离线挑战-响应链路 (Offline Mutual Authentication)

母标签不只是一个"读出来的 ID"，它具备离线认证子标签的能力。这是最核心的技术绑定。

**协议流程**：

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│  手机 App │         │ 母标签    │         │ 子标签    │
│  (中继)   │         │ (Key)    │         │ (Crown)  │
└────┬─────┘         └────┬─────┘         └────┬─────┘
     │                     │                     │
     │  1. 读取母标签       │                     │
     │ ──────────────────→ │                     │
     │                     │                     │
     │  2. 母标签生成       │                     │
     │     随机 Nonce       │                     │
     │ ←────────────────── │                     │
     │                     │                     │
     │  3. 将 Nonce 作为    │                     │
     │     Challenge 转发   │                     │
     │ ─────────────────────────────────────────→ │
     │                     │                     │
     │                     │    4. 子标签使用与母标签 │
     │                     │       绑定的密钥签名    │
     │                     │       Response          │
     │ ←───────────────────────────────────────── │
     │                     │                     │
     │  5. 验证签名         │                     │
     │     (可离线完成)     │                     │
     │                     │                     │
```

**关键约束**：

- 子标签必须在 **50ms** 内给出基于与母标签绑定密钥的正确响应
- 超时或响应错误 → 物理层面的"纠缠"被视为断裂
- 整个过程**不需要经过云端**，在断网环境下依然可以证明归属权
- 挑战值（Nonce）每次随机生成，防止重放攻击

**离线认证的安全保证**：

| 攻击向量 | 防御机制 |
|----------|----------|
| 重放旧的 Response | Nonce 每次随机，旧响应无效 |
| 中间人截获并转发 | 50ms 时间窗口极短，物理距离约束（NFC 感应范围 1-3cm） |
| 伪造子标签响应 | 需要芯片内部的绑定密钥，无法从外部提取 |
| 伪造母标签 Nonce | 母标签的 Nonce 由硬件 RNG 生成，不可预测 |

> **代码权威**：EV2 认证握手以 `rc-core/src/crypto/ev2_auth.rs` 为准。

### 2.3 第三层：动态 URL 的影子绑定 (SDM Shadow Binding)

在子标签的 SUN (Secure Dynamic Messaging) 格式中，母标签的身份被作为"影子参数"混入 CMAC 计算。

**技术实现**：

```
# 子标签 SDM 配置时，将母标签 UID 的单向哈希混入 CMAC 计算
Shadow_Param = SHA-256(M_UID)[..4]    # 母标签 UID 摘要的前 4 字节

# CMAC 计算素材中包含影子参数
SDM_Vector = C_UID ‖ CTR ‖ Shadow_Param
CMAC = AES-128-CMAC(K2, SDM_Vector)

# 最终 SUN URL
https://rc-protocol.com/v?u={C_UID_HEX}&c={CTR_HEX}&m={CMAC_HEX}
```

**验证逻辑**：

1. 后台收到子标签的动态 URL
2. 从数据库检索该子标签对应的"纠缠母标"
3. 使用母标签 UID 的摘要重建 `SDM_Vector`
4. 重新计算 CMAC 并与 URL 中的值比对
5. 如果母子 ID 匹配不上，即便子标签本身的 CTR 是正确的，CMAC 也会校验失败

**设计意义**：

- 子标签的每一次 NFC 感应产生的动态 URL 中，都隐含了母标签的"指纹"
- 攻击者无法通过修改数据库中的映射关系来"重新绑定"子标签
- 影子参数不出现在 URL 明文中，外部观察者无法感知母标签的存在

> **代码权威**：SUN 消息规范以 `rc-common/src/sun_spec.rs` 为准。

## 三、近场双因子协议 (Proximity 2FA)

### 3.1 设计理念

近场双因子将实物资产的交接从"点击确认"升华为物理层面的握手仪式。通过物理距离作为"密码"，彻底杜绝远程指令劫持的可能性。

**核心约束**：只有当母标签和子标签在物理距离 5cm 以内同时被感应时，芯片内部的"锁定标志位"才会被清除。这种基于物理距离的算法约束，是真正意义上的"实物主权"。

### 3.2 物理环境：双标签纠缠场 (The Entanglement Field)

交接开始时，App 利用手机 NFC 射频场的多标签防碰撞（Anti-collision, ISO 14443-3A）特性。

```
┌─────────────────────────────────────────┐
│              NFC 射频场 (5cm)             │
│                                          │
│    ┌──────────┐       ┌──────────┐      │
│    │ 母标签    │       │ 子标签    │      │
│    │ (Key)    │  ←→   │ (Crown)  │      │
│    │ M_UID    │ 5cm内  │ C_UID    │      │
│    └──────────┘       └──────────┘      │
│                                          │
│              ┌──────────┐                │
│              │  手机 NFC │                │
│              │  读写器   │                │
│              └──────────┘                │
└─────────────────────────────────────────┘
```

**感应触发规则**：

| 检测到的标签 | App 状态 | UI 反馈 |
|-------------|----------|---------|
| 无标签 | 等待中 | "请将母标签和子标签同时靠近手机" |
| 仅子标签 | 锁定 | "检测到资产，请同时出示主权之匙" |
| 仅母标签 | 锁定 | "检测到主权之匙，请同时出示资产" |
| 子标签 + 母标签 | 激活 | "纠缠场已建立，正在验证..." |
| 子标签 + 非匹配母标签 | 拒绝 | "主权之匙与此资产不匹配" |

### 3.3 技术逻辑：嵌套式挑战响应 (Nested Challenge-Response)

当双标签同时出现在射频场中，App 发起一次原子化的加密会话，确保这不是两个被克隆的 ID，而是真实的物理芯片。

**三重握手协议**：

```
时间线 ──────────────────────────────────────────────────────→

Phase 1: 母标背书 (The Endorsement)
┌─────┐                    ┌─────┐
│ App │ ── AuthEV2First ──→│母标签│
│     │ ←── Session Key ── │     │
│     │ ── 请求转让意图 ──→ │     │
│     │ ←── 签名报文 ────── │     │  报文 = Sign(K_root_asset, "TRANSFER_INTENT" ‖ C_UID ‖ Timestamp)
└─────┘                    └─────┘

Phase 2: 子标接收 (The Reception)
┌─────┐                                        ┌─────┐
│ App │ ── 母标签签名报文作为 Challenge ────────→│子标签│
│     │                                         │     │  内部校验：
│     │                                         │     │  1. 提取报文中的母标签特征位
│     │                                         │     │  2. 与 EEPROM 中存储的绑定信息比对
│     │                                         │     │  3. 确认这是来自"母体"的指令
│     │ ←── 复合凭证 (The Deed) ─────────────── │     │
└─────┘                                         └─────┘

复合凭证结构：
┌──────────────────────────────────────────────┐
│ Deed = {                                      │
│   mother_uid_hash: SHA-256(M_UID)[..8],       │
│   child_uid: C_UID,                           │
│   child_state: current_asset_state,           │
│   cmd_ctr: current_command_counter,           │
│   signature: CMAC(K_chip, all_above_fields)   │
│ }                                             │
└──────────────────────────────────────────────┘

Phase 3: 云端确认 (The Settlement)
┌─────┐                    ┌─────┐
│ App │ ── Deed + Bio ────→│云端  │
│     │                    │     │  验证：
│     │                    │     │  1. Deed 签名有效性
│     │                    │     │  2. 母子 UID 映射一致性
│     │                    │     │  3. 生物特征匹配
│     │                    │     │  4. 状态转换合法性
│     │ ←── 确认/拒绝 ──── │     │
└─────┘                    └─────┘
```

### 3.4 交接仪式：三位一体的终极确认

在技术握手完成后，App 进入用户感知的仪式性步骤：

**步骤 A：物理同场感应**

- 屏幕显示两个标签正在"融合"的动画
- 系统验证了"物在场"（子标签）与"主权在场"（母标签）
- UI 话术："Entanglement Field Established — 纠缠场已建立"

**步骤 B：生物信息签署**

- 卖家执行 FaceID / 指纹识别
- 将"人的意志"与"物理凭证"绑定
- 系统生成由 `K_chip`、`UID` 和 `Biometric_Hash` 共同构建的签名
- UI 话术："Sovereign Identity Confirmed — 主权身份已确认"

**步骤 C：原子化变更**

- 上述所有证明同时上云
- 云端执行原子化状态转换：`LegallySold(3)` → `Transferred(4)` 或 `Transferred(4)` → `Transferred(4)`
- 使用乐观锁确保并发安全（`WHERE status = expected_status`）
- UI 话术："Key Transferred — 主权之匙已交付新的持匙者"

**完整时序**：

```
[物理同场感应]  →  [嵌套挑战响应]  →  [生物识别]  →  [原子化上云]
    ~2s               ~1s              ~0.5s           ~1s
                                                        │
                                              AssetState: Transferred
                                              Owner_ID: A → B
```

### 3.5 为什么要这么设计

| 设计目标 | 实现方式 |
|----------|----------|
| 防止误操作 | 需要特意拿出保险柜中的母标签，不可能口袋里误触 |
| 物理防盗 | 小偷偷走实物（子标签），但无法提供母标签在场证明，永远无法"洗白"或转卖 |
| 远程劫持防御 | 物理距离 5cm 约束，远程指令无法触发转让 |
| 仪式感赋能 | "贴一贴、扫一扫、脸一照"模拟古代契约交接中"按指纹、盖公章"的庄重感 |
| 原子性保证 | 所有证明必须同时上云，任何一环缺失则整个交易回滚 |

## 四、安全性分析

### 4.1 攻击场景与防御

| 攻击场景 | 攻击描述 | 防御机制 | 攻击成本 |
|----------|----------|----------|----------|
| 数据库篡改 | 修改云端母子映射关系 | SDM 影子绑定导致 CMAC 校验失败 | 无效 |
| 子标签克隆 | 复制子标签的 UID 和密钥 | PUF 物理指纹不可复制 + CTR 回溯检测 | 极高（硬件级） |
| 母标签克隆 | 复制母标签的认证能力 | 硬件 RNG + 滚动计数器 + 密钥永不出芯片 | 极高（硬件级） |
| 中间人攻击 | 截获并转发认证报文 | 50ms 时间窗口 + NFC 1-3cm 物理距离 | 不可行 |
| 远程指令劫持 | 远程触发转让操作 | 近场双因子要求物理同场 | 不可行 |
| 重放攻击 | 重放旧的认证会话 | Nonce 随机 + CTR 单调递增 | 无效 |
| 社工攻击 | 诱骗持有人操作 | 生物识别 + 母标签物理在场 | 需同时获取母标签 + 人脸/指纹 |

### 4.2 信任根分析

```
信任根 (Root of Trust)
├── 硬件层：NTAG 424 DNA 芯片的 PUF 和 AES-128 引擎
│   └── 密钥永不离开芯片，所有运算在安全区域内完成
├── 密码学层：HMAC-SHA256 派生链的交叉依赖
│   └── 子母密钥互相包含对方的特征摘要
├── 物理层：NFC 近场通信的距离约束（1-3cm）
│   └── 射频场的物理特性保证了"在场证明"
└── 生物层：FaceID / 指纹的本地验证
    └── 生物特征不上云，仅本地比对后生成哈希签名
```

## 五、双卡纠缠归还验证 (Re-Verify Protocol)

> 来源：`docs/系统质押模式升级.md` — 质押/借出场景下的归还验证需求

### 5.1 设计背景

当资产通过 L3.5 许可层的 Usage Token 借出后（如品牌巡展、使用权租赁），归还时需要验证"归还的实物与借出的实物是同一件"。这是近场双因子协议在质押金融场景下的自然延伸。

### 5.2 归还验证流程 (The Re-pairing)

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│  手机 App │         │ 母标签    │         │ 子标签    │
│  (中继)   │         │ (Key)    │         │ (Crown)  │
└────┬─────┘         └────┬─────┘         └────┬─────┘
     │                     │                     │
     │  1. 读取子标签       │                     │
     │ ─────────────────────────────────────────→ │
     │                     │                     │
     │  2. 获取当前计数器   │                     │
     │     C_now            │                     │
     │ ←───────────────────────────────────────── │
     │                     │                     │
     │  3. 将 C_now 转发    │                     │
     │     给母标签签名     │                     │
     │ ──────────────────→ │                     │
     │                     │                     │
     │  4. 母标签使用       │                     │
     │     K_root_asset     │                     │
     │     对 C_now 离线签名│                     │
     │ ←────────────────── │                     │
     │                     │                     │
     │  5. 纠缠验证：母卡签名与子卡派生算法匹配    │
     │     + Tamper 状态检测                       │
     │                     │                     │
```

### 5.3 验证判定矩阵

| 母卡签名 | Tamper 状态 | 判定结果 | 后续动作 |
|----------|------------|----------|----------|
| ✅ 匹配 | ✅ 正常 | 原样归还，验证成功 | 退还保证金，结算租金 |
| ✅ 匹配 | ❌ 触发 | 资产曾被非法拆解 | 扣除保证金，标记 Tampered |
| ❌ 不匹配 | — | 资产被掉包 | 扣除保证金，报警，标记 Disputed |
| — | 超时无响应 | 母卡不在场 | 拒绝归还确认，要求 Owner 介入 |

### 5.4 与近场双因子的区别

| 维度 | 近场双因子 (Proximity 2FA) | 归还验证 (Re-Verify) |
|------|--------------------------|---------------------|
| 触发场景 | 所有权转让 | 使用权归还 |
| 验证目的 | 证明"主权意志" | 证明"物品一致性" |
| 状态转换 | Transferred(4) | On_Loan → Normal |
| 额外检测 | — | Tamper-Detection 物理完整性 |
| 保证金逻辑 | 无 | 根据验证结果自动结算 |

### 5.5 防掉包机制

- 掉包者可仿造外观，但无法克隆 NTAG 424 DNA 芯片 UID
- 没有母卡无法伪造符合纠缠协议的动态 MAC
- Tamper-Detection：包内衬缝合处布置导电纤维连接芯片 Tamper 引脚，拆解尝试永久改变芯片状态位
- 归还时即便双卡纠缠验证通过，若 `Tamper_Status` 已触发，系统立即报警
