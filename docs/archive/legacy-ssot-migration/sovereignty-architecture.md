## 一、主权金字塔：密钥分发体系

整个系统的安全性源于离线根密钥，通过硬件卡片的等级制度逐级授权。

| 等级 | 名称 | 核心职能 | 持有对象 | 权限范围 |
|------|------|----------|----------|----------|
| L0 | 离线根密钥 | 整个系统的生存基石，不进入任何在线环境 | 品牌创始人 / 保险柜 | 生成 L1 创世母卡 |
| L1 | 创世母卡 | 系统的"硬件根"，用于生成和激活下级管理工具 | 品牌总部 | 授权 L2 区域母卡，重置核心算法 |
| L2 | 区域母卡 | 区域化的管理中枢，用于规模化下发生产权限 | 区域分部 / 代理商 | 授权 L3 点睛卡 |
| L3 | 点睛卡 | 生产端的核心工具，存储受限的派生逻辑 | 生产线 / 代工厂 | 执行子标签赋灵 |

### 三层实体，四级逻辑

物理层保留 L0-L3 的实物卡片等级制度（用于业务话术和组织管理），但代码中的密钥派生维持高性能的两级链路。L1/L2/L3 的主权踪迹通过派生因子（Diversification Data）注入 Chip_Key：

```
物理层（组织架构）          逻辑层（代码实现）
─────────────────          ──────────────────
L0 离线根密钥        ───→   Root_Key (HSM 物理隔离)
L1 创世母卡                      │
L2 区域母卡                      ▼
L3 点睛卡             Brand_Key = HMAC-SHA256(Root_Key, brand_id ‖ system_id)
                                 │
                                 ▼
                      Chip_Key = HMAC-SHA256(Brand_Key, UID ‖ epoch)[..16]
```

> **设计决策**：代码中不实现四级派生链。L1/L2/L3 的授权关系由业务层（rc-api）的配额管理和权限矩阵控制，而非密钥派生层级。这保证了密码学层的简洁性和可审计性。
>
> **技术权威**：密钥派生公式以 `rc-core/src/crypto/kdf.rs` 的实现为准。

### 硬件母卡 + 软件私钥分权模型

为解决"品牌方控制权"与"平台运营效率"之间的矛盾，系统采用"混合动力"分权模型：

| 角色 | 持有者 | 权限范围 | 限制 |
|------|--------|----------|------|
| 硬件母卡 (Root Mother Card) | 品牌总部 | 唯一可生成新资产或母卡的签名权，激活子标签资产 | 物理控制，无法被平台或第三方篡改 |
| 软件私钥 (Delegated Operational Key) | 平台 | 管理已激活资产状态（激活/冻结/转移/异常标记），支持自动化操作与远程控制 | 无法生成未激活资产，无法激活或伪造创世母卡签名 |

**权力分层原则**：

1. 发行权归品牌：创世母卡物理掌握，任何新资产激活都需要其签名
2. 日常操作由平台：软件私钥仅管理已激活资产状态
3. 审计透明：所有操作有签名与时间戳，可随时由品牌方验证
4. 扩展性：软件私钥支持高并发、远程操作、自动化系统

**M-of-N 多签机制**：

创世根卡不应只有一张。品牌方持有 N 张根卡，任何重大的密钥重置或权限变更需要至少 M 张根卡同时靠近读写器才能生效（如 3 张中的 2 张）。即使一名高管离职或弄丢一张卡，系统根基依然稳固。

**分层密钥派生 (KDF) 与平台隔离**：

根密钥（Root Secret）的派生过程在品牌方的物理根卡中完成，`rc-kms` 仅持有经过品牌方授权后的"派生临时私钥"。平台只提供技术流水线，不持有品牌方的"母本"。品牌方随时可以利用物理根卡单方面关闭平台权限。

**风险对策**：

| 风险 | 对策 |
|------|------|
| 软件私钥被攻破 | 限制私钥权限，仅能操作已激活资产；引入多因素认证 |
| 创世母卡丢失或损坏 | 采用 M-of-N 多重备份母卡，严格保管流程 |
| 离职员工滥用权限 | 软件私钥可撤销，权限可定期刷新；创世母卡操作受审计 |
| 非法资产激活 | 所有激活必须有创世母卡签名；系统进行签名验证 |

## 二、赋灵流水线：从"凡品"到"资产"

这是系统中"工厂端逻辑"的核心实现路径。一件普通商品通过两个阶段获得数字主权身份。

### 2.1 盲扫阶段（Digital Pre-registry）

| 要素 | 说明 |
|------|------|
| 动作 | 生产线使用普通读写设备批量读取未初始化的子标签 UID |
| 标签状态 | 出厂态（Virgin State），密钥为全零 |
| 系统记录 | 云端建立"待激活资产"条目，仅记录 UID |
| 安全意义 | 防止标签在进入点睛流程前被盗或替换 |
| 信息可见性 | 工厂仅看到"扫描成功"，无任何商品或品牌信息 |

```
工厂读写器 ──扫描──→ 子标签（Virgin）──UID──→ 云端登记
                                                 │
                                          PreMinted(0) → FactoryLogged(1) → Unassigned(10)
                                          品牌: NULL
                                          SKU:  NULL
```

### 2.2 点睛阶段（The Finishing / Provisioning）

| 要素 | 说明 |
|------|------|
| 动作 | 授权员使用点睛卡（L3）扫描产品上的子标签 |
| 前置条件 | 点睛卡已获得 L2 区域母卡授权，子标签已完成盲扫登记 |

**点睛流程**：

1. **认证**：点睛卡与子标签进行 `AuthenticateEV2First` 握手，建立加密会话
2. **派生**：系统根据点睛卡内的算法 + 子标签 UID 派生出专属的 K_chip（芯片密钥）和 K_honey（蜜标密钥）
3. **赋灵**：执行 `ChangeKey`，将全零出厂密钥替换为私有密钥，并配置 SDM 动态 URL
4. **结果**：产品获得"品牌灵魂"，正式成为品牌实物资产

```
点睛卡(L3) ──AuthEV2First──→ 子标签（Unassigned）
    │                              │
    │  ← 加密会话建立 →            │
    │                              │
    ├── 派生 K_chip, K_honey       │  Unassigned(10) → RotatingKeys(11)
    ├── ChangeKey(K0..K4)  ───────→│  密钥替换        → EntangledPending(12)
    ├── WriteData(SDM URL) ───────→│  配置动态消息
    └── ChangeFileSettings ───────→│  启用 SDM + CMAC → Activated(2)
                                   │
                            状态: Activated（已赋灵）
```

### 2.3 漏洞封堵

| 风险场景 | 防御机制 |
|----------|----------|
| 工厂私自加单 | 没有点睛卡无法赋予品牌灵魂，空白标签无商业价值 |
| 新品信息泄密 | 工厂系统仅有 UID，无品牌/SKU 信息 |
| 标签被盗 | 空白 UID 无法激活，盲扫记录可追溯 |
| 点睛卡被盗 | 配额限制 + 地理围栏 + 远程挂失（Kill Switch） |

## 三、所有权双轨制：子母标签的物理纠缠

这是系统中"用户端逻辑"的核心，旨在实现使用权与归属权的物理分离。

### 3.1 子标签（Sub-Tag）：实物的使用权

| 属性 | 说明 |
|------|------|
| 物理位置 | 永久贴合/镶嵌在实物产品上 |
| 核心功能 | **"看"** — 验证真伪、展示资产信息 |
| 交互方式 | 手机扫描 → 校验 SUN 动态签名 → 弹出资产主页 |
| 展示内容 | 真伪验证结果、生产日志、品牌故事 |
| 权限限制 | 无法修改所有权、无法挂失、无法进行高级配置 |

### 3.2 母标签（Parent-Tag）：数字的归属权

| 属性 | 说明 |
|------|------|
| 物理位置 | 独立于产品，作为"所有权证书"由持有人在安全处保管 |
| 核心功能 | **"管"** — 控制资产处置权 |
| 交互方式 | 母标签相当于物理世界的"多签私钥" |
| 权限范围 | 所有权转让、二级市场授权、主权锁定/释放 |

### 3.3 权力分离模型

| 角色 | 持有物 | 权力 | 类比 |
|------|--------|------|------|
| 归属者 | 母标签 Key（匙） | 资产处置权：过户、转让、流通授权 | 投票权 |
| 使用者 | 子标签 Crown（冠）+ 实物 | 资产使用权：穿戴、展示、验真 | 分红权 |

### 3.4 所有权变更：三因子验证

当发生转赠、二手交易或挂失时，系统要求执行三因子验证：

| 因子 | 验证内容 | 证明目的 |
|------|----------|----------|
| 子标签在场 | 物理感应子标签 NFC | 证明实物在手 |
| 母标签在场 | 物理感应母标签 NFC | 证明所有权在手 |
| 生物信息识别 | FaceID / 指纹 | 证明操作者是当前合法持有人 |

```
所有权变更流程：

  子标签感应 ──→ UID + SUN 签名验证 ──→ 实物确认 ✓
       +
  母标签感应 ──→ 母卡签名验证 ──→ 所有权确认 ✓
       +
  生物识别 ──→ FaceID/指纹 ──→ 身份确认 ✓
       │
       ▼
  三因子全部通过 → 执行所有权转移
```

## 四、开发核心逻辑审计与建议

### 4.1 标签生命周期状态机

代码实现的 14 态模型（`rc-common/src/states.rs`）是唯一真值来源。本文档的三态模型是面向业务的心智抽象，映射关系如下：

| 主权抽象 (3 态) | 对应底层状态 (14 态) | 核心准入条件 |
|-----------------|---------------------|-------------|
| Virgin (初始态) | PreMinted(0), FactoryLogged(1), Unassigned(10) | 密钥为全零，仅有 UID 被录入 |
| Enlightened (赋灵态) | RotatingKeys(11), EntangledPending(12), Activated(2) | 执行了 ChangeKey，K_chip 已注入，品牌信息已绑定 |
| Tangled (纠缠态) | LegallySold(3), Transferred(4) | 子母标签在云端完成逻辑锚定，所有权确立 |

终态和冻结态不属于三态抽象，它们是跨阶段的安全机制：

| 分类 | 状态 | 说明 |
|------|------|------|
| 终态 | Consumed(5), Legacy(6), Tampered(7), Compromised(8), Destructed(9) | 不可逆，进入后永久锁定 |
| 冻结态 | Disputed(13) | 可逆冻结，需三重验证解冻 |

> **注意**：盲扫后系统必须进入 Unassigned(10)，而非直接进入 Activated。因为此时工厂只知道"有一个标签"，但并不知道它将属于哪个 Brand 或 SKU。只有当点睛卡介入时，才发生 Unassigned → RotatingKeys → EntangledPending → Activated 的状态推进。

```
Virgin 阶段                    Enlightened 阶段                    Tangled 阶段
┌──────────────────────┐      ┌──────────────────────────┐      ┌─────────────────┐
│ PreMinted(0)         │      │ RotatingKeys(11) [瞬态]  │      │ LegallySold(3)  │
│   ↓                  │      │   ↓                      │      │   ↓              │
│ FactoryLogged(1)     │ ───→ │ EntangledPending(12)[瞬态]│ ───→ │ Transferred(4)  │
│   ↓                  │      │   ↓                      │      │   ↓ (可重复)     │
│ Unassigned(10)       │      │ Activated(2)             │      │ Transferred(4)  │
└──────────────────────┘      └──────────────────────────┘      └─────────────────┘
         密钥: 全零                    密钥: 私有                    子母纠缠完成
         品牌: 无                      品牌: 已绑定                  所有权: 已确立
```

### 4.2 安全性隔离建议

| 隔离层 | 要求 |
|--------|------|
| 前端隔离 | 普通消费者使用的 H5 页面应仅包含读取和验证 SUN 签名的逻辑，不得包含任何密钥操作代码 |
| 后端隔离 | 涉及 `ChangeKey` 或所有权转移的敏感 API，必须要求由特定的"点睛卡硬件指纹"或"母卡加密报文"触发，严禁仅通过 App ID 触发 |
| 密钥隔离 | 各级密钥派生在独立的安全上下文中执行，中间密钥用完即清零 |

### 4.3 容灾机制

| 场景 | 处理方案 |
|------|----------|
| 母标签丢失 | 持有人向品牌方申诉 → 品牌方通过 L1 创世母卡权限作废旧母卡 → 重新签发新母卡 |
| 点睛卡丢失 | 区域管理员通过 L2 区域母卡远程挂失（Kill Switch）→ 重新签发新点睛卡 |
| 创世母卡丢失 | CEO + 技术总监 + 平台方三方多签 → 使用 L0 离线根密钥生成新创世母卡 |
| L0 根密钥丢失 | 灾难性事件，需启动品牌级密钥重置流程（所有下级密钥需重新派生） |

## 六、质押场景下的主权操作

> 详细设计参见 `docs/系统质押模式升级.md`

### 6.1 质押与主权的关系

四层架构赋予了 RCP 协议处理"复杂金融关系"的能力。当资产被质押时，用户依然是 Owner (L3)，但 Auth (L4) 层的权限被临时托管给质押平台，用户无法行使转让权。

### 6.2 多重签名锁定器

质押期间，资产处置权需要 `User_Key + Platform_Key` 共同签名：

| 操作 | 所需签名 | 说明 |
|------|----------|------|
| 查看资产 | 无需签名 | 公开信息 |
| 展示/验真 | 子标签即可 | 使用权不受影响 |
| 转让/出售 | User_Key + Platform_Key | 质押期间双签才能转让 |
| 强制平仓 | Platform_Key（保护期过期后） | 坏账时平台单方面行使 |

### 6.3 质押状态机扩展

在现有 14 态模型基础上，L3 账户层新增质押相关状态：

```
Normal（正常态）
  ├─ 用户发起质押 → Pledged（质押态）
  │                    ├─ 正常还款 → Normal
  │                    ├─ 价值波动 → Warning（预警态）
  │                    │               ├─ 追加质押 → Pledged
  │                    │               └─ 超时未响应 → Frozen
  │                    └─ 用户违约 → Frozen（冻结态）
  │                                    ├─ L3 账户全部资产冻结
  │                                    └─ 进入清算 → Defaulted（违约态）
  │                                                    ├─ L2 全局标记违约
  │                                                    ├─ WID 列入黑名单
  │                                                    └─ AID 划拨至清算账户
  └─ 签发使用权 → On_Loan（借出态）
                    ├─ 正常归还（双卡纠缠验证） → Normal
                    └─ 使用者违约 → 吊销使用权限，Owner 不受影响
```

### 6.4 使用权租赁：L3.5 许可层

在 L3（账户层）和 L4（认证层）之间引入动态插件层 L3.5：

- Owner (L3) 签发有时限的 Usage Token (UT) 给 User (L3.5)
- UT 定义：`AID + Time_Range + Permission_Level`
- 借出期间子标签在 L3 层标记为 `On_Loan`，权限仅限"基础展示"和"位置上报"
- 归还时通过双卡纠缠感应场验证一致性

### 6.5 双卡纠缠归还验证

借出资产归还时的验证流程：

```
子卡扫码 → 获取 C_now → 母卡触碰 → 离线签名(C_now) → 纠缠验证
                                                          │
                                              ├─ 签名匹配 + Tamper 正常 → "原样归还"
                                              ├─ 签名不匹配 → "资产被掉包"
                                              └─ Tamper 触发 → "资产曾被非法拆解"
```

## 七、总结与下一步

这套架构将原本抽象的品牌溢价转化为一种"物理可感知的主权"。

### 对品牌的价值

- 防止代工厂私自加单 — 没有点睛卡无法赋予品牌灵魂
- 供应链全程可追溯 — 盲扫→点睛→激活→流转，每一步留痕
- 二级市场净化 — 主权锁定机制斩断非授权流通

### 对用户的价值

- 极高仪式感的所有权证明 — 母标签即物理世界的"产权证"
- 二手流转中的无可抵赖信用基础 — 三因子验证确保交易安全
- 使用权与处置权分离 — 赠予场景下的优雅权力控制
