## 📜 Regalis Clavis Protocol (RCP) 体系定义书

## 1. 协议愿景

**Regalis Clavis**（拉丁语：皇室之钥）旨在通过硬件级密钥绑定（Key-Binding），将物理世界的实体转化为具备主权属性的数字资产。我们的目标不是“防伪”，而是“确权”：**定义并保护资产的真实归属。**

## 2. 三层命名与逻辑架构

- **Layer 1: RCP 协议层 (The Protocol)**
    
    - **定义**：底层逻辑、加密原语、NIST SP 800-108 密钥派生算法。
        
    - **职责**：确保数学层面的绝对不可伪造性。
        
- **Layer 2: Sovereign Object 主权对象 (The Identity)**
    
    - **定义**：物理实物在数字世界中的孪生体。
        
    - **职责**：承载权属历史、品牌元数据及金融化接口。
        
- **Layer 3: RC-Key 资产之钥 (The Touchpoint)**
    
    - **定义**：用户手中的实物硬件（如 NTAG 424 DNA 芯片）。
        
    - **职责**：作为唯一的物理凭证，实现扫码即验真、扫码即行权的交互体验。
        

## 3. 生命周期状态机 (The Lifecycle)

1. **盲扫 (Discovery)**：由授权设备执行，建立数字存证（Census）。状态：`FACTORY_NEW`。
    
2. **纠缠 (Entanglement)**：核心主权更替仪式。将出厂密钥翻转为品牌专有主权密钥。状态：`SOVEREIGN_AUTHENTIC`。
    
3. **验真 (Sovereign Verification)**：基于挑战-响应机制（Challenge-Response）的实时质询。
    
4. **熔断 (Circuit Break)**：检测到克隆、重放或物理破坏时，主权状态自动降级或锁定。



这套三层架构正是 **Regalis Clavis Protocol (RCP)** 能够从简单的“防伪标签”演变成“资产主权系统”的底层支撑。

我们可以将这三层密钥逻辑进一步升华为**“发行权、治理权、所有权”**的权力结构。为了让这个体系更加闭环，我为你补充了每一层在实际生产和权属流转中的具体职能：

## 1. 品牌方密钥 (The Issuance Key - 发行主权)

- **定位**：资产的“铸造模具”。
    
- **职责**：
    
    - **限定总量**：只有持有品牌密钥的设备，才能在 `blind_log` (盲扫) 阶段生成有效的资产存证。
        
    - **渠道隔离**：通过派生不同的次级密钥，品牌方可以追踪该资产是属于“全球旗舰店”还是“特定代理商”。
        
- **补充点**：品牌方密钥不直接触碰芯片，它通常存在于品牌方的 **HSM (硬件安全模块)** 中，通过 RCP 协议远程授权生产线进行“纠缠激活”。
    

## 2. 平台方密钥 (The Governance Key - 治理主权)

- **定位**：数字世界的“公证处”与“宪法”。
    
- **职责**：
    
    - **跨品牌互认**：平台密钥确保“品牌 A”无法伪造“品牌 B”的资产，因为所有 KDF (密钥派生) 的根源（Root Key）由平台托管。
        
    - **双重验证 (Attestation)**：当用户扫码时，平台密钥负责验证当前的挑战值（Challenge）是否由合法的 RCP 协议生成，防止伪造的服务器欺骗用户。
        
- **补充点**：这是整个生态的“信任锚点”。如果品牌方倒闭了，只要平台密钥和数据库还在，用户手中的资产主权依然可以被证明。
    

## 3. 用户/芯片密钥 (The Sovereign Key - 处分主权)

- **定位**：资产的“数字灵魂”。
    
- **职责**：
    
    - **硬件锁定**：这是存储在 NTAG 424 DNA 内部的 $K_{chip}$。它不可被读取，只能被质询。
        
    - **权属表达**：当资产发生转移（二手交易）时，用户通过物理上移交这个“硬件私钥”，在数字空间完成所有权的原子级切换。
        
- **补充点**：在“纠缠”完成后，这个密钥就是唯一的。即便品牌方想撤回这件商品，只要物理芯片在用户手里，用户就拥有不可剥夺的**“离线确权”**能力。

## 💡 为使体系更严谨，建议补充的“第四维度”：

除了上述三层静态密钥，在实际架构中，我们需要引入一个**“动态层”**来防御实时攻击：

#### **4. 会话临时密钥 (Ephemeral Session Key)**

- **必要性**：如果只靠前三层，黑客可以通过监听 NFC 通信重放（Replay）你的信号。
    
- **逻辑**：在每一次 `AuthenticateEV2` 过程中，芯片和读卡器会协商出一个只存在几秒钟的“临时密钥”。
    
- **价值**：它确保了即便是完全一样的扫码动作，每一秒钟产生的数据包都是独一无二的。这解决了“物理克隆”后的“数字重放”问题。

## 🛡️ 主权自愈协议：三重确认与数字断层

我们不能“重写”原来的芯片，我们必须执行一次**“主权迁移”**。

## 1. 触发机制：双重证据链 (Dual-Evidence)

自愈不能只靠用户一句话，必须满足以下两个物理条件：

- **物理证据**：用户必须将带有损坏芯片的实物带回品牌官方网点（Boutique）。
    
- **数字存证**：品牌方通过其“发行密钥”确认该实物符合之前的“盲扫”记录。
    

## 2. 执行逻辑：熔断与链接 (Burn & Link)

这不是简单的“找回密码”，而是一个**“销毁旧身份，继承旧历史”**的过程：

1. **吊销 (Revoke)**：平台密钥将旧的 UID（损坏的芯片）在全球范围内标记为 `DECEASED`（已失效）。
    
2. **纠缠新密钥 (New Entanglement)**：品牌方拿出一枚全新的芯片，将其与原有的资产元数据（生产日期、工艺、历史所有权）进行**重新绑定**。
    
3. **主权继承 (Inheritance)**：新的 `RC-ID` 会携带一个 `Successor_Of` 标签，指向那个死去的 UID。
    

## 3. 风险对冲：预防“幽灵资产” (Anti-Ghosting)

这是你担心的漏洞：**如果旧芯片没坏，用户只是谎称坏了以骗取第二个合法 ID 怎么办？**

- **冷却期 (Cooling-off Period)**：新主权在激活后会有 30 天的“观察期”，期间不具备转让权。
    
- **双重扫描预警**：一旦被标记为 `DECEASED` 的旧芯片再次出现在任何联网的扫描设备上，系统会立即触发 **“主权冲突警告”**。
    
- **代价性修复**：自愈动作应记录在资产的“信用历史”中。就像事故车一样，经历过“主权自愈”的资产，其二级市场价值会有透明的记录，从而震慑恶意欺诈行为。
