## 序言

在一个充斥着仿冒、篡改与信任危机的世界里，我们如何证明一件物品是真实的？

几千年来，人类依赖印章、签名、证书和权威机构来建立对物品的信任。但这些手段正在失效——印章可以伪造，证书可以复制，权威机构可以被收买或消失。

Regalis Clavis Protocol（RCP）提出一个根本性的回答：

> 让物品自己证明自己。

RCP 是一套**物理资产共识协议**（Physical Asset Consensus Protocol）。它不依赖任何单一机构的背书，而是通过密码学、硬件安全元件和开放验证规范，让每一件物理资产在数字世界中拥有**不可伪造的身份、不可篡改的历史、不可否认的主权**。

本宪法定义的不是一套软件，而是一套规则——一套关于"真实性"如何被定义、验证和裁决的规则。

它适用于所有参与 RCP 生态的品牌、验证者、机构和个人。

## 第一章 核心原则

RCP 的一切设计决策，均源自以下五项不可违背的原则。

### 第一条 主权独立

每一件资产的真实性与状态，不依赖任何单一服务器、单一公司或单一国家。

任何人，在任何地方，只要遵循本协议的公开规范，都可以独立验证一件资产的真伪。

### 第二条 可验证性

任何关于资产真实性的结论，必须基于可复现的密码学证明和公开的协议规则。

不接受"因为我说它是真的，所以它是真的"。

### 第三条 不可抵赖性

资产生命周期中的每一次状态变更——从铸造到流通，从转让到消亡——都必须产生可验证的证明。

任何参与方都不可事后否认已发生的操作。

### 第四条 渐进信任

世界不是非黑即白的。RCP 允许在不同条件下输出不同等级的信任结论：

- **可信（TRUE）** — 完整的密码学证明和历史链条均通过验证
- **降级可信（DEGRADED）** — 部分验证通过，但存在硬件异常或信息缺失
- **未知（UNKNOWN）** — 信息不足以做出判断
- **伪造（FALSE）** — 发现明确的伪造证据或逻辑冲突

### 第五条 优雅降级

系统的任何单一组件——芯片、服务器、密钥、甚至运营方本身——都可能失效。

RCP 的设计确保：当部分组件失效时，系统降级运行而非整体崩溃。硬件损坏不等于资产消亡，服务器关闭不等于真实性消失。

## 第二章 资产的数字身份

### 第六条 资产的构成

在 RCP 的世界观中，一件"资产"由三个不可分割的要素构成：

- **物理实体** — 一件真实存在的物品
- **安全元件** — 嵌入物品中的硬件密码学芯片
- **身份绑定** — 将物理实体与其数字身份不可逆地关联

三者缺一，资产不成立。

### 第七条 身份的唯一性

每件资产拥有一个由硬件芯片生成的唯一标识（UID），以及基于该标识派生的密码学密钥体系。

这套密钥体系赋予资产三种核心能力：

- **证明自己是谁** — 通过密码学签名回应质询
- **保护自己的隐私** — 对未授权的读取进行加密屏蔽
- **记录自己的历史** — 通过不可逆的状态转移留下轨迹

### 第八条 状态与生命周期

每件资产在其生命周期中经历一系列状态转换。这些状态构成一个有限状态机，典型的生命轨迹如：

> 铸造 → 运输 → 交付 → 激活 → 流通 → 消亡

状态转换遵循严格的规则：单向不可逆（除非协议明确允许），且每一次转换都必须生成密码学证明。

## 第三章 验证模型

### 第九条 双重验证

RCP 采用静态验证与动态验证相结合的双重机制：

**静态验证**检验资产的"出生证明"——它的初始配置和发行签名是否完整、未被篡改。

**动态验证**检验资产的"此刻存在"——通过向芯片发送不可预测的随机挑战，要求芯片实时计算密码学响应，证明"它此时此刻就在你面前"。

两者结合，既能识别伪造的复制品，也能识别录制回放的"幽灵"。

### 第十条 验证的独立性

验证不是一项特权，而是一项公共能力。

RCP 公开其完整的验证规范，包括密钥派生规则、签名计算方式、状态机定义和证明格式。任何实体——无论是品牌方、第三方鉴定机构还是独立开发者——都可以据此实现自己的验证器。

不同验证器对同一资产的验证结果必须一致。

### 第十一条 离线验证

验证不应依赖网络连接。

任何验证者只要持有资产的身份标识、发行快照、状态转移证明链和协议规范，就必须能够在完全离线的环境下完成验证。

服务器是便利，不是必需。

## 第四章 信任层级

### 第十二条 四级信任体系

RCP 定义四个递进的信任等级，每一级建立在前一级的基础之上：

| 等级 | 名称 | 验证内容 | 含义 |
| ---- | ---- | -------- | ---- |
| L0 | 物理真实性 | 芯片密码学响应（CMAC） | 这块芯片是真的 |
| L1 | 发行真实性 | 主权快照签名（Snapshot） | 这件资产是被合法发行的 |
| L2 | 历史真实性 | 状态转移证明链（TP Chain） | 这件资产的历史轨迹是完整且未被篡改的 |
| L3 | 社会真实性 | 外部机构签名（External Signatures） | 这件资产获得了权威机构的背书 |

### 第十三条 冲突裁决

当不同验证者或不同信任等级给出矛盾结论时，默认规则为：高等级信任覆盖低等级信任。

在无法确定优先级的情况下，采用加权共识机制裁决。

## 第五章 密钥治理

### 第十四条 五权分立的密钥架构

RCP 将资产的安全能力分配到五个独立的密钥槽位，每个槽位承担一项且仅一项职责：

| 密钥 | 职责 | 说明 |
| ---- | ---- | ---- |
| Key 0 | 主权根密钥 | 资产的"生命线"，仅用于密钥更替和主权迁移，严禁参与日常业务 |
| Key 1 | 隐私保护 | 加密资产的敏感元数据，确保未授权扫描时身份不可见 |
| Key 2 | 真实性验证 | 生成和验证动态密码学签名，是防伪的核心 |
| Key 3 | 生命周期控制 | 管理资产的状态转换，控制从"流通"到"消亡"的物理锁定 |
| Key 4 | 能力扩展 | 为第三方合作伙伴提供受限的、可审计的访问能力 |

这五个密钥不是独立随机生成的，而是通过确定性的密钥派生函数从品牌主密钥逐级派生，实现硬件隔离与算法隔离的双重保障。

### 第十五条 密钥纪元

密钥不是永恒的。RCP 引入"密钥纪元"机制：

- 每一代品牌主密钥定义一个纪元
- 新资产使用当前纪元的密钥
- 旧资产在旧纪元下仍可验证
- 当密钥泄露发生时，仅废弃受影响的纪元，而非整个系统

这将"系统级灾难"降级为"版本级退化"。

### 第十六条 密钥派生的命名空间隔离

所有密钥派生必须包含多维命名空间参数：应用标识、用途域、槽位索引和版本号。

这确保即使在多品牌、多系统共存的环境下，不同品牌、不同用途的密钥之间也完全正交，不存在跨域重用的可能。

## 第六章 状态证明

### 第十七条 主权快照

资产在完成初始绑定（"纠缠仪式"）时，系统生成一份不可伪造的主权快照，记录资产在"出生"那一刻的完整配置特征。

这份快照是资产的"出生证明"，任何人都可以据此验证资产是否曾被非法重新初始化。

### 第十八条 状态转移证明

资产生命周期中的每一次状态变更，都必须生成一份包含前序状态哈希的转移证明。

这些证明首尾相连，形成一条不可断裂的状态链——即便服务器数据库被摧毁，只要芯片在手，任何人都可以通过这条链还原资产的完整生命轨迹。

### 第十九条 两阶段提交

涉及不可逆物理操作的状态变更（如永久锁定），必须采用两阶段提交机制：

> 当前状态 → 待确认状态 → 最终确认

这确保在通信中断或硬件异常时，资产不会陷入不可恢复的"半死不活"状态。

## 第七章 失败语义

一个全球协议必须正面回答一个问题：当系统出错时，世界如何继续运行？

### 第二十条 芯片失效不等于资产消亡

芯片可能损坏、无法通信或被物理破坏。RCP 定义三种芯片状态：

- **存活（Alive）** — 完全功能，支持动态验证
- **降级（Degraded）** — 无法实时响应，但历史证明仍然有效
- **死亡（Dead）** — 无法通信，仅可依赖已存档的证明链验证

硬件故障被降级为信任等级下降，而非系统崩溃。

### 第二十一条 密钥泄露的可控爆炸半径

密钥泄露时，通过密钥纪元机制将影响限制在特定版本范围内。新资产立即切换到新纪元，旧资产在标记后仍可验证。

### 第二十二条 服务端消失不影响验证

RCP 的设计确保：即使协议运营方停止运营，已发行资产的验证能力不受影响。

任何持有协议规范和资产证明链的验证者，都可以独立完成验证。信任锚定在协议本身，而非任何特定的服务器。

### 第二十三条 验证分裂的裁决

当不同验证者给出矛盾结论时，依据信任层级（第十二条）进行裁决。物理层证据优先于逻辑层推断，完整证据优先于部分证据。

## 第八章 主权模型

### 第二十四条 三权分立

RCP 将资产主权拆分为三项独立的权力，分别由不同的参与方持有：

| 权力 | 持有者 | 说明 |
| ---- | ------ | ---- |
| 发行权（Mint Authority） | 品牌方 | 决定哪些物品被赋予数字身份，掌握主权根密钥 |
| 验证权（Verify Authority） | 开放验证网络 | 任何遵循协议规范的实体都可以验证，不需要许可 |
| 仲裁权（Arbitration Authority） | 多方机构 | 在争议场景中，基于证明链进行裁决 |

没有任何单一实体同时掌握三项权力。

### 第二十五条 主权迁移

品牌方可以通过主权根密钥发起密钥轮换，将资产的验证体系迁移到新的密钥纪元或新的验证网络。

迁移过程不需要收回任何物理资产。已发行资产的历史证明链永久有效。

## 第九章 协议演进

### 第二十六条 版本兼容

所有协议结构必须包含版本标识。验证器必须支持至少前一个版本的验证能力。

### 第二十七条 升级原则

协议的任何升级必须遵守两条铁律：

1. 不破坏对历史资产的验证能力
2. 不要求收回已发行的物理资产

### 第二十八条 开放标准

RCP 的验证规范、密钥派生规则、状态机定义和证明格式必须完整公开。

协议的安全性依赖密钥的保密性，而非规则的保密性。这是密码学的基本原则（Kerckhoffs 原则），也是 RCP 的基本信仰。

## 第十章 安全目标

RCP 承诺实现以下安全目标：

| 目标 | 含义 |
| ---- | ---- |
| 防复制（Anti-cloning） | 无法通过复制芯片数据来伪造资产 |
| 防重放（Anti-replay） | 无法通过录制和回放历史验证数据来欺骗验证器 |
| 不可抵赖（Non-repudiation） | 任何已发生的操作都有密码学证据，不可否认 |
| 容错（Fault tolerance） | 单点故障不导致系统崩溃 |
| 去中心验证（Decentralized verification） | 验证能力不被任何单一实体垄断 |

## 第十一章 物理绑定

### 第二十九条 数字身份与物理实体的绑定

资产的真实性最终锚定在物理世界。密码学芯片的证明与物理实体的特征共同构成完整的真实性判定：

> 真实性 = 芯片密码学证明 + 物理特征指纹

数字身份可以证明"这块芯片是真的"，物理绑定证明"这块芯片属于这件物品"。

## 结语

RCP 不是一个产品，不是一个平台，不是一家公司的私有协议。

它是一套规则——关于物理世界中的真实性如何被定义、验证和裁决的规则。

它回答的是一个古老而紧迫的问题：

> **在一个不可信的世界里，我们如何就"什么是真的"达成共识？**

RCP 的答案是：不依赖权威，不依赖信任，只依赖数学和物理定律。

让每一件物品，自己证明自己。
