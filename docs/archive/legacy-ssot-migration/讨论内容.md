## 一、核心架构定位

### 1.1 系统本质

RCP 不是一个简单的 NFC 防伪方案,而是一个**物理资产共识协议 (Physical Asset Consensus Protocol)**。

它解决的核心问题:

> 如何在不可信环境中,建立对"物理世界真实性"的共识?

### 1.2 架构层级

```text
Layer 0 → Physical Object (物理实体)
Layer 1 → Secure Element (NTAG 424 DNA 芯片)
Layer 2 → Cryptographic Identity (密码学身份: CMAC / KDF)
Layer 3 → Stateful Proof (状态证明: Snapshot + TP)
Layer 4 → Verification Spec (开放验证规范)
Layer 5 → Failure Semantics (失败语义与系统韧性)
```

### 1.3 设计哲学

- **主权独立** — 不依赖单一服务器或机构
- **可验证性** — 基于密码学证明,而非权威背书
- **不可抵赖** — 所有状态变更留下不可篡改的证明链
- **渐进信任** — 支持 TRUE / DEGRADED / UNKNOWN / FALSE 四级信任输出
- **优雅降级** — 单点故障不导致系统崩溃

## 二、密钥架构演进

### 2.1 NTAG 424 DNA 五权分立

RCP 将 NTAG 424 DNA 的 5 个密钥槽位映射为五项独立的安全职责:

| 槽位  | RCP 语义                    | 职责                                   | 实现状态 |
| ----- | --------------------------- | -------------------------------------- | -------- |
| Key 0 | 主权根密钥 (Sovereign Root) | 密钥更替与主权迁移,严禁日常业务        | ✅ 已实现 |
| Key 1 | 隐私加密 (Privacy ENC)      | 加密 SUN 消息中的 UID 和 Counter       | ✅ 已实现 |
| Key 2 | 验真核心 (Auth CMAC)        | 生成和验证动态 CMAC,防伪的灵魂         | ✅ 已实现 |
| Key 3 | 生命周期锁 (Lifecycle Lock) | 控制状态转换与物理锁定                 | ✅ 已实现 |
| Key 4 | 合作伙伴密钥 (Partner Key)  | 为第三方提供受限访问能力               | ⚠️ 部分实现 |

**核心观点:**

> NTAG 424 DNA 的 5 个 Key Slot 不是"5 把钥匙",而是**5 个安全职责的硬件隔离执行位**。

### 2.2 KDF 架构强化

#### 当前实现 (✅ 已完成)

**基础 KDF 链:**

```text
Root_Key → Brand_Key → K_chip
```

- 使用 AES-128-CMAC KDF (符合 NTAG 424 DNA 硬件约束)
- 支持 HMAC-SHA256 KDF (向后兼容,已标记 deprecated)
- Brand_Key 缓存机制 (RwLock + double-check locking)
- 密钥材料 Zeroize 保护

**实现位置:**

- `crates/rc-core/src/crypto/kdf.rs` — 核心 KDF 函数
- `crates/rc-kms/src/services/key_derivation.rs` — 服务层封装

#### 待强化功能 (🔄 规划中)

**1. 多维命名空间隔离**

**问题:** 当前 KDF Context 过于简单,存在跨品牌 Key 重用攻击风险。

**升级方案:**

```rust
K_slot[n] = AES-CMAC(
  K_brand,
  UID || AppID || Domain || SlotIndex || Version
)
```

- **AppID** — 应用级隔离 (如 `RCP-01-PROD`)
- **Domain** — 用途域 (`AUTH_CMAC` / `DATA_ENC` / `ADMIN_CTRL`)
- **Version** — 密钥版本号,支持未来升级

**优先级:** P1 (高)  
**预计工作量:** 2-3 天  
**依赖:** 需要更新 `rc-core::crypto::kdf` 模块

**2. 密钥纪元 (Key Epoch) 机制**

**问题:** 密钥泄露时,当前只能废弃整个系统。

**升级方案:**

```text
K_brand_v1 → K_brand_v2 → K_brand_v3
```

- 新资产使用当前纪元
- 旧资产在旧纪元下仍可验证
- 泄露时仅废弃受影响纪元

**优先级:** P1 (高)  
**预计工作量:** 3-5 天  
**依赖:** 需要数据库 schema 升级,增加 `key_epoch` 字段

## 三、验证模型强化

### 3.1 双重验证机制

#### 当前实现 (✅ 已完成)

**静态验证:**

- Snapshot 签名验证
- 配置一致性校验

**动态验证:**

- CMAC 实时计算与验证
- Counter 单调性检查
- 重放攻击检测

**实现位置:**

- `crates/rc-kms/src/services/verification.rs`
- `crates/rc-core/src/protocol/nfc/sun.rs`

#### 待强化功能 (🔄 规划中)

**1. 远程证明 (Remote Attestation Lite)**

**问题:** 当前只能证明"当前状态",无法证明"芯片未被重新初始化"。

**升级方案:**

在 Entanglement 时生成主权快照:

```text
Attestation = Sign(K_server, UID || Key0_Status || Config_Hash || Timestamp)
```

验证时比对芯片当前配置与原始快照是否一致。

**优先级:** P1 (高)  
**预计工作量:** 3-4 天  
**依赖:** 需要 HSM 签名能力,数据库存储 Attestation

**2. 动态挑战-响应 (Challenge-Response)**

**问题:** 静态 Snapshot 可被重放攻击。

**升级方案:**

```text
1. Reader 下发随机挑战: Challenge = Random(128bit)
2. Tag 计算: V = AES-CMAC(K_sdm, UID || Counter || Challenge)
3. Server 同时验证 Snapshot (静态) 和 V (动态)
```

**优先级:** P2 (中)  
**预计工作量:** 2-3 天  
**依赖:** 需要前端 H5 支持生成随机数

## 四、状态机与审计

### 4.1 状态机实现

#### 当前实现 (✅ 已完成)

**状态定义:**

```rust
pub enum AssetState {
    BlindLogged,    // 盲扫入库
    Entangled,      // 纠缠绑定
    Activated,      // 激活流通
    Transferred,    // 转让中
    Locked,         // 主权锁定
    Consumed,       // 已消费
    Destructed,     // 已销毁
    Disputed,       // 争议中
}
```

**状态转换规则:**

- 单向不可逆 (除非明确允许)
- 乐观锁保证原子性
- 状态转换触发审计日志

**实现位置:**

- `crates/rc-core/src/state_machine/states.rs`
- `crates/rc-core/src/state_machine/transitions.rs`

#### 待强化功能 (🔄 规划中)

**1. 状态转移证明链 (Transition Proof Chain)**

**问题:** 当前只记录"当前状态",无法证明"历史轨迹"。

**升级方案:**

每次状态变更生成不可伪造的转移证明:

```text
TP_n = Sign(K_sv, UID || S_{n-1} || S_n || Nonce || Timestamp)
```

- TP_n 包含前序状态哈希,形成不可断裂的状态链
- 在可能的情况下,将 TP_n 摘要写回芯片受控文件区
- 即便服务端数据库被摧毁,可通过芯片还原完整生命轨迹

**优先级:** P0 (最高)  
**预计工作量:** 5-7 天  
**依赖:** 需要 HSM 签名能力,数据库 schema 升级

**2. 两阶段提交 (2PC Lite)**

**问题:** NFC 中断、掉电可导致 Tag 进入"半锁死状态"。

**升级方案:**

```text
State A → Pending_B → Confirmed_B
```

- File 中保留 `current_state` 和 `pending_state`
- Key 3 只能写 pending
- Server 确认后 finalize

**优先级:** P1 (高)  
**预计工作量:** 3-4 天  
**依赖:** 需要 NFC 写入逻辑重构

### 4.2 审计日志

#### 当前实现 (✅ 已完成)

**审计事件分类:**

- ASSET — blind_log, entangle, activate, consume, destruct
- TRANSFER — initiate, confirm, cancel, expire
- SOVEREIGNTY — lock, release, recovery
- SECURITY — replay_detected, clone_suspected, circuit_break
- AUTH — login, logout, token_revoke
- ADMIN — brand_register, quota_adjust

**实现位置:**

- `crates/rc-kms/src/stores/audit.rs`
- `crates/rc-api/src/middleware/audit.rs`

#### 待强化功能 (🔄 规划中)

**1. TimescaleDB 审计日志压缩**

**问题:** 审计日志无限增长,查询性能下降。

**升级方案:**

```sql
-- 创建 hypertable (按天分区)
SELECT create_hypertable('audit_logs', 'created_at',
    chunk_time_interval => INTERVAL '1 day');

-- 7 天后自动压缩
SELECT add_compression_policy('audit_logs', INTERVAL '7 days');

-- 365 天后自动删除
SELECT add_retention_policy('audit_logs', INTERVAL '365 days');
```

**优先级:** P2 (中)  
**预计工作量:** 2-3 天  
**依赖:** 需要 PostgreSQL + TimescaleDB 扩展

## 五、失败语义与韧性

### 5.1 失败场景定义

RCP 必须正面回答四类失败场景:

| 失败类型       | 当前处理                | 待强化方案                          | 优先级 |
| -------------- | ----------------------- | ----------------------------------- | ------ |
| 芯片失效       | ✅ 三级状态 (Alive/Degraded/Dead) | 增加离线验证能力                    | P2     |
| 密钥泄露       | ⚠️ 手动废弃              | 自动化 Key Epoch 切换               | P1     |
| 服务端消失     | ⚠️ 部分支持              | 完整离线验证规范                    | P0     |
| 验证分裂       | ❌ 未实现                | Trust Hierarchy 冲突裁决机制        | P1     |

### 5.2 离线验证能力

#### 待实现功能 (❌ 未开始)

**目标:** 任何验证者只要持有以下四项,必须可以完成验证:

1. UID
2. Snapshot (签名)
3. Transition Proof Chain
4. Canonical Spec

**验证逻辑:**

```text
Verify(Asset):
  if !VerifySnapshot():  return FALSE
  if !VerifyCMAC():      return DEGRADED
  if !VerifyTPChain():   return UNKNOWN
  return TRUE
```

**优先级:** P0 (最高)  
**预计工作量:** 7-10 天  
**依赖:** 需要完成 Transition Proof Chain 实现

## 六、协议标准化

### 6.1 开放验证规范

#### 待实现功能 (❌ 未开始)

**目标:** 定义 RCP Verification Canonical Spec v1.0,包括:

- CMAC 计算方式
- KDF 规则
- Domain 枚举
- 状态机转移规则
- Snapshot 结构
- Proof 格式

**意义:** 任何第三方可独立实现验证器,但无法伪造资产。

**优先级:** P1 (高)  
**预计工作量:** 10-14 天  
**交付物:** `docs/RCP_Verification_Spec_v1.0.md`

### 6.2 三权分立

#### 当前实现 (⚠️ 部分完成)

| 权力                              | 持有者       | 当前状态      |
| --------------------------------- | ------------ | ------------- |
| 发行权 (Mint Authority)           | 品牌方       | ✅ 已实现 (Key0) |
| 验证权 (Verify Authority)         | 开放验证网络 | ⚠️ 部分实现    |
| 仲裁权 (Arbitration Authority)    | 多方机构     | ❌ 未实现      |

#### 待强化功能 (🔄 规划中)

**1. 独立验证器参考实现**

提供一个最小化的独立验证器实现,证明 RCP 的去中心化验证能力。

**优先级:** P1 (高)  
**预计工作量:** 5-7 天  
**交付物:** `crates/rc-verifier` (独立 crate)

**2. 多方仲裁机制**

在争议场景中,基于 Transition Proof Chain 进行裁决。

**优先级:** P2 (中)  
**预计工作量:** 7-10 天  
**依赖:** 需要完成 TP Chain 实现

#### 待实现功能 (❌ 未开始)

```text
验证请求 ──► L1 (DashMap) ──miss──► L2 (Redis) ──miss──► L3 (PostgreSQL)
                │                        │                      │
                │ hit                    │ hit                  │
                ▼                        ▼                      ▼
            返回 CTR               INCR + 返回            SELECT + 返回
                │                        │                      │
                └────────────────────────┴──────────────────────┘
                                         │
                              异步批量刷盘 (每 60s / 1000 条)
```

**优先级:** P2 (中)

## 结语

RCP 已经从"一个很强的系统"演进到"具备成为行业基础设施潜力的协议雏形"。

当前系统完成度:

```text
✅ Identity (身份)       — 已完成
✅ State (状态)          — 已完成
⚠️ History (历史)        — 部分完成 (缺 TP Chain)
✅ Liveness (活性)       — 已完成
⚠️ Spec (规范)           — 部分完成 (缺开放验证规范)
⚠️ Failure Semantics (失败语义) — 部分完成
```

能否成为标准,取决于三件事:

| 维度 | 状态     | 下一步                          |
| ---- | -------- | ------------------------------- |
| 技术 | 基本完成 | 完成 P0/P1 级功能               |
| 制度 | 刚刚开始 | 输出开放验证规范                |
| 生态 | 尚未构建 | 提供独立验证器参考实现          |
