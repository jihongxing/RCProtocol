# RCP 兼容迁移与回滚手册（数据/权限/API/审计）

## 1. 适用范围

- 本手册用于 B 端治理平面上线时，保证对现有 Rust 核心链路零破坏。
- 涵盖四类兼容：数据、权限、API、审计。
- 原则：先增量兼容，后切流；先可回滚，再可扩展。

## 2. 数据层迁移

### 2.1 迁移策略

- 仅做增量 schema（add column/add table/add index），不删除旧字段。
- 新表统一命名为 `ops_*`（治理侧），核心资产表继续由 Rust 维护。
- 所有 DDL 必须支持前滚与回滚脚本成对交付。

### 2.2 发布步骤

1. 执行前置校验：连接性、磁盘余量、慢SQL基线。
2. 执行 DDL（无锁或低锁优先）。
3. 启动双写影子（仅治理侧元数据，不写核心资产状态）。
4. 对账（行数、checksum、抽样业务比对）。
5. 切换读路径（先报表，再控制台）。

### 2.3 回滚步骤

- 未切流：直接回滚 DDL 或停用新表引用。
- 已切流：切回旧读路径，保留新表数据用于复盘。
- 严禁：删除核心历史链数据、覆盖资产主状态。

## 3. 权限层迁移

### 3.1 角色映射

| 旧模型 | 新模型（Go治理） | Rust落地 |
| --- | --- | --- |
| Admin | PlatformAdmin/RiskOfficer/Auditor | 统一落到 `Admin`，由 Go 做细分 |
| Brand | BrandAdmin/PartnerOps | 落到 `Brand` + brand_id |
| Factory | FactoryOperator | 落到 `Factory` + device/session claims |
| Owner | Owner（保持） | 保持 `Owner` + wid |

### 3.2 迁移策略

- 双层授权并行一段时间：
  - 外层 Go IAM 先判定。
  - 内层 Rust `require_role` 最终判定。
- 高风险动作（revoke/rotate/recovery complete）必须启用审批前置。

### 3.3 回滚策略

- 若新 IAM 出现异常：
  - 立即退回旧 JWT 签发流程。
  - 保留 Go 审批记录用于追溯，不影响 Rust 业务放行。

## 4. API 层迁移

### 4.1 版本与兼容窗口

- `rc-api` 维持现有 `/v1/*` 不破坏。
- Go 治理层使用 `/ops/v1/*`，对前端提供稳定封装。
- 兼容窗口建议：至少 2 个发布周期。

### 4.2 变更类型

- `兼容新增`：新增可选字段、新增 endpoint。
- `软弃用`：保留旧字段但标记 deprecated。
- `禁止`：直接移除必填字段或改变关键语义。

### 4.3 回滚策略

- API 网关保留“旧路由映射开关”。
- 上游 Rust 异常时，Go 层降级到只读模式（禁写不禁查）。

## 5. 审计层迁移

### 5.1 统一审计模型

- 每次治理动作至少产出一条审计事件，字段最小集：
  - `trace_id`
  - `approval_id`
  - `operator`
  - `action`
  - `target`
  - `result`
  - `timestamp`
- Rust 已有审计（`BusinessAuditStore` / `AuditStore`）继续保留。
- Go 审计仅做补充，不替代 Rust 审计真源。

### 5.2 对账与稽核

- 每日任务对齐 Go 与 Rust 的审计总量、关键动作一致性。
- 不一致触发 P1 工单，冻结高风险操作入口。

### 5.3 回滚策略

- 审计写入失败即中断对应写操作（fail-close）。
- 若审计系统故障，切入“保护模式”：仅允许低风险只读查询。

## 6. 切流策略（灰度）

- 阶段1：内部账号 5% 灰度（只开放品牌与SKU功能）。
- 阶段2：扩大到 30%（开放工厂批次与委托授权）。
- 阶段3：100%（开放转移、恢复、争议闭环）。
- 每阶段必须通过：
  - 错误率门槛
  - 对账门槛
  - 回滚演练门槛

## 7. 风险清单与应对

| 风险 | 触发信号 | 处置 |
| --- | --- | --- |
| 幂等失效导致重复写 | 同工单多次生效 | 强制校验 `X-Idempotency-Key` + payload hash |
| 角色映射错误 | 越权或拒绝合法请求 | 双层授权日志比对，快速回退旧IAM |
| API语义漂移 | 前端行为异常 | 契约测试 + 金丝雀灰度 |
| 审计断链 | trace_id无法串联 | fail-close + 审计通道健康探测 |

## 8. 变更门禁（上线前必须通过）

- 已完成一次“全量前滚 + 全量回滚”演练并有记录。
- 已完成一次“审计系统故障注入”演练。
- 已完成一次“Go治理层故障”降级演练，验证 Rust 核心可独立运作。
