# V5.0 安全与加密专题文档

---

## 一、KMS 密钥管理系统

### 1.1 分层衍生体系 (KDF)

绝不在数据库明文存储芯片密钥，使用密钥衍生函数：

1. **根密钥 (Master Root Key)**：存储在物理加密机 (HSM) 中，任何人无法读取
2. **品牌派生密钥 (Brand Key)**：由 Root_Key 结合 Brand_ID 派生，每个品牌独立密钥空间
3. **芯片终端密钥 (Leaf Key / K2)**：由 Brand_Key 结合 UID 派生
   - 公式：`K_chip = HMAC-SHA256(Brand_Key, UID)`
   - 数据库只需存 UID，不需要存密钥

### 1.2 核心组件

| 组件 | 功能 |
|------|------|
| 安全执行环境 (TEE) | 所有加密运算在隔离内存中完成，对操作系统不可见 |
| 动态指令生成器 | 接收 UID，验证配额，生成 K2，封装加密 APDU 指令流 |
| 确权认证网关 | 接收 UID+CTR+CMAC，派生 K2，对比 CMAC，发放 JWT |

### 1.3 熔断机制

- **计数器回溯拦截**：CTR 小于等于历史记录则判定克隆卡攻击，拉黑请求
- **频率限制**：单个 UID 1分钟内查询超10次则冻结验证功能
- **地理围栏校验**：5分钟内跨洲扫码则触发风控标记

### 1.4 部署架构

- **业务层与加密层分离**：业务云处理 H5/UI/品牌后台，私有 KMS 通过专线接收请求
- **高可用性**：毫秒级响应，Redis 缓存 UID-CTR 映射表，验证逻辑在内存中完成

## 二、母卡防克隆保护协议

### 2.1 三位一体防伪架构

1. **物理层 (PUF 芯片)**：利用半导体生产中微小物理差异，每颗芯片有唯一硅指纹，无法克隆
2. **算法层 (双向动态握手)**：App 发送随机数，母卡内部计算 HMAC，主权密钥永远不出芯片
3. **策略层 (滚动机理)**：母卡内置不可逆计数器，云端检测到流水号回溯或不连续跳变则全局熔断

### 2.2 分级管理体系

| 级别 | 名称 | 持有者 | 核心权限 |
|------|------|--------|----------|
| L0 | 创世母卡 (Genesis) | 品牌方 CEO / 首席法务 | 最高主权：激活和授权下级母卡；重置品牌根密钥 |
| L1 | 区域母卡 (Regional) | 区域负责人 | 分拨权：在授权额度内分配给工厂或仓库 |
| L2 | 点睛母卡 (Operator) | 一线质检员 / 仓库管理员 | 操作权：执行具体的子卡点睛 |

### 2.3 防丢失与挂失流程

- **丢失预防**：App 必须检测母卡物理存在 + 操作员生物特征，双重验证后才下发指令
- **紧急挂失 (Kill Switch)**：管理员在 Sovereign Console 点击挂失，KMS 将母卡公钥加入黑名单，全球失效
- **主权修复**：L0 创世母卡丢失需品牌方 CEO + 技术总监 + 平台方三方数字私钥多签认证，生成新根密钥

## 三、设备端安全接入

### 3.1 双层证书体系

- **L1 生产级设备 (Factory Reader)**：仅具备盲扫权限，证书权限受限且具备过期机制
- **L2 核心级设备 (Brand Activator)**：具备纠缠与定义权限，需绑定品牌方硬件安全密钥

### 3.2 通讯安全

- **mTLS 双向认证**：设备证书 + 私钥签名 + 订单令牌三要素
- **HMAC 校验**：每条 API 请求 Body 包含基于设备私钥的签名
- **时钟同步**：强制时间戳校验（误差正负30秒），防止离线录入后批量重放

### 3.3 证书生命周期

- **任务制证书**：生产订单下达时激活，订单完成或扫描数量达上限后自动吊销
- **紧急吊销**：监控发现异常高频扫描，OCSP 实时阻断设备通讯

## 四、全链路加密传输

### 4.1 通讯层

强制 mTLS，每个终端内置独一无二的设备证书。

### 4.2 数据层

AES-256-GCM 对 API body 二次加密，设备私钥签名 + 平台公钥加密。

### 4.3 密钥派生

`K_tag = CMAC(K_master, UID || System_ID)`，单芯片被破解不影响全局。

### 4.4 SUN 消息

芯片每次扫码用 AES-128 产生动态密文，含自增计数器防重放。

## 五、NFC 动态 URL 校验逻辑

### 5.1 动态 URL 结构

```
https://v5.auth/verify?uid=04A1B2C3D4E5F6&ctr=000005&cmac=8A3F...
```

- `uid`：芯片全球唯一 7 字节序列号（明文）
- `ctr`：芯片内部 3 字节 NFC 计数器，每感应一次 +1
- `cmac`：由芯片使用内部密钥对 uid 和 ctr 进行 AES-128 计算生成的动态密文

### 5.2 后端校验三连检

1. **密钥匹配**：根据 uid 从数据库提取该芯片对应的认证密钥
2. **计数器校验**：ctr 小于等于历史值则判定重放攻击；大于则更新数据库
3. **算法验签**：后端使用相同算法和密钥重新计算 CMAC，与 URL 中的 cmac 比对

### 5.3 后端伪代码

```python
def verify_v5_tag(uid, ctr, cmac_from_url):
    k2 = get_key_by_uid(uid)
    if int(ctr) <= last_recorded_ctr(uid):
        return "REPLAY_ATTACK_DETECTED"
    sv = construct_sdm_vector(uid, ctr)
    expected_cmac = aes_cmac(k2, sv)
    if cmac_from_url == expected_cmac:
        update_last_ctr(uid, ctr)
        return "SUCCESS_GENUINE_ASSET"
    else:
        return "COUNTERFEIT_DETECTED"
```

## 六、安全审计规则

- **Anti-Replay**：每次调用校验 SUN 消息中的计数器，计数值小于等于历史值则判定克隆攻击
- **Geo-Fencing**：盲扫接口记录工厂 IP 范围，非授权经纬度则标记 SUSPICIOUS
- **Entropy Check**：某母卡短时间内纠缠大量子标则封锁该 M-UID

## 七、隐私主权架构

### 7.1 核心原则

V5.0 协议在数据处理上遵循"零知识、物理隔离、主权自持"的最高原则。对于高净值人群而言，"被看见"比"被偷走"更令其恐惧。系统必须在技术架构上主动放弃"窥探权"，以赢得用户的"信任权"。

### 7.2 零知识验证 (Zero-Knowledge Proofs)

- 系统可以验证"某人拥有某资产"，但系统本身不知道"此人是谁"
- 用户生物信息（FaceID/指纹）仅保留在本地设备，云端存储经过哈希脱敏的匿名权属令牌
- 即便数据库被攻破，攻击者只能看到匿名 ID 与资产的映射关系，无法对应到自然人

### 7.3 地理位置"阅后即焚"

- 绝不记录用户轨迹，只验证当下
- 扫码验证时，地理位置仅用于防范异地调包等风险触发，验证通过后立即物理抹除空间坐标
- 系统不记录用户带着资产去了哪里，只证明资产在持有人手中

### 7.4 主权冷存储

- 核心权属密钥不在云端，而在用户手中的母标签硬件中
- 即便平台停止运营，只要用户持有母标签，仍可通过去中心化节点证明资产归属
- 平台定位为"数字化镜像服务商"，不持有用户资产

### 7.5 匿名身份锚点（最小化数据占用）

身份简化为"匿名手机号 + 物理母标签 + 生物特征"的分布式信任网络：

- **匿名手机号**：仅作为单向脱敏的验证码接收通道，数据库存储 SHA-256 加盐哈希值，无法反向推导真实号码
- **品牌方隔离**：品牌方在后台只能看到资产流转状态，绝对看不到用户手机号（包括脱敏版本）
- **匿名中转通讯**：品牌方联系资产持有人必须通过 V5 匿名中转平台，品牌方发消息给平台，平台根据哈希值找到对应通道转发

### 7.6 品牌空投的隐私平衡

品牌方触达用户时必须遵循"克制且优雅"的原则：

- **静默空投**：不发推送、不弹窗，用户进入藏品馆时才看到品牌方投放的内容
- **主权选择权**：用户可关闭特定品牌的空投，但关闭即放弃该品牌的稀缺配额获取资格
- **匿名交互**：品牌方只看到"已向 N 个持有人发送"，用户主动授权前品牌方无法获取任何联系方式
