# 阶梯式防御与主权自愈体系（Graduated Defense & Sovereign Recovery）

## 核心问题

当 CircuitBreaker 检测到异常频率后直接冻结资产（Disputed 状态），这个安全机制本身可能被武器化：

- **地毯式轰炸攻击**：攻击者（竞争对手或黑产）通过分布式脚本对全网已知 UID 进行高频扫描，诱发系统冻结
- **后果**：大量老钱资产无故进入 Disputed 状态，造成用户恐慌，摧毁 RC 协议最核心的价值——"主权的确定性"
- **本质**：安全机制的"一刀切"设计让防御成本转嫁给了被保护者

## 设计原则

> 不要轻易"冻结"，要学会"隐藏"。把解冻变成"特权服务"而非"申诉表"。

## 阶梯式防御策略（Graduated Response）

### 第一阶段：降级解析（Degraded Resolution）

- **触发条件**：同一 UID 在 1 分钟内被扫描超过阈值（默认 20 次）
- **系统行为**：
  - KMS 不再返回详细资产信息和过户接口
  - 返回 `SECURITY_VERIFICATION_PENDING`（"安全验证中"）中间态
  - 资产在二级市场暂时无法交易
  - 用户 H5 私人藏馆依然显示正常
- **资产状态**：不变（仍为 Activated/LegallySold/Transferred 等）
- **SecurityLevel**：从 Normal 升级为 Elevated
- **对攻击者**：看到的是"系统维护"或"解析失败"
- **对持有者**：App 内看到"安全防护中"

### 第二阶段：主动质询（Active Challenge）

- **触发条件**：降级解析持续且异常模式未消退
- **系统行为**：
  - 向当前持有者的 App 推送安全警报："您的资产 [UID] 正在遭受异常扫描，请确认安全"
  - 持有者点击"这是我在展示" → 拦截立即解除，SecurityLevel 回退为 Normal
  - 持有者无响应 → 维持降级状态，不升级冻结
- **SecurityLevel**：Elevated（维持）
- **设计意图**：给持有者"知情权"和"一键解除权"，避免误伤

### 第三阶段：标记存疑 / 冻结（Disputed）

- **触发条件**：仅在以下高置信度异常时才触发
  - 异地并发扫描（同一 UID 同时在不同洲被解析）
  - 密钥碰撞尝试（CMAC 验证失败但 UID/CTR 格式正确，疑似克隆卡）
  - 计数器回滚（CTR 值倒退，物理克隆的铁证）
- **系统行为**：
  - 资产状态转换为 Disputed（协议级冻结）
  - 冻结一切流转操作
  - 同时启动"主权自愈"流程入口
- **SecurityLevel**：Critical

## 主权自愈流程（Sovereign Recovery）

### 设计理念

解冻不是"填申诉表"，而是"开启超级金库"的尊荣仪式。通过"人-物-权"三位一体验证，既确保安全性，又给老钱提供"我的资产被最高等级保护"的体验。

### 三重验证（Triple Verification）

#### 1. 物理证明（The Object）
- 用户必须感应该资产的物理 NFC 标签
- 证明实物在手，排除远程攻击
- 系统验证 CMAC 有效性 + CTR 严格递增

#### 2. 生物核验（The Person）
- 调用手机端生物识别（FaceID / 指纹）
- 确保是主权持有者本人操作
- 生物特征不上传服务器，仅本地验证 + 签名证明

#### 3. 母卡授权（The Key）
- 感应母卡（最高权限的物理体现）
- 母卡 CMAC 验证通过
- 证明持有者拥有该资产的"主权钥匙"

### 解冻执行

三项验证全部通过后：
1. rc-kms 通过 `validate_transition(Disputed → 原状态)` 验证合法性
2. 可选：派生新的 Epoch Salt 进行密钥轮换（增强安全性）
3. 资产状态回滚至冻结前的原状态
4. SecurityLevel 重置为 Normal
5. 记录完整的审计日志（含三重验证证据）

## 攻击成本补偿机制

### 攻击源隔离
- 自动屏蔽攻击源的所有 IP / 设备指纹
- 对已识别的攻击源返回"蜜罐数据"（伪造的验证结果），让攻击者在虚假数据中"自嗨"
- 真实用户完全无感

### 蜜罐响应（Honeypot Response）
- 当检测到攻击模式时，不返回错误（避免暴露防御逻辑）
- 返回看似正常但实际为伪造的验证结果
- 攻击者无法区分真实响应和蜜罐响应，增加攻击成本

## 数据模型扩展

### SecurityLevel 枚举
```
Normal(0)      — 正常状态
Elevated(1)    — 降级解析中（第一/二阶段）
Critical(2)    — 已冻结（第三阶段，Disputed）
```

### AssetRecord 扩展字段
- `security_level`: SecurityLevel 枚举值
- `security_elevated_at`: 进入 Elevated 的时间戳（用于超时自动回退）
- `challenge_sent_at`: 主动质询推送时间戳
- `challenge_acknowledged`: 持有者是否已确认（布尔值）
- `disputed_reason`: 冻结原因（仅 Critical 时有值）

### RecoveryRecord（主权自愈记录）
- `recovery_id`: UUID
- `uid`: 资产 UID
- `object_verified`: 物理标签验证通过（布尔值 + 时间戳）
- `biometric_verified`: 生物核验通过（布尔值 + 时间戳）
- `mother_card_verified`: 母卡授权通过（布尔值 + 时间戳）
- `previous_state`: 冻结前的原状态
- `recovered_at`: 解冻完成时间戳
- `epoch_rotated`: 是否执行了密钥轮换

## 与现有 Disputed 状态的关系

现有的 Disputed 状态保持不变（AssetState 枚举值 13），但触发条件从"任何 CircuitBreaker 规则触发"收紧为"仅高置信度异常触发"。新增的 SecurityLevel 字段提供了 Disputed 之前的缓冲层（Elevated），避免"一刀切"冻结。

## 实施优先级

1. **MVP 阶段**：保持现有 Disputed 机制，但收紧触发条件（仅异地并发 + 计数器回滚 + 密钥碰撞）
2. **V5.1**：实现 SecurityLevel + 降级解析 + 主动质询
3. **V5.2**：实现完整的主权自愈流程（人-物-权三重验证）
4. **V5.3**：实现蜜罐响应 + 攻击源隔离
