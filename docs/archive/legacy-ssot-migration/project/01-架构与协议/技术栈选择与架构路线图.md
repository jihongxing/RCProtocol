# V5.0 技术栈选择与架构路线图

---

## 一、技术栈决策结论

首选 Python 进行协议打磨与 MVP 开发，成熟后将核心加密模块向 Golang/Rust 迁移。

---

## 二、三维度权衡分析

| 维度 | Python（首选） | Golang / Rust | C++ / C# |
|------|---------------|---------------|-----------|
| 可实现性 | 极高。成熟的 pyscard 和 crypto 库，处理 NTAG 424 的 APDU 指令流最快 | 中。交叉编译方便，但 NFC 底层库的调试成本高 | 低。开发周期长，语法冗余，不适合快速迭代 |
| 安全性 | 高。逻辑清晰，易于审计。核心密钥可调用本地安全模块（HSM） | 极高。内存安全（Rust），编译后的二进制文件极难被反编译 | 中。容易出现内存泄漏或溢出风险 |
| 性能 | 足够。NFC 读写受限于物理感应速度（毫秒级），Python 的延迟可忽略 | 优秀。适合处理高并发的云端确权请求 | 顶级。但对于当前硬件交互场景属于性能过剩 |

## 三、第一阶段选择 Python 的核心理由

### 3.1 APDU 指令调试的灵活性

NTAG 424 DNA 的交互涉及复杂的 AES-128 三轮互认证。开发阶段需要频繁修改指令序列（APDU Commands），Python 的脚本特性支持秒级修改与测试，编译型语言会显著拖慢迭代速度。

### 3.2 生态链的完整性

- `pyscard`：目前与 ACR122U 兼容性最好的库，能直接操作 PC/SC 智能卡读写器
- `cryptography`：工业级加密库，完整支持 V5 协议所需的 CMAC、CBC 模式等数学计算

### 3.3 原型即产品

Python 可快速集成 Web 框架（Flask/FastAPI），实现扫码验真的同时后台实时输出加密握手日志，适合面向品牌方的 Demo 演示。

## 六、硬件自检代码参考

以下为 V5 协议的硬件状态自检程序，用于验证 ACR122U 读写器与 NTAG 424 DNA 标签的连接状态：

```python
from smartcard.System import readers
from smartcard.util import toHexString

def check_v5_hardware():
    # 1. 获取系统中所有已连接的读写器
    r = readers()
    print(f"[*] 正在扫描 V5 硬件...")

    if len(r) == 0:
        print("[!] 错误：未发现 ACR122U 读写器，请检查 USB 连接。")
        return

    reader = r[0]
    print(f"[+] 发现读写器: {reader}")

    # 2. 尝试连接读写器
    connection = reader.createConnection()
    try:
        connection.connect()
        # 获取卡片的 ATR (Answer To Reset) - 卡片的身份标识
        print(f"[+] 硬件握手成功！卡片 ATR: {toHexString(connection.getATR())}")
        print("[*] 状态：V5 创世节点已就绪。")
    except Exception as e:
        print(f"[-] 读写器在线，但未发现 NTAG 424 DNA 标签。")

if __name__ == "__main__":
    check_v5_hardware()
```
