# 当前实现与目标协议差距分析

> 文档类型：Foundation  
> 状态：Active  
> 权威级别：Authoritative  
> 最后更新：2026-04-09

---

## 1. 文档目的

本文件用于正面澄清 RCProtocol 当前代码实现与目标需求之间的关键理解差异。

本文件不讨论“是否能先做 MVP”，而讨论一个更根本的问题：

**当前工程是否已经在构建“协议”，还是仍然主要在构建“中心化验真平台”。**

结论先行：

> 当前工程已经具备协议内核雏形，但整体系统仍然属于“中心化平台 + 密码学验真能力”的架构，尚未达到“平台与品牌方共同持有信任根、协议可脱离单一外部系统成立”的目标状态。

---

## 2. 目标需求的严格定义

根据当前需求，RCProtocol 的目标不是“一个更强的防伪后台”，而是一个满足以下条件的协议系统。

### 2.1 目标一：协议独立性

真品成立条件不能依赖某个平台数据库中是否存在一条记录。

协议应当满足：

- 实物标签真实性由密码学证明支撑
- 资产合法发行由协议承诺支撑
- 平台数据库可以做索引、缓存、审计、投影
- 平台数据库不能成为“真品成立”的最终裁判

### 2.2 目标二：平台与品牌方共同信任根

验真权不能被平台单独垄断，也不能被品牌方单独垄断。

协议应当满足：

- 平台持有独立秘密份额
- 品牌方持有独立秘密份额
- 任何一方单独都不能完成最终验真
- 只有平台与品牌方共同参与，才能生成或恢复最终可用的验真能力

### 2.3 目标三：第三方不可伪造

协议的安全边界应当建立在密码学与硬件边界上，而不是建立在“攻击者拿不到平台数据库”这种假设上。

协议应当满足：

- 第三方无法通过接口仿真、数据库泄露、流量重放、普通软件逆向来构造“真品证明”
- 第三方无法绕开品牌与平台的共同信任根
- 协议最终接受的剩余攻击面应尽量收敛为“一换一”

### 2.4 目标四：可接受的最终攻击面

系统可以接受的主要剩余攻击面是：

- 攻击者将真实标签从真品上拆下，再贴到假货上

系统不应接受的攻击面包括：

- 批量复制标签后仍能通过验真
- 不持有共同密钥的一方单独构造验真结果
- 仅凭平台数据库记录即可伪造真品存在性

## 3. 当前实现的事实基线

本节描述当前仓库所体现的真实现状。

### 3.1 已具备的协议内核能力

当前实现已经具备以下强协议特征：

- 基于 `NTAG 424 DNA` 的 `UID + CTR + CMAC` 动态认证链路
- Rust 密码学能力与测试向量
- KMS 抽象与品牌维度派生能力
- 资产状态机、审计、风控与过户流程
- 虚拟母卡 / 物理母卡双授权设计
- 对重放、冲突、风险动作的显式处理

这说明当前项目不是纯 CRUD 平台，而是已经有真实的协议型安全内核。

### 3.2 当前验真链路的真实依赖

当前验真并非“完全由协议自洽成立”，而是依赖如下链路：

1. 扫描芯片得到 `uid / ctr / cmac`
2. 先到中心化数据库通过 `uid` 查询 `assets`
3. 从数据库取得 `brand_id / epoch / current_state`
4. 使用平台 KMS 派生 `chip key`
5. 执行 `CMAC` 校验
6. 使用数据库中的 `last_verified_ctr` 与资产状态做 replay / 风控判断
7. 输出验真结果，并写入审计事件

结论：

> 当前验真流程虽然使用密码学，但仍然显著依赖中心化数据库与平台 KMS。

### 3.3 当前资产真相源

当前系统中的以下核心事实主要由平台数据库承载：

- 资产是否存在
- 资产属于哪个品牌
- 当前状态是什么
- 当前 owner 是谁
- 是否已售出、已转移、被冻结、已恢复
- 过户请求是否 pending / confirmed / rejected
- 虚拟母卡凭证与部分运营治理信息

结论：

> 当前系统的资产真相源仍然是平台数据库，而不是协议承诺本身。

### 3.4 当前密钥结构的真实边界

当前安全模型采用：

- `Root_Key -> Brand_Key -> K_chip_child / K_chip_mother / K_honey`

这说明系统已经具备品牌隔离与分层密钥派生能力。

但当前模型中：

- 平台掌握 root key
- brand key 由平台侧 KMS 派生
- 品牌方是否掌握独立秘密份额，并不是当前实现的必要条件

结论：

> 当前模型是“平台主导派生的品牌隔离模型”，不是“平台与品牌方共同持密的双边信任根模型”。

## 4. 当前实现与目标需求的差距矩阵

| 目标维度 | 目标要求 | 当前实现 | 判断 |
|------|------|------|------|
| 协议独立性 | 真品成立不依赖平台数据库单点 | 验真第一步即查 `assets` 表，状态与上下文依赖 PostgreSQL | 不满足 |
| 双边共钥 | 平台与品牌任一方都不能单独验真 | 平台 KMS 可独立完成品牌级派生 | 不满足 |
| 第三方不可伪造 | 伪造必须同时突破双边信任根与硬件边界 | 已有芯片与 CMAC 边界，但平台仍是单侧主控 | 部分满足 |
| 数据库角色 | 数据库仅做索引 / 审计 / 缓存 / 投影 | 数据库仍承载资产存在性、状态、owner、transfer 真相 | 不满足 |
| 品牌角色 | 品牌是协议共同信任根之一 | 品牌目前更像命名空间与业务主体，不是密码学共同根 | 不满足 |
| 一换一攻击收敛 | 系统唯一主要剩余攻击面是一换一 | 芯片层接近，但系统层仍受平台真相源约束 | 部分满足 |
| 平台定位 | 平台是协议参与者之一，不是绝对裁判 | 当前平台兼任协议执行者、密钥主控者、数据真相源 | 不满足 |

## 5. 当前方向中“正确”的部分

以下方向应当保留，不应被误判为问题。

### 5.1 芯片动态认证方向是正确的

`NTAG 424 DNA`、`SUN/SDM`、`CMAC`、`CTR` 防回放，这些是协议安全内核的重要组成部分，应继续加强而不是削弱。

### 5.2 Rust 协议核心收敛方向是正确的

将资产生命周期、验真、风险动作、过户、审计集中到统一 Rust 核心服务，是在为协议收敛铺路，而不是在偏离方向。

### 5.3 数据库做工程承载是合理的

PostgreSQL、Redis、BFF、Gateway、IAM、Workorder 作为工程支撑并不是问题。

问题不在于“用了数据库”，而在于：

- 数据库现在是不是最终真相源
- 平台现在是不是单方信任根

---

## 6. 当前方向中“危险”的部分

以下偏差如果不及时修正，项目会稳定滑向“验真 SaaS 平台”，而不是“协议”。

### 6.1 把数据库记录当成真品本体

如果资产存在性必须先由平台库确认，数据库就不是“辅助设施”，而是“主裁判”。

这会带来三个后果：

1. 协议不能脱离平台成立
2. 品牌方无法成为真正的共同信任根
3. 攻击面不再只集中于硬件与密码学，而会重新回到平台系统控制权

### 6.2 把品牌隔离误当成品牌共钥

当前 `brand_id` 参与派生，不等于品牌方参与持密。

“品牌隔离”与“品牌共同信任根”不是一回事：

- 品牌隔离：平台能为不同品牌算不同 key
- 品牌共钥：平台单独算不出最终可用 key

如果这一点不澄清，项目会在概念上长期自我误导。

### 6.3 把托管后台合理性误当成协议合理性

平台提供品牌后台、API Key、SKU 托管、工单与运营闭环，这些都可以作为商业化能力存在。

但这些能力不能倒过来定义协议本体。

正确关系应是：

- 协议先定义真伪、发行、所有权与授权事实
- 平台后台只是协议的运营实现层

而不是：

- 平台后台先持有资产真相
- 协议只是在后台之下提供一层验真技术

## 7. 必须立即统一的几个定义

### 7.1 “真品成立”定义

当前项目必须统一一句话定义：

**什么条件下，一个资产可以被判定为协议上的真品。**

建议禁止再使用模糊表达，例如：

- “数据库里有记录就是真品”
- “能扫出来就是真品”
- “平台验过就是真品”

这些定义都会把协议重新拉回平台中心化真相。

### 7.2 “品牌参与”定义

当前项目必须统一一句话定义：

**品牌方到底是业务参与方，还是密码学共同信任根。**

如果品牌只是业务参与方，那么项目应明确承认当前是平台主控系统。

如果品牌是密码学共同信任根，那么后续架构必须落实：

- 品牌独立秘密份额
- 品牌独立签发或参与签发
- 品牌不在线或不协同时，平台单方面不能完成完整验真链路

### 7.3 “数据库角色”定义

当前项目必须统一一句话定义：

**数据库在协议中到底是“真相源”还是“投影层”。**

如果继续模糊，工程实现会自然把所有关键语义都沉到库表结构里。

## 8. 对当前项目的正式判断

### 8.1 当前项目不属于“完全跑偏”

原因：

- 已有真实的芯片密码学基础
- 已有状态机、KMS、审计与风险动作抽象
- 文档中已经多次强调“治理层不能复制核心业务状态机”
- 项目内核明显有协议化倾向

### 8.2 当前项目也不属于“已经实现目标协议”

原因：

- 真品成立仍依赖平台数据库 `assets`
- 平台仍是密钥派生与验真链路的主控方
- 品牌尚未成为不可替代的共同信任根
- 数据库仍承担资产存在性、状态、owner 与 transfer 的最终真相

### 8.3 当前项目的准确定位

当前 RCProtocol 的更准确定位应当是：

> 一个以协议安全内核为基础、以中心化平台实现为当前承载方式的 MVP 系统。

它不是纯平台 CRUD 系统；但它也还不是终局意义上的“协议”。

---

## 9. 接下来所有设计与开发的硬约束

从本文件生效起，后续设计讨论必须遵循以下约束：

1. 不得再把“品牌隔离”表述成“品牌共同持密”
2. 不得再把“平台数据库中的资产记录”表述成“协议独立成立”
3. 不得再把“后台可运营”表述成“协议真相已收敛”
4. 任何涉及验真、发行、所有权、转移、授权的设计，都必须先回答：
   - 真相源是什么
   - 共同信任根是谁
   - 平台单独能否完成该动作
   - 品牌单独能否完成该动作
5. 如果某项设计不能回答上述问题，则该设计不得进入实现阶段
