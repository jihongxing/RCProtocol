---

## 1. 文档目的

本文件用于封口 RCProtocol 中最容易被混用、误用、偷换概念的核心术语。

后续所有需求讨论、架构讨论、spec、roadmap、代码设计与对外表述，凡涉及以下术语，均必须以本文件为准。

本文件的目标不是扩展概念，而是**压缩歧义**。

### 2.1 只能按本文件含义使用

以下术语只允许按本文件定义使用：

- 协议
- 平台
- 真品
- 验真
- 品牌参与
- 品牌隔离
- 品牌共同信任根
- 数据库真相源

- 承诺
- `AssetCommitment`
- 资产所有权
- 资产转移
- 一换一攻击

### 2.2 禁止模糊替代

禁止使用模糊说法替代正式定义，例如：

- “差不多就是协议”

- “品牌现在也算参与密钥了”
- “数据库只是存一下，虽然它也决定真伪”

这些说法都会造成实现和需求继续偏离。

## 3. 基础术语定义

### 3.1 协议（Protocol）

**协议** 是一套可验证、可重复、可独立表达真相的规则集合。

在 RCProtocol 中，“协议”必须至少满足：

1. 真品成立条件有明确数学或密码学定义
2. 关键参与方职责明确
3. 真相不能由单一后台系统随意裁定
4. 第三方可依据协议规则理解真品为何成立

以下情况不能称为“协议已经成立”：

- 只有平台数据库记录
- 只有平台后台可以解释规则
- 只有平台单方持密即可给出完整真品结论

### 3.2 平台（Platform）

**平台** 是 RCProtocol 的运行、接入、治理、聚合、审计与运维承载者。

平台可以提供：

- Gateway
- IAM
- BFF
- 后台管理
- 审计与风控
- 品牌 API 接入
- 数据库存储与缓存

但平台不应被定义为：

- 协议真相本体
- 真品定义本身
- 唯一信任根

### 3.3 品牌方（Brand）

**品牌方** 是资产发行与品牌声明的一方主体。

品牌方至少有两种可能角色：

1. **业务参与方**：在平台中注册、调用 API、管理品牌资产
2. **协议共同信任根**：对资产承诺承担不可替代的密码学职责

必须明确区分这两种角色，不能混用。

## 4. 真伪与验真相关术语

### 4.1 真品（Authentic Asset）

在 RCProtocol 终局定义中，**真品** 不是“数据库中存在的一条资产记录”，而是同时满足以下条件的资产：

1. 物理标签认证通过
2. 品牌发行承诺存在且有效
3. 平台共同承诺存在且有效
4. 品牌承诺与平台承诺绑定到同一 `AssetCommitment`

因此以下情况都不能单独定义真品：

- 平台数据库有记录
- 芯片能扫出 UID
- 平台后台显示“已激活”
- 品牌后台说“这是我发的”

### 4.2 验真（Verification）

**验真** 是根据协议规则验证某资产是否满足真品成立条件的过程。

终局验真应至少检查：

- 标签动态证明
- 品牌承诺
- 平台承诺
- 风险与状态约束

### 4.3 当前 MVP 验真链路

当前 MVP 的验真链路是：

- 读取 `uid / ctr / cmac`
- 查询平台数据库中的资产上下文
- 由平台侧 KMS 派生芯片密钥
- 校验 CMAC 与 CTR
- 基于数据库状态与风控规则输出结果

这应被称为：

> **当前可运行的 MVP 验真实现**

而不是：

> **终局协议验真定义**

## 5. 品牌相关术语

### 5.1 品牌隔离（Brand Isolation）

**品牌隔离** 指不同品牌处于彼此隔离的命名空间、密钥空间、数据权限空间中。

例如：

- 不同品牌拥有不同 API Key
- 不同品牌的资产不能互相访问
- 平台侧根据 `brand_id` 派生不同 `Brand_Key`

品牌隔离解决的是：

- 多品牌共用同一平台时的边界问题

品牌隔离不等于：

- 品牌共同持密
- 品牌成为共同信任根
- 平台单独无法验真

### 5.2 品牌参与（Brand Participation）

**品牌参与** 指品牌方在业务或协议流程中发挥作用。

品牌参与可能只是：

- 注册品牌
- 调用 API
- 审批激活
- 配置外部 SKU
- 查看审计记录

如果只是上述参与，则品牌仍然只是业务参与方。

### 5.3 品牌共同信任根（Brand as Co-Root of Trust）

**品牌共同信任根** 指品牌方不是只做业务动作，而是承担不可替代的密码学职责。

成立条件是：

- 品牌方持有独立秘密份额或独立签名权
- 没有品牌参与，平台单方面不能完成完整真品证明
- 品牌承诺是协议真品定义的必要条件之一

这一定义远强于“品牌参与”。

## 6. 数据库相关术语

### 6.1 数据库真相源（Database as Truth Source）

**数据库真相源** 指系统在逻辑上依赖数据库中的记录来最终决定某个事实是否成立。

典型表现包括：

- 资产存在性由 `assets` 表中的一行决定
- 验真第一步先查数据库，不存在就直接判定不成立
- 所有权、转移、状态合法性本体由数据库字段承载

### 6.2 数据库投影层（Database as Projection Layer）

**数据库投影层** 指数据库保存的是协议事实的可查询表示，而不是协议真相本体。

典型职责包括：

- 索引
- 列表查询
- 审计回放
- 缓存恢复
- 快照聚合

### 6.3 当前数据库角色

当前 RCProtocol 中，数据库仍然明显承担了大量真相源角色。

因此当前只能表述为：

> 当前数据库是主要真相承载层之一

不能表述为：

> 当前数据库已经只是投影层

---

## 7. 承诺相关术语

### 7.1 承诺（Attestation / Commitment）

**承诺** 是某参与方向某个协议对象给出的不可抵赖声明。

承诺应至少满足：

- 指向明确对象
- 可验证来源
- 不可随意伪造
- 能被后续验真流程引用

### 7.2 `AssetCommitment`

`AssetCommitment` 是 RCProtocol 终局应围绕收敛的统一协议对象。

建议最小构成包括：

- `brand_id`
- `asset_uid` 或协议级唯一资产标识
- `chip_binding`
- `epoch`
- `metadata_hash`

可表达为：

```text
AssetCommitment = H(brand_id || asset_uid || chip_binding || epoch || metadata_hash)
```

### 7.3 为什么必须有 `AssetCommitment`

因为如果没有统一承诺对象，系统就会继续围绕：

- 数据库主键
- 资产表行
- 各类散落字段

来表达真相，最终导致协议永远不能脱离平台数据库定义。

---

## 8. 所有权与转移相关术语

### 8.1 资产所有权（Ownership）

**资产所有权** 是协议中关于“谁是当前合法持有人”的事实表达。

当前实现中，`owner_id` 是主要承载字段。

终局要求中，`owner_id` 应逐步降级为：

- 当前状态投影
- 查询友好表示

真正的权属合法性应由：

- 协议状态机
- 授权证明
- 转移事件
- 必要时的承诺或确认规则

共同决定。

### 8.2 资产转移（Transfer）

**资产转移** 是经过协议授权校验后发生的状态变化。

终局中：

- `asset_transfers` 应被理解为协议动作的审计与查询视图

不得倒置为：

- 先写一条 transfer 记录，所以协议认为已转移

## 9. 攻击面相关术语

### 9.1 一换一攻击（One-to-One Replacement Attack）

**一换一攻击** 指攻击者将真实标签从真品上拆下，再贴到假货上。

这是物理世界中最难完全消灭、但可以通过工艺和运营显著抬高成本的攻击面。

### 9.2 当前安全边界

当前 RCProtocol 可以合理宣称已经在对抗：

- 普通复制标签
- 重放动态消息
- 普通接口回放
- 越权状态推进

### 9.3 当前不能宣称的终局结论

当前不能宣称：

- 系统已只剩一换一攻击面
- 数据库泄露对真相已经无决定性影响
- 平台单独绝对无法构造完整真品结论

## 10. 常见错误表述与正确改写

### 10.1 错误：我们已经是协议了

正确改写：

> 我们已经具备协议内核，但当前仍是平台承载的 MVP 形态，终局协议尚未落地。

### 10.2 错误：品牌现在已经参与共同密钥

正确改写：

> 当前实现是品牌隔离密钥模型，不等于品牌共同持密模型。

### 10.3 错误：数据库只是存储，不影响真相

正确改写：

> 当前数据库仍然承载大量关键真相；终局目标才是将其降级为投影层。

### 10.4 错误：API Key 已经代表品牌成为协议参与方

正确改写：

> API Key 代表品牌已获得平台接入能力，不等于品牌已成为协议共同信任根。

### 10.5 错误：现在已经只剩一换一攻击

正确改写：

> 当前芯片安全边界在向一换一攻击收敛，但系统层尚未完成终局协议安全收敛。

## 11. 文档与实现约束

自本文件生效起，后续任何需求、spec、roadmap、架构设计、代码设计都必须遵守以下约束：

1. 使用“协议”一词时，必须说明是当前 MVP 协议内核，还是终局协议定义
2. 使用“品牌参与”一词时，必须说明是业务参与还是共同信任根参与
3. 使用“数据库”一词时，必须说明当前是承载层，还是目标投影层
4. 使用“验真”一词时，必须说明讨论的是当前实现链路，还是终局真品定义
5. 任何故意省略这些边界的表述，都视为不合格表述
