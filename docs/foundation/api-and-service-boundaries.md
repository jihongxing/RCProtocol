# 服务与 API 边界

> 文档类型：Foundation  
> 状态：Active  
> 权威级别：Authoritative  
> 最后更新：2026-04-09（对齐协议术语、终局架构与平台/协议边界）

---

## 1. 权威说明

本文件定义 RCProtocol 的服务边界、职责分工、调用约束、接口分类与执行规则。

本文件必须同时回答两类问题：

1. **当前 MVP 中，哪些能力由哪个服务负责**
2. **当前服务边界不应被误读成终局协议已经落地了什么**

凡涉及以下问题，均以本文件为准：

- 某项能力由哪个服务负责
- 哪些动作属于协议核心真源
- 治理层如何调用协议层
- 当前 API 与未来协议承诺机制如何分层
- 写操作必须满足哪些前置条件

---

## 2. 总体边界原则

### 2.1 当前协议核心是真源执行层

以下能力只能由协议核心负责：

- 状态机推进
- 权限校验
- 当前密钥派生
- 动态认证
- 冻结 / 恢复 / 终态推进
- 协议级审计事实记录

注意：

> 当前“协议核心是真源执行层”指的是当前工程实现中的执行真源，不等于终局协议真相已经摆脱数据库承载。

### 2.2 治理层不承载协议执行真源

治理层可以承担：

- 草稿
- 组织岗位
- 策略编排
- 工单推进
- 聚合查询
- 报表视图
- 品牌 Web 后台操作转发

但治理层不得：

- 直接写协议核心表替代状态流转
- 绕过协议核心推进资产状态
- 复制一套独立的状态真相

### 2.3 当前与终局必须区分

当前可以成立的说法：

- 写操作必须收敛到 Rust Core
- Gateway / IAM / BFF / Workorder 不能复制状态机
- 查询允许聚合与投影

当前不能成立的说法：

- 因为写操作收敛到 Rust Core，所以终局协议已经成立
- 因为 Go 层不复制状态机，所以平台已经不是中心化真相承载方
- 因为有品牌 API，所以品牌已经成为共同信任根

### 2.4 资源中心原则

接口设计以资源与协议动作为中心，而不是以前端页面为中心。

当前核心资源包括：

- Brand
- Asset
- AuthorityDevice
- Batch
- Session
- Workorder
- Wallet Snapshot
- ApiKey
- Webhook

终局新增的关键协议对象包括：

- AssetCommitment
- BrandAttestation
- PlatformAttestation

## 3. 服务划分

### 3.1 Rust 协议核心

Rust 协议核心承担：

- 状态机校验与推进
- 权限矩阵校验
- 标签动态认证
- 安全判定
- 资产与品牌核心写操作
- 授权绑定与母卡分层校验
- 协议级审计事实写入

当前边界：

- 它是唯一协议执行真源
- 但其执行仍然显著依赖平台数据库中的资产上下文与平台侧 KMS

终局要求：

- Rust Core 应逐步围绕 `AssetCommitment` 执行
- 验真不应永久依赖“先查资产表，再决定真品是否成立”

### 3.2 Rust KMS

Rust KMS 当前承担：

- Root / Brand / Chip 级密钥派生
- 安全敏感运算
- 认证辅助能力
- 风险相关密钥辅助逻辑

当前边界：

- 当前 KMS 是平台主导派生模型的一部分

终局要求：

- 应演进为品牌与平台共同参与的模型
- 近期优先支持承诺与签名相关能力
- 远期可升级到联合派生 / MPC / HSM 协作

### 3.3 Go Gateway / BFF

Go Gateway / BFF 承担：

- 前端接入聚合
- 会话透传
- 跨服务响应拼装
- 错误标准化映射
- 品牌 API Key 认证与上下文映射

它不负责：

- 协议规则最终判定
- 状态机推进
- 真品最终定义

### 3.4 Go IAM

Go IAM 承担：

- 组织结构
- 岗位模板
- RBAC / ABAC 治理策略
- 治理身份映射
- 品牌 API Key 签发与管理

它负责“治理侧谁能发起什么”，不负责“协议侧最终是否成立”。

### 3.5 Go Workorder / Policy / Approval

这些服务承担：

- 工单编排
- 人工决策上下文
- 策略版本管理（后续）

MVP 阶段说明：

- 品牌方在自己的系统中完成审批
- `go-approval` 不是 MVP 必需服务
- 这些服务属于治理层，不是协议真相源

### 3.6 Reporting / Audit Projection

这些服务承担：

- 审计检索
- 报表聚合
- 投影查询
- 看板汇总

它们可以读真源、建投影，但不能反向覆盖真源。

## 5. API 分类

### 5.1 Public Verification API

用途：

- 验真
- 基础公开信息查询

特点：

- 尽量只读
- 限流优先
- 无需暴露治理信息

当前边界：

- 当前 Public Verification API 返回的是 MVP 验真结果
- 它不代表终局真品定义已经完全按双边共同信任根执行

### 5.2 Core Business API

用途：

- 品牌注册
- 盲扫登记
- 资产激活
- 售出合法化
- 过户
- 冻结 / 恢复

特点：

- 必须经过 Rust Core
- 有严格状态校验
- 需要角色与边界判定

### 5.3 Brand Integration API

用途：

- 品牌方通过 API Key 直接调用的对外接口
- 适用于有技术团队、已有 ERP/OMS 系统的品牌

当前意义：

- 它代表品牌可以接入平台系统
- 它不代表品牌已经承担协议承诺职责

### 5.4 Governance API

用途：

- 工单推进
- 组织与角色治理
- 策略配置

特点：

- 不直接写协议真相
- 必要时调用 Core Business API 落地

### 5.5 Admin / Ops API

用途：

- 快照重建
- 校准
- 审计检索
- 运维修复

约束：

- 只能影响工程承载与恢复过程
- 不得偷偷替代协议真相定义

## 6. 当前关键边界判断

### 6.1 当前可以下的结论

当前可以明确下的结论是：

- Rust Core 是执行真源
- Go 治理层不是执行真源
- 品牌可通过 API Key 或 Web 后台接入
- 写操作收敛规则已经比较清晰

### 6.2 当前不能下的结论

当前不能下的结论是：

- 品牌已成为协议共同信任根
- 终局验真已经落地
- 数据库已降级为纯投影层
- `asset_id` 与数据库资产记录已经退出真相定义中心

## 7. 从当前 API 到终局协议接口的演进方向

### 7.1 当前接口的主要问题

当前接口多数仍围绕：

- `asset_id`
- `uid`
- 当前资产表行
- 状态字段

展开。

这对 MVP 足够，但对终局协议不够。

### 7.2 后续必须引入的协议对象接口

后续应逐步引入以下对象与接口：

- `AssetCommitment`
- `BrandAttestation`
- `PlatformAttestation`
- `Verification V2`（芯片证明 + 双承诺校验）

### 7.3 推荐分层

推荐把后续接口分成三类：

1. **当前 MVP 兼容接口**
2. **协议承诺对象接口**
3. **终局验真接口**

这样可以在不立刻推翻现有前后端的前提下，逐步切换协议真相模型。

---

## 8. 设计与实现硬约束

自本文件生效起，后续任何接口设计都必须明确回答：

1. 这个接口操作的对象是工程对象，还是协议承诺对象
2. 这个接口依赖的真相源是什么
3. 这个接口是否仍然允许平台单方完成关键判断
4. 这个接口未来如何迁移到 `AssetCommitment` 视角

如果回答不清，则不得进入实现。
