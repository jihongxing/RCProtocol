## 1. 权威说明

本文件定义 RCProtocol 的安全边界、密钥体系、动态认证、授权校验、风控与安全实现约束。

本文件必须同时区分两件事：

1. **当前 MVP 安全基线是什么**
2. **目标协议终局安全模型是什么**

禁止再把两者混写成同一件事。

凡涉及密钥、认证、风控与安全状态的问题，均以本文件为准；凡涉及“当前实现与目标协议的差距”，以 `protocol-gap-analysis.md` 为准；凡涉及“终局协议应当长成什么样”，以 `target-protocol-architecture.md` 为准。

---

## 2. 安全目标

RCProtocol 的安全体系服务于以下目标：

1. 验证物理标签真实性
2. 防止重放、克隆与伪造
3. 保证品牌之间的密钥隔离
4. 保证平台与品牌方的职责边界清晰
5. 支撑冻结、恢复、审计与风险处置
6. 支撑虚拟母卡与物理母卡的分层授权校验
7. 最终将系统主要剩余攻击面收敛到“一换一”

其中第 7 条是终局目标，不代表当前实现已经达成。

---

## 3. 安全模型分层

### 3.1 当前 MVP 安全模型

当前已经稳定落地、并可作为工程实现依据的安全模型包括：

- `NTAG 424 DNA` 动态认证链路
- `UID + CTR + CMAC` 校验
- 平台侧 KMS 密钥派生
- 品牌隔离的密钥空间
- 物理母卡 / 虚拟母卡分层授权
- API Key 安全边界
- 冻结、恢复、审计与风险标记

### 3.2 目标协议终局安全模型

目标终局不是“平台主控的高安全系统”，而是：

- 平台与品牌方共同信任根
- 协议真相优先于数据库真相
- 双边共同参与才能形成最终真品证明
- 数据库只保存证明缓存、投影、索引与审计

### 3.3 当前与终局的关系

当前 MVP 安全模型是终局协议的过渡实现，不是终局本身。

因此必须明确：

- **当前已实现的是品牌隔离，不是品牌共同持密**
- **当前已实现的是平台承载，不是协议脱离平台独立成立**
- **当前已具备芯片级安全内核，不等于已实现终局共同信任根**

## 4. 当前 MVP 密钥模型

### 4.1 当前密钥派生基线

当前实现采用平台侧 KMS 主导的派生链：

```text
Root_Key
  -> Brand_Key = HMAC-SHA256(Root_Key, Brand_ID || System_ID)
  -> K_chip_child = HMAC-SHA256(Brand_Key, UID || Epoch_LE)[..16]
  -> K_chip_mother = HMAC-SHA256(Brand_Key, UID || Epoch_LE || "MOTHER")[..16]
  -> K_honey = HMAC-SHA256(Brand_Key, Serial_BE)
```

### 4.2 当前模型的安全含义

当前模型能保证：

- 不同品牌之间的密钥空间隔离
- 平台可根据品牌、UID、Epoch 派生芯片相关密钥
- 子标签、母卡、Honey Key 可以按用途分层

### 4.3 当前模型的边界

当前模型不能被描述成：

- 平台与品牌共同持密
- 任一方单独无法验真
- 终局协议共同信任根已经落地

原因是：

- 平台掌握 `Root_Key`
- `Brand_Key` 由平台 KMS 派生
- 品牌方是否掌握独立秘密份额，不是当前实现成立的必要条件

结论：

> 当前是“平台主导的品牌隔离密钥模型”，不是“平台与品牌共同持密模型”。

---

## 5. 目标协议终局密钥模型

### 5.1 终局要求

终局协议中的关键验真能力、发行承诺能力或关键派生能力，必须满足：

- 平台单独不能完成
- 品牌方单独不能完成
- 双方共同参与才能形成最终可用证明

### 5.2 可接受的终局实现形态

可以接受以下任一种终局形态：

#### 方案 A：双边共同派生

```text
K_verify = KDF(platform_secret_share, brand_secret_share, uid, epoch, domain)
```

#### 方案 B：双签名承诺

```text
Brand attests AssetCommitment
Platform attests AssetCommitment
Verify = chip proof valid + both attestations valid
```

#### 方案 C：阈值签名 / MPC / HSM 协作

用于高安全版本。

### 5.3 当前推荐演进方向

结合当前工程基线，推荐：

- **近期优先落地双签名承诺模型**
- **远期再升级到联合派生 / MPC / HSM 协作**

这样既能尽快让“品牌是共同信任根之一”变成事实，又不会立刻推翻当前 NTAG 424 DNA 主链路。

## 6. 动态认证基线

### 6.1 当前基线

当前标签认证采用动态消息为核心的校验方式，最小验证元素包括：

- UID
- CTR
- CMAC

### 6.2 Mode A

MVP 阶段可使用明文兼容模式，在 URL 中传递 UID 与 CTR，并对动态消息进行 CMAC 校验。

### 6.3 Mode B

支持加密隐私模式，将 UID / CTR 通过 SDM 加密传输，用于更高隐私等级场景。

### 6.4 当前动态认证的正确表述

当前动态认证可以表述为：

- 标签真实性校验基础已经存在
- 芯片级重放防护已经存在
- 芯片级安全是协议内核的重要组成部分

但不能表述为：

- 仅凭当前 CMAC 链路，终局协议已经成立
- 只要芯片认证通过，就已经不依赖平台数据库
- 只要芯片认证通过，就已经实现共同信任根

---

## 7. 当前 MVP 验真安全链路

当前可运行验真链路的安全含义如下：

1. 读取标签动态消息
2. 根据 `uid` 查询平台资产记录
3. 获取 `brand_id / epoch / current_state / ctr` 等上下文
4. 由平台 KMS 派生 `K_chip`
5. 验证 `CMAC`
6. 使用 `CTR` 做 replay 检查
7. 根据状态与风险规则输出验真结果
8. 写入审计事件

结论：

> 当前验真链路是“芯片密码学 + 平台数据库上下文 + 平台 KMS”的组合安全链路。

它是合理的 MVP 安全实现，但不是终局协议安全定义。

## 8. 分层授权校验模型

协议核心层对母卡形态无感知。校验逻辑根据 `authority_type` 分发到不同校验链路。

### 8.1 物理母卡校验（Physical NFC）

安全锚点：**硬件 AES-128 CMAC 动态签名**

校验流程：
1. UID 必须匹配 `authority_devices.physical_chip_uid`
2. CTR 防重放（必须大于上次记录 `last_known_ctr`）
3. 派生母卡密钥 `K_chip_mother`
4. 验证 CMAC

适用场景：高价值商品。

### 8.2 虚拟母卡校验（Virtual QR / Virtual App）

安全锚点：**账号身份 + 生物识别 + 虚拟凭证**

校验流程：
1. 用户 ID 必须匹配绑定关系
2. 验证虚拟凭证 Token（Hash 比对）
3. 验证生物识别 Token（JWT / WebAuthn）

适用场景：低中价值商品。

### 8.3 正确边界

分层授权校验是当前系统的重要安全能力，但它解决的是：

- 处置动作授权
- 所有权转移授权
- 母卡参与校验

它不自动等同于：

- 真品成立条件
- 品牌与平台共同信任根
- 协议终局验真模型

## 9. API Key 安全模型

### 9.1 用途

API Key 用于品牌方通过 API 对接协议执行层。

### 9.2 当前安全约束

- API Key 由平台签发，存储时必须 Hash 化
- 支持轮换
- API Key 仅授权品牌方自身品牌范围内的操作
- 所有 API Key 调用必须携带 `X-Trace-Id` 和 `X-Idempotency-Key`
- API Key 调用的审计记录与 JWT 调用同等级别

### 9.3 正确边界

API Key 代表：

- 品牌 API 接入能力
- 品牌在平台系统中的已认证调用身份

API Key 不代表：

- 品牌已经成为协议共同信任根
- 品牌已经持有独立密码学秘密份额
- 终局协议的品牌承诺已经落地

## 10. 风险检测规则

### 10.1 计数器回滚

如果接收到的 CTR 小于或等于历史记录，应视为高风险事件，通常意味着重放或克隆尝试。

### 10.2 高频访问

同一资产在异常时间窗口内被高频读取，应触发限流、降级或冻结前检查。

### 10.3 地理围栏

短时间内出现不合理的跨区域认证行为，应触发风控标记或冻结流程。

### 10.4 授权绑定异常

当授权绑定关系与期望关系不匹配时，应拒绝后续处置动作。

### 10.5 虚拟母卡特有风险

- 账号被盗用后可能导致虚拟母卡被滥用
- Token 泄露可能导致凭证被重放
- 虚拟母卡恢复流程需要子标签物理验证 + 生物识别双重确认

## 11. 冻结与终态原则

### 11.1 Disputed

`Disputed` 是冻结治理状态，不等于安全终态。

### 11.2 Tampered

`Tampered` 用于确认物理篡改或物理安全链路断裂后的终态标记。

### 11.3 Compromised

`Compromised` 用于确认已泄露、已失陷或无法恢复信任的终态标记。

---

## 12. 可接受与不可接受的攻击面

### 12.1 当前阶段主要防御目标

当前阶段重点防御：

- 普通复制标签
- 重放动态消息
- 越权调用接口
- 未授权过户或冻结
- 虚拟母卡盗用

### 12.2 终局阶段目标

终局阶段应将系统主要剩余攻击面收敛为：

- 一换一攻击

### 12.3 当前不能宣称已消除的风险

当前不得宣称：

- 平台单方绝无可能构造最终验真结论
- 数据库泄露对协议真相完全没有影响
- 系统已完全只剩一换一攻击面

---

## 13. 安全实现约束

### 13.1 密钥对象要求

- 实现 `Zeroize` / `ZeroizeOnDrop`
- 禁止明文泄漏到日志
- 禁止普通比较方式处理敏感值
- 禁止无约束复制敏感密钥材料
- 虚拟母卡敏感材料在数据库中必须加密存储

### 13.2 比较要求

密码学值比较必须采用常量时间比较。

### 13.3 中间密钥处理

派生链中的中间密钥应尽快清零，避免在非必要范围内驻留内存。

### 13.4 审计要求

所有安全相关动作必须可追踪，包括：

- 冻结
- 恢复
- 认证失败
- 密钥轮换过程
- 高风险策略触发
- 虚拟母卡签发与恢复
- API Key 签发与轮换

---

## 14. 当前明确废止的安全叙事写法

以下表述禁止继续作为当前权威说法：

1. 把品牌隔离直接表述成品牌共同持密
2. 把平台侧 KMS 派生直接表述成双边共同信任根
3. 把数据库中存在资产记录表述成协议已经独立成立
4. 把“芯片动态认证已实现”表述成“终局协议安全模型已实现”
5. 使用纯愿景式“绝对不可克隆”描述替代具体安全边界

## 15. 后续升级约束

后续如升级安全模型，必须明确写出：

1. 当前已实现部分是什么
2. 新增的共同信任根机制是什么
3. 品牌方新增承担的密码学职责是什么
4. 平台单方能力被削弱到什么程度
5. 数据库在新模型中被降级到什么角色

未回答上述问题的安全改动，不得宣称为“协议升级完成”。
