# 目标协议终局架构

## 1. 文档目的

本文件定义 RCProtocol 的目标终局架构。

本文件讨论的不是“当前代码怎么实现”，而是：

**RCProtocol 最终究竟要成为一个什么协议。**

本文件用于统一以下问题：

- 真品成立的最终定义
- 平台、品牌方、标签、用户之间的信任关系
- 平台数据库在终局协议中的正确角色
- 系统接受与不接受的攻击面
- 从当前 MVP 架构向目标协议演进的路线

---

## 2. 终局目标定义

RCProtocol 的终局目标不是“一个由平台托管的高安全验真系统”，而是：

> 一个以物理标签为硬件锚点、以平台与品牌方共同信任根为核心、以密码学证明为真相基础、以数据库为投影与审计设施的实物资产验真与主权协议。

该定义包含四个不可替代的前提：

1. **物理锚点存在**：标签必须是协议安全的一部分，而不是营销二维码
2. **双边信任根存在**：平台与品牌方共同参与真相建立
3. **密码学真相优先**：协议事实优先于数据库事实
4. **数据库降级**：数据库只负责记录、索引、投影、缓存和审计

---

## 3. 终局协议中的核心原则

### 3.1 真相不能由单方系统定义

任何单一参与方都不能单方面定义“这是真品”。

不能接受的情况包括：

- 平台数据库里有记录，所以是真品
- 品牌后台里点了确认，所以是真品
- 某个前端页面显示正常，所以是真品

终局协议要求：

- 真品必须由协议可验证事实成立
- 任一单方系统只能陈述观点，不能成为最终裁判

### 3.2 平台不能单独验真

平台可以提供验真服务，但平台不应拥有“绕过品牌独立秘密份额、单独给出最终真品结论”的能力。

否则 RCProtocol 只是“平台主控系统”。

### 3.3 品牌不能单独验真

品牌方也不应独自构成最终裁判。

否则协议退化为“品牌自证清白”，第三方信任难以成立。

### 3.4 数据库不构成协议真相源

数据库可以保存大量协议相关信息，但这些信息在终局中应属于：

- 证明缓存
- 投影索引
- 可查询视图
- 审计留痕
- 恢复材料

数据库不能在逻辑上成为：

- 真品存在性的唯一依据
- 所有权存在性的唯一依据
- 验真上下文的唯一可信来源

## 4. 终局协议中的参与方与角色

### 4.1 品牌方（Brand）

品牌方是资产发行权与品牌声明权的来源之一。

在终局协议中，品牌方必须承担两种职责：

1. **业务职责**：声明某资产属于该品牌的合法发行范围
2. **密码学职责**：持有不可替代的秘密份额或签名权

品牌方不能只保留业务职责，而把密码学职责全部交给平台。

### 4.2 平台方（Platform）

平台方是协议协调者、执行者、路由者、审计者与公开验证入口提供者。

在终局协议中，平台方必须承担两种职责：

1. **运行职责**：提供接入、API、缓存、索引、BFF、审计、运维与风控设施
2. **密码学职责**：持有独立秘密份额或独立签名权

平台方不能同时独占：

- 资产真相源
- 品牌发行真相
- 全部密钥控制权
- 全部最终裁决权

### 4.3 标签 / 芯片（Tag）

标签是协议的硬件锚点。

其作用不是“携带一个可查询 ID”，而是：

- 持有无法被普通软件复制的芯片级秘密或认证能力
- 对扫描挑战给出动态响应
- 将协议安全从纯数字世界落到物理世界

### 4.4 用户 / 持有人（Holder）

用户不是决定真品与否的信任根，但可以成为：

- 当前持有人事实的接收者
- 转移动作的参与者
- 母卡 / 虚拟母卡授权链路的执行者

## 5. 终局协议中的真品成立条件

### 5.1 一句话定义

终局协议中，一个资产被判定为“协议上的真品”，必须同时满足：

1. **物理标签认证通过**：标签给出的动态证明可被验证
2. **品牌发行承诺存在且有效**：品牌对该资产身份做过不可抵赖的协议级声明
3. **平台共同承诺存在且有效**：平台对该资产协议接入与可验证性做过不可抵赖的共同声明
4. **两类承诺在同一资产承诺对象上达成一致**

只有四者同时满足，资产才成立为协议真品。

### 5.2 推荐的承诺对象

终局协议不应直接把“数据库主键”当成真相对象，而应建立统一的 `Asset Commitment`。

建议承诺对象至少包含：

- `brand_id`
- `asset_uid` 或协议级唯一资产标识
- `chip_binding`
- `epoch`
- 可选的系列 / SKU / 批次摘要
- 可选的发行时间窗

然后形成：

```text
AssetCommitment = H(brand_id || asset_uid || chip_binding || epoch || metadata_hash)
```

品牌承诺与平台承诺都应围绕同一个 `AssetCommitment` 展开。

### 5.3 数据库的正确使用方式

数据库可以保存：

- `AssetCommitment`
- 品牌承诺证明
- 平台承诺证明
- 当前查询索引
- 风控投影
- 审计历史

但数据库中是否有行记录，本身不应构成真品定义。

## 6. 终局协议中的密钥与信任根模型

### 6.1 必须从“平台单侧派生”升级到“双边共同信任根”

终局模型要求：

- 平台持有 `Platform Secret Share`
- 品牌方持有 `Brand Secret Share`
- 最终验真能力、发行承诺能力或关键派生能力必须依赖两者共同参与

### 6.2 允许的实现形态

终局协议不强制某一种具体密码学结构，但必须满足“任一方单独不可完成”的性质。

可以接受的实现包括：

#### 方案 A：双边共同派生

```text
K_verify = KDF(platform_secret_share, brand_secret_share, uid, epoch, domain)
```

约束：

- 平台单独无法恢复 `K_verify`
- 品牌单独无法恢复 `K_verify`
- 只有双方在协议中协作，才能产生最终可用的验证材料

#### 方案 B：双签名承诺

```text
Brand attests AssetCommitment
Platform attests AssetCommitment
Verify = both signatures valid + chip proof valid
```

约束：

- 双签名缺一不可
- 平台单签不构成真品
- 品牌单签也不构成真品

#### 方案 C：阈值签名 / MPC / HSM 协作

适用于后期更高安全等级场景。

要求仍然一致：

- 任何一方单独无法完成最终真品证明

### 6.3 当前推荐过渡策略

对 RCProtocol 当前工程基线而言，推荐采用：

- 近期：**双签名承诺模型** 优先落地
- 远期：在高价值品牌或高安全版本中演进到 **联合派生 / MPC / HSM 协作**

原因：

- 双签名承诺模型更容易在当前架构上落地
- 能先把“品牌是共同信任根”这件事从概念变成事实
- 不会立刻推翻现有 NTAG 424 DNA 的芯片验真主链路

## 7. 终局协议中的验真流程

### 7.1 推荐终局流程

1. 读取标签动态消息：`uid / ctr / cmac / optional encrypted fields`
2. 校验芯片动态证明是否成立
3. 计算或定位对应的 `AssetCommitment`
4. 验证品牌承诺是否存在且有效
5. 验证平台承诺是否存在且有效
6. 验证二者是否绑定到同一 `AssetCommitment`
7. 检查协议状态与风险约束
8. 输出验真结论与风险语义

### 7.2 终局流程中的“查库”意义

终局流程中依然可能查数据库，但查库的含义应变成：

- 获取承诺缓存
- 获取索引映射
- 获取状态投影
- 获取风控历史

而不是：

- 通过数据库决定资产是否存在
- 通过数据库直接决定其是否是真品

---

## 8. 终局协议中的所有权与转移模型

### 8.1 所有权不应只是平台数据库字段

当前 `owner_id` 作为数据库字段可以保留，但在终局协议中应逐步降级为：

- 当前所有权状态投影
- 当前主权绑定视图
- 前端查询友好表示

最终的权属变更合法性应由：

- 协议状态机
- 持有人授权证明
- 母卡 / 虚拟母卡授权证明
- 必要时的品牌 / 平台共同确认规则

共同决定。

### 8.2 转移必须是协议动作，不是后台记录动作

终局要求：

- `transfer` 的本体是经过协议授权的状态变化
- 数据库中的 `asset_transfers` 是该动作的审计与查询投影
- 不能倒置成“数据库先写一条 transfer，所以协议认为已经发生过户”

## 9. 终局协议接受与不接受的攻击面

### 9.1 可接受攻击面

终局协议可以接受的主要剩余攻击面是：

#### 一换一攻击

攻击者把真实标签从真品上拆下，贴到假货上。

这是物理世界中难以完全消除、但可以通过：

- 封装工艺
- 拆毁即毁标签设计
- 绑定包装结构
- 二次人工检查

来降低概率的攻击面。

### 9.2 不可接受攻击面

终局协议不应接受以下攻击面：

1. 仅凭平台数据库泄露即可伪造真品
2. 平台单方持密即可构造完整验真结论
3. 品牌单方自签即可构造完整验真结论
4. 通过重放、复制、接口回放可稳定通过验真
5. 通过普通软件复制可模拟真实标签动态认证

---

## 10. 数据库在终局中的正确定位

### 10.1 可以做的事

数据库可以承担：

- 承诺索引
- 状态投影
- 审计历史
- 查询聚合
- 风控缓存
- 列表分页
- 回放恢复基础设施

### 10.2 不能做的事

数据库不能承担：

- 真品定义本身
- 协议信任根本身
- 单方面决定发行真假的裁决职能
- 单方面决定所有权合法性的最终职能

---

## 11. 平台产品与协议的正确关系

### 11.1 平台可以存在，而且应该存在

协议化不等于不要平台。

RCProtocol 完全可以同时存在：

- 协议层
- 平台层
- 品牌后台层
- 消费者 App
- 品牌 API 接入层

### 11.2 但平台不是协议本体

正确关系应当是：

- 协议定义真相
- 平台提供服务
- 品牌提供发行参与与共同信任根
- 用户通过平台使用协议能力

而不是：

- 平台持有真相
- 协议只是平台下的一层安全能力

## 12. 从当前实现到终局协议的演进路线

### 12.1 Phase 1：先统一定义，不再混用概念

立即完成：

- 统一“品牌隔离”与“品牌共同信任根”的定义差异
- 统一“数据库真相源”与“数据库投影层”的定义差异
- 统一“平台验真系统”与“协议”的定义差异

这是所有后续工程改造的前提。

### 12.2 Phase 2：引入资产承诺对象

新增统一 `AssetCommitment` 概念，并让以下能力逐步围绕它收敛：

- 激活
- 验真
- 外部 SKU 映射
- 风险记录
- 审计日志
- 转移授权

目标是让数据库中的资产记录不再等同于协议本体。

### 12.3 Phase 3：引入品牌承诺证明

品牌方应当对 `AssetCommitment` 给出协议级承诺，例如：

- 品牌签名
- 品牌 HSM 出具的签发证明
- 品牌侧秘密份额参与生成的发行证明

此阶段完成后，平台将不再能单独定义发行合法性。

### 12.4 Phase 4：引入平台共同承诺证明

平台方也应对同一 `AssetCommitment` 给出共同承诺。

此阶段完成后，真品成立条件升级为：

- 标签证明 + 品牌承诺 + 平台承诺

### 12.5 Phase 5：将数据库降级为索引与投影层

完成以下改造：

- 验真结果依赖协议证明对象，不直接依赖资产表行记录存在性
- `owner_id`、`transfer`、`current_state` 从“主定义”转为“投影视图”
- 所有关键查询都可从承诺与事件回放重建

### 12.6 Phase 6：高安全版本再升级到联合派生 / MPC / HSM 协作

对高价值品牌或高安全 SKU，可以进一步升级到：

- 联合派生
- 多方计算
- 云 KMS / HSM 协作
- 品牌侧签发设备独立部署

## 13. 对 roadmap 和工程实现的约束

自本文件生效起，后续 roadmap、spec、系统架构和安全模型需要遵守以下约束：

1. 任何“协议完成度”判断，必须先检查是否已经实现双边共同信任根
2. 任何“验真完成度”判断，必须先检查是否已经实现协议级承诺，而非只完成数据库查验 + CMAC
3. 任何“品牌开放接入”判断，不能把品牌 API 调用能力等同于品牌成为协议共同信任根
4. 任何“去中心化”或“协议化”表述，都必须说明：
   - 真相源是什么
   - 承诺对象是什么
   - 双边共同参与体现在哪里
   - 数据库被降级到什么程度

---

## 14. 终局判断标准

只有当下列条件同时满足时，RCProtocol 才能被认为真正达到了目标协议状态：

1. 真品成立不再依赖平台数据库中的单条资产记录
2. 平台与品牌方任一方单独都不能完成完整验真
3. 标签动态认证、品牌承诺、平台承诺三者共同构成真品证明
4. 数据库可作为索引和投影被重建，而不会导致协议真相丢失
5. 系统主要剩余攻击面收敛为“一换一”
