## 1. 项目定义

RCProtocol 是一套面向高价值实物资产的数字主权协议与平台。

必须明确区分两个层面：

1. **当前工程形态**：平台承载的协议内核 MVP
2. **目标终局形态**：平台与品牌方共同信任根、以协议承诺为真相基础的实物资产协议

因此，本项目当前既不是普通防伪后台，也还不是已经完成终局形态的协议网络。

更准确的说法是：

> RCProtocol 当前是一个以 `NTAG 424 DNA`、状态机、权限模型与密码学认证为安全内核、以平台工程系统为承载方式的协议型 MVP。

---

## 2. 项目想解决的问题

当前高价值实物资产存在四类核心问题：

1. **真伪验证成本高**：传统鉴定依赖人工经验，难以规模化、标准化
2. **权属链路断裂**：售出后品牌、平台与持有者之间缺乏可信的数字关系
3. **二级流转缺少可信公证层**：交易只能证明“像是真的”，难以证明“它在协议中是谁、处于什么状态、由谁合法持有”
4. **品牌与平台缺乏统一治理能力**：工厂、品牌、门店、消费者、审核员之间缺少明确的权限与状态边界

RCProtocol 的目标不是只解决“扫码防伪”，而是把实物资产管理从“内容展示”升级为“协议化主权治理”。

---

## 3. 核心方法

RCProtocol 通过以下方法建立闭环：

- **资产唯一身份**：每个物理标签对应唯一 UID 与协议内身份对象
- **标签动态认证**：通过 `SUN / CMAC / CTR` 等能力验证物理标签真实性
- **授权绑定**：通过母卡 / 授权设备与资产标签的绑定，建立激活与处置前提
- **状态机**：通过统一状态模型管理资产生命周期
- **角色与权限矩阵**：约束平台、品牌、工厂、消费者、审核员等角色边界
- **治理与审计**：所有关键操作可追踪、可幂等、可审计、可冻结、可恢复

当前已经落地的关键方法包括：

- `NTAG 424 DNA` 动态认证链路
- 平台侧 KMS 派生链路
- Rust 核心状态机与审计能力
- 品牌 API / Web 后台双接入模式

终局仍需补齐的关键方法包括：

- `AssetCommitment`
- 品牌承诺证明
- 平台共同承诺证明
- 平台与品牌方共同信任根模型
- 数据库从真相承载层向投影层的降级

## 4. 当前系统边界

### 4.1 协议层

协议层定义：

- 资产身份
- 状态机
- 权限模型
- 当前安全模型
- 动态认证规则
- 风险与冻结规则

### 4.2 产品层

产品层包含两个主要方向：

- **B 端治理系统**：品牌、工厂、后台运营、风控、审计、托管接入
- **C 端资产系统**：验真、资产馆、展示、过户、荣誉态

### 4.3 工程层

工程层以 Rust 核心协议实现为主，Go 承担治理编排，运维层提供缓存、数据库、审计与恢复能力。

必须明确：

- 当前工程层仍然显著依赖平台数据库与平台 KMS
- 这使当前系统具备“协议内核”，但尚未实现“协议独立成立”

---

## 5. 当前确定的实现基线

当前文档体系确认以下实现基线：

- **物理载体基线**：`NTAG 424 DNA`
- **核心密码学基线**：`HMAC-SHA256`、`AES-128 CMAC`
- **状态机基线**：14 态模型
- **角色基线**：Platform / Factory / Brand / Consumer / Moderator
- **核心工程方向**：Rust Core + Go Governance + PostgreSQL + Redis

需要特别说明：

- 当前密钥模型是**平台主导的品牌隔离模型**
- 当前验真模型是**芯片证明 + 平台数据库上下文 + 平台 KMS**
- 当前数据库仍是**关键真相承载层之一**

所以，当前实现基线不能被表述成：

- 平台与品牌已经共同持密
- 真品成立已经脱离平台数据库
- 协议终局已经完成

## 6. 项目核心对象

本项目的核心对象包括：

- **Asset**：被协议管理的实物资产
- **Tag**：附着于资产上的物理识别与认证载体
- **Authority Device / Mother Tag**：发起激活或处置授权的设备或载体
- **AssetCommitment**：终局协议应围绕收敛的统一承诺对象
- **Brand**：资产发行参与方，当前为业务参与方；终局应升级为共同信任根之一
- **Platform**：运行、路由、治理、审计与运维承载方，终局应是共同信任根之一而不是绝对裁判
- **Factory**：负责盲扫登记与入库流程
- **Consumer**：资产合法持有者与流转参与者
- **Moderator**：负责冻结、恢复与安全审计的审核角色

这些对象的正式定义见 `domain-model.md` 与 `protocol-definition.md`。

---

## 7. 项目不是什么

为了避免后续文档继续发散，明确以下边界：

- 本项目**不是**泛行业、硬件无关的抽象资产平台
- 本项目**不是**单纯的品牌数字营销系统
- 本项目**不是**普通二维码 / 静态 NFC 防伪系统
- 本项目**不是**只要平台数据库有一条记录就可成立真品的中心化后台
- 本项目**不是**当前已经完成共同信任根的终局协议

它是一个有明确协议约束、状态机约束、权限约束、安全约束和工程边界的实物资产协议型系统。

---

## 8. 当前定位与终局定位

### 8.1 当前定位

当前 RCProtocol 的准确定位是：

> 平台承载的协议内核 MVP。

### 8.2 终局定位

终局 RCProtocol 的目标定位是：

> 以物理标签为锚点、以平台与品牌方共同信任根为基础、以协议承诺为真相核心、以数据库为索引与审计设施的实物资产协议。

### 8.3 两者关系

- 当前定位是可交付、可验证、可演示的工程承载形态
- 终局定位是后续必须演进到的协议真相形态
- 两者不能混写，更不能偷换
