# 项目总览

> 文档类型：Foundation  
> 状态：Active  
> 权威级别：Authoritative

---

## 1. 项目定义

RCProtocol 是一套面向高价值实物资产的数字主权协议与平台。

它以具备安全认证能力的物理载体为锚点，当前以 `NTAG 424 DNA` 为核心实现对象，通过统一的身份、状态、安全、治理与流转规则，使实物资产获得可验证、可流转、可审计的数字主权。

项目不是单点防伪工具，而是一套围绕“资产存在、权属变化、风险控制、商业清算”建立的协议系统。

---

## 2. 项目想解决的问题

当前高价值实物资产存在四类核心问题：

1. **真伪验证成本高**：传统鉴定依赖人工经验，难以规模化、标准化。
2. **权属链路断裂**：商品售出后，品牌、平台与持有者之间缺乏可信的数字关系。
3. **二级流转缺少公证层**：交易只能证明“像是真的”，难以证明“它是谁的”。
4. **品牌与平台缺乏统一治理能力**：工厂、品牌、门店、消费者、审核员之间缺少明确的权限与状态边界。

RCProtocol 的目标是把实物资产管理从“内容展示”升级为“主权治理”。

---

## 3. 核心方法

RCProtocol 通过以下方法建立完整闭环：

- **资产唯一身份**：每个物理标签对应唯一 UID 与协议内身份记录。
- **授权绑定 / 母子纠缠**：通过授权设备与资产标签的绑定，建立资产激活与处置前提。
- **KDF 密钥派生**：通过 Root Key → Brand Key → Chip Key 的派生链管理认证密钥。
- **动态认证消息**：通过 SUN / CMAC / HMAC 等机制验证物理标签真实性。
- **生命周期状态机**：通过 14 态模型描述资产从预铸造到终态的全部演进。
- **角色与权限矩阵**：通过平台、品牌、工厂、消费者、审核员等角色约束操作边界。
- **治理与审计**：所有关键操作可追踪、可幂等、可审计、可冻结、可恢复。

## 4. 当前系统边界

### 4.1 协议层

协议层定义：

- 资产身份
- 状态机
- 权限模型
- 密钥体系
- 动态认证规则
- 风险与冻结规则

### 4.2 产品层

产品层包含两个主要方向：

- **B 端治理系统**：品牌、工厂、后台运营、审核、发布、委托、风控
- **C 端资产系统**：验证、藏馆、预览、过户、展示、荣誉态

### 4.3 工程层

工程层以 Rust 核心协议实现为主，Go 承担治理编排，运维层提供缓存、数据库、审计与恢复能力。

---

## 5. 当前确定的实现基线

当前文档体系确认以下实现基线：

- **物理载体基线**：`NTAG 424 DNA`
- **核心密码学基线**：`HMAC-SHA256`、`AES-128 CMAC`
- **状态机基线**：14 态模型
- **角色基线**：Platform / Factory / Brand / Consumer / Moderator
- **核心工程方向**：Rust 核心 + Go 治理编排 + 运维支撑组件

历史上存在的 Python MVP 路线、通用资产平台抽象、概念性白皮书等内容，均不再作为当前项目基线，只保留为归档背景材料。

---

## 6. 项目核心对象

本项目的核心对象包括：

- **Asset**：被协议管理的实物资产
- **Tag**：附着于资产上的物理识别与认证载体
- **Authority Device / Mother Tag**：发起激活或处置授权的设备或载体
- **Brand**：资产定义方与激活方
- **Platform**：仲裁、治理、密钥托管与审计平台
- **Factory**：负责盲扫登记与入库流程
- **Consumer**：资产合法持有者与流转参与者
- **Moderator**：负责冻结、恢复与安全审计的审核角色

这些对象的正式定义见 `domain-model.md`。

---

## 7. 项目不是什么

为了避免后续文档继续发散，明确以下边界：

- 本项目**不是**泛行业、硬件无关的抽象资产平台
- 本项目**不是**单纯的品牌数字营销系统
- 本项目**不是**普通二维码 / 静态 NFC 防伪系统
- 本项目**不是**仅面向融资展示的概念白皮书集合

它是一个有明确协议约束、状态机约束、权限约束、安全约束和工程实现边界的实物资产主权系统。
