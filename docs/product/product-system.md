# 产品体系

> 文档类型：Product  
> 状态：Active  
> 权威级别：Authoritative

---

## 1. 产品目标

RCProtocol 的产品体系服务于两个方向：

1. **B 端治理与运营**：让平台、品牌、工厂、审核员能够管理资产生成、激活、发布、审批、风控与审计。
2. **C 端资产验证与持有体验**：让持有者与交易参与者能够验证资产、查看档案、分享预览、发起流转与查看荣誉态。

产品层不定义协议真相，而是把 Foundation 层定义的状态、权限和安全能力转化为可执行的业务流程与页面能力。

---

## 2. 产品组成

### 2.1 B 端：Sovereign Console / BConsole

面向：

- 平台运营
- 品牌总部
- 工厂 / 仓储
- 审核员

核心能力：

- 品牌与产品管理
- 盲扫批次与生产任务管理
- 激活与发布流程管理
- 委托授权与策略应用
- 风控、争议与恢复工单
- 审计与报表

### 2.2 C 端：Sovereign Vault

面向：

- 消费者 / 合法持有者
- 验真用户
- 交易意向方

核心能力：

- 资产验证
- 资产档案查看
- 资产列表与价值展示
- 主权预览分享
- 过户参与
- 荣誉态资产展示

## 3. 关键产品流程

### 3.1 验真流程

1. 用户感应或扫描资产标签
2. 系统校验动态消息与当前状态
3. 返回验证结果与资产基础信息
4. 根据状态显示正常、受限或风险提示

### 3.2 盲扫与激活流程

1. 工厂侧完成 UID 盲扫登记
2. 品牌侧认领待分配资产
3. 系统执行密钥轮换与绑定确认
4. 资产进入 `Activated`

### 3.3 销售与入馆流程

1. 资产从 `Activated` 推进到 `LegallySold`
2. 合法持有关系成立
3. 资产进入持有者的 Vault 视图

### 3.4 过户流程

1. 持有者发起转移意图
2. 目标接收方参与确认
3. 系统校验权限、状态与必要安全前提
4. 权属更新为 `Transferred`

### 3.5 冻结与恢复流程

1. 风险检测或争议触发冻结
2. 审核员进入工单处理
3. 资产保持 `Disputed`
4. 根据审计结论恢复或推进为安全终态

## 4. 页面与能力映射

### 4.1 验证页

用于展示：

- 真伪验证结果
- 当前状态摘要
- 必要的风险提示
- 基础资产信息

### 4.2 Vault 资产页

用于展示：

- 当前持有资产
- 状态
- 基础价值信息
- 流转记录摘要
- 荣誉态呈现

### 4.3 预览页

用于在受控分享场景下展示资产概况，不应越权暴露完整治理信息。

### 4.4 BConsole 治理页

用于执行：

- 品牌 / 产品发布
- 盲扫任务管理
- 激活与授权流程
- 风险工单和恢复操作
- 审计检索

---

## 5. 产品与协议映射原则

### 5.1 页面不创造状态

页面只能读取或触发 Foundation 定义的状态变化，不能自己发明新状态。

### 5.2 文案不替代规则

产品文案可以使用：

- 主权锁定
- 荣誉态
- 赋灵
- 已认证

但这些文案必须映射到正式状态或正式安全规则。

### 5.3 业务旅程服从权限边界

无论产品体验如何设计，都不能绕开角色与权限模型。

## 6. 产品流程与状态映射

本节用于把页面动作、业务动作与正式状态机对齐，避免产品流程脱离协议实现。

### 6.1 工厂盲扫流程

| 步骤 | 页面 / 操作方 | 期望动作 | 状态变化 |
|------|---------------|----------|----------|
| 1 | 工厂任务页 | 创建批次 / 会话 | 无 |
| 2 | 扫描设备 / 工厂页 | 读取 UID 并登记 | `PreMinted -> FactoryLogged` |
| 3 | 工厂确认页 | 完成入库确认 | `FactoryLogged -> Unassigned` |

### 6.2 品牌激活流程

| 步骤 | 页面 / 操作方 | 期望动作 | 状态变化 |
|------|---------------|----------|----------|
| 1 | 品牌待激活列表 | 认领待分配资产 | 无 |
| 2 | 激活操作页 | 触发密钥轮换 | `Unassigned -> RotatingKeys` |
| 3 | 激活确认页 | 建立绑定待确认 | `RotatingKeys -> EntangledPending` |
| 4 | 激活完成页 | 激活完成 | `EntangledPending -> Activated` |

### 6.3 销售合法化流程

| 步骤 | 页面 / 操作方 | 期望动作 | 状态变化 |
|------|---------------|----------|----------|
| 1 | 品牌 / 门店销售页 | 发起售出确认 | 无 |
| 2 | 协议校验 | 校验可售出 | 无 |
| 3 | 销售完成页 | 写入售出结果 | `Activated -> LegallySold` |

### 6.4 用户入馆与持有展示

| 条件 | 页面表现 | 对应状态 |
|------|----------|----------|
| 资产已合法售出 | 出现在持有者 Vault | `LegallySold` |
| 已发生一次或多次过户 | 持有者继续可见 | `Transferred` |
| 已消耗 | 移入荣誉态展示 | `Consumed` |
| 遗珍 | 移入荣誉态展示 | `Legacy` |

### 6.5 过户流程

| 步骤 | 页面 / 操作方 | 期望动作 | 状态变化 |
|------|---------------|----------|----------|
| 1 | 持有者资产页 | 发起转移 | 无 |
| 2 | 接收方确认页 | 确认接收 | 无 |
| 3 | 协议校验 | 校验角色 / 状态 / 安全前提 | 无 |
| 4 | 过户完成页 | 写入结果 | `LegallySold -> Transferred` 或 `Transferred -> Transferred` |

### 6.6 冻结与恢复流程

| 步骤 | 页面 / 操作方 | 期望动作 | 状态变化 |
|------|---------------|----------|----------|
| 1 | 风控系统 / 审核员工单 | 发起冻结 | `any -> Disputed` |
| 2 | 审核处理页 | 审核事实 | 无 |
| 3 | 恢复页 | 恢复到前状态 | `Disputed -> previous_state` |
| 4 | 安全结论页 | 判定失陷 / 篡改 | `Disputed -> Compromised` 或相关安全终态 |

## 8. 开发落地要求

### 8.1 页面动作必须可追溯到接口动作

每个关键页面按钮最终都应映射到明确的接口动作，例如：

- 激活按钮 -> activate API
- 售出按钮 -> legal-sell API
- 过户确认按钮 -> transfer API
- 冻结按钮 -> freeze API

### 8.2 页面状态必须来源于正式状态

前端展示状态不得自行定义平行枚举。展示文案必须由正式状态映射而来。

### 8.3 异常展示必须区分两类

- **协议拒绝**：权限 / 状态 / 安全前提不满足
- **系统失败**：超时、上游失败、缓存不一致、网络问题
