> 文档类型：Product

> 最后更新：2026-04-07（合并 B 端极简化、C 端愿景升级、虚拟母卡过户等变更）

---

## 1. 产品目标

RCProtocol 的产品体系服务于两个方向：

1. **B 端治理与运营**：作为品牌 ERP 的"防伪插件"，让品牌方以最低成本接入防伪、验真和过户追踪能力。支持 API 对接和 Web 后台两种模式。
2. **C 端资产验证与持有体验**：让持有者与交易参与者能够验证资产、查看档案、发起流转。长期愿景定位为"高净值人群的数字身份证"。

产品层不定义协议真相，而是把 Foundation 层定义的状态、权限和安全能力转化为可执行的业务流程与页面能力。

## 2. 产品组成

### 2.1 B 端：Sovereign Console / BConsole

定位：**品牌 ERP 的"防伪插件"，不是品牌的 ERP 系统。**

面向：

- 平台运营
- 品牌方（有/无技术团队均支持）
- 工厂 / 仓储
- 审核员

核心能力（MVP）：

- 极简品牌注册（3 字段）
- API Key 管理
- 资产列表与外部 SKU 映射
- 盲扫批次与生产任务管理
- 激活操作（含外部 SKU 映射绑定、虚拟母卡生成）
- 售出确认
- 基础审计查询

不承担（MVP 阶段）：

- SKU 详情管理（由品牌方自己的系统管理）
- 审批流程（品牌方自行审批）
- 复杂组织治理
- 委托授权与策略应用
- 高级报表

### 2.2 C 端：Sovereign Vault

定位：MVP 阶段为基础验真与资产馆；长期愿景为"高净值人群的数字身份证"。

面向：

- 消费者 / 合法持有者
- 验真用户
- 交易意向方

核心能力（MVP）：

- 资产验证（扫码/NFC 验真）
- 基础资产详情查看
- 基础 Vault 持有列表
- 过户发起/接收（虚拟母卡自动调用，2 步完成）

长期愿景能力（Backlog，不进入 MVP）：

- 主权分数（Sovereignty Score）与等级体系
- 身份认证与第三方验证
- 权益解锁与会员体系
- 社交层（动态、圈子、关注）
- 资产估值与增值展示
- 高级预览分享

### 2.3 品牌接入模式

| 模式 | 适用场景 | 接入方式 | 品牌方工作量 |
|------|---------|---------|-------------|
| API 对接（推荐） | 有 ERP/PLM 系统和技术团队 | 集成 3 个核心 API + 配置 Webhook | 1-2 天 |
| Web 后台（备选） | 无技术团队或 ERP | 登录 Web 后台手动操作 | 20 分钟 |

## 3. 关键产品流程

### 3.1 品牌注册流程

1. 品牌方填写 3 个核心字段（brand_id、brand_name）
2. 系统自动生成 API Key
3. 品牌方可选配置 Webhook 回调地址、Logo、官网
4. 完成注册，可立即开始使用

### 3.2 验真流程

1. 用户感应或扫描资产标签
2. 系统校验动态消息与当前状态
3. 返回验证结果与资产基础信息
4. 显示资产��属者信息
5. 根据状态显示正常、受限或风险提示

### 3.3 盲扫与激活流程

1. 工厂侧完成 UID 盲扫登记（通过 API 或 App）
2. 品牌方在自己的系统中完成审批
3. 品牌方通过 API 或 Web 后台调用激活（绑定外部 SKU 映射）
4. 系统执行密钥轮换、生成虚拟母卡、建立绑定
5. 资产进入 `Activated`

### 3.4 销售与入馆流程

1. 资产从 `Activated` 推进到 `LegallySold`
2. 合法持有关系成立
3. 资产进入持有者的 Vault 视图
4. Webhook 通知品牌方（如已配置）

### 3.5 过户流程

1. 持有者发起转移意图
2. 目标接收方参与确认
3. 系统分层授权校验：
   - 虚拟母卡：生物识别 + 扫子标签（2 步，母卡自动调用）
   - 物理母卡：生物识别 + 扫母卡 + 扫子标签（3 步）
4. 权属更新为 `Transferred`
5. Webhook 通知品牌方（如已配置）

### 3.6 冻结与恢复流程

1. 风险检测或争议触发冻结
2. 审核员进入工单处理
3. 资产保持 `Disputed`
4. 根据审计结论恢复或推进为安全终态

## 4. 页面与能力映射

### 4.1 验证页

用于展示：

- 真伪验证结果
- 当前状态摘要
- 资产归属者信息
- 必要的风险提示

### 4.2 Vault 资产页

用于展示：

- 当前持有资产
- 状态
- 外部 SKU 信息（名称、跳转链接）
- 流转记录摘要

### 4.3 BConsole 治理页

用于执行：

- 极简品牌注册
- API Key 管理
- 盲扫任务管理
- 激活与外部 SKU 映射
- 售出确认
- 审计检索

## 5. 产品与协议映射原则

### 5.1 页面不创造状态

页面只能读取或触发 Foundation 定义的状态变化，不能自己发明新状态。

### 5.2 文案不替代规则

产品文案可以使用：

- 主权锁定
- 荣誉态
- 赋灵
- 已认证

但这些文案必须映射到正式状态或正式安全规则。

### 5.3 业务旅程服从权限边界

无论产品体验如何设计，都不能绕开角色与权限模型。

## 6. 产品流程与状态映射

### 6.1 工厂盲扫流程

| 步骤 | 页面 / 操作方 | 期望动作 | 状态变化 |
|------|---------------|----------|----------|
| 1 | 工厂任务页 / 品牌 API | 创建批次 / 会话 | 无 |
| 2 | 扫描设备 / 品牌 API | 读取 UID 并登记 | `PreMinted -> FactoryLogged` |
| 3 | 工厂确认页 / 品牌 API | 完成入库确认 | `FactoryLogged -> Unassigned` |

### 6.2 品牌激活流程

| 步骤 | 页面 / 操作方 | 期望动作 | 状态变化 |
|------|---------------|----------|----------|
| 1 | 品牌方自己的系统 | 审批激活申请 | 无（不在我们系统内） |
| 2 | 品牌 API / Web 后台 | 调用激活（含外部 SKU 映射） | `Unassigned -> RotatingKeys` |
| 3 | 系统自动 | 密钥轮换 + 虚拟母卡生成 | `RotatingKeys -> EntangledPending` |
| 4 | 系统自动 | 绑定确认 | `EntangledPending -> Activated` |

### 6.3 销售合法化流程

| 步骤 | 页面 / 操作方 | 期望动作 | 状态变化 |
|------|---------------|----------|----------|
| 1 | 品牌 / 门店 | 发起售出确认 | 无 |
| 2 | 协议校验 | 校验可售出 | 无 |
| 3 | 系统 | 写入售出结果 | `Activated -> LegallySold` |

### 6.4 用户入馆与持有展示

| 条件 | 页面表现 | 对应状态 |
|------|----------|----------|
| 资产已合法售出 | 出现在持有者 Vault | `LegallySold` |
| 已发生一次或多次过户 | 持有者继续可见 | `Transferred` |
| 已消耗 | 移入荣誉态展示 | `Consumed` |
| 遗珍 | 移入荣誉态展示 | `Legacy` |

### 6.5 过户流程

| 步骤 | 页面 / 操作方 | 期望动作 | 状态变化 |
|------|---------------|----------|----------|
| 1 | 持有者资产页 | 发起转移 | 无 |
| 2 | 持有者 | 生物识别 + 扫子标签 | 无 |
| 3 | 系统自动 | 调用虚拟母卡校验（或物理母卡扫描） | 无 |
| 4 | 接收方确认页 | 确认接收 | 无 |
| 5 | 协议校验 | 分层授权校验 + 角色/状态检查 | `LegallySold -> Transferred` 或 `Transferred -> Transferred` |

### 6.6 冻结与恢复流程

| 步骤 | 页面 / 操作方 | 期望动作 | 状态变化 |
|------|---------------|----------|----------|
| 1 | 风控系统 / 审核员工单 | 发起冻结 | `any -> Disputed` |
| 2 | 审核处理页 | 审核事实 | 无 |
| 3 | 恢复页 | 恢复到前状态 | `Disputed -> previous_state` |
| 4 | 安全结论页 | 判定失陷 / 篡改 | `Disputed -> Compromised` 或相关安全终态 |

## 8. 开发落地要求

### 8.1 页面动作必须可追溯到接口动作

每个关键页面按钮最终都应映射到明确的接口动作，例如：

- 注册按钮 -> brand registration API
- 激活按钮 -> activate API（含外部 SKU 映射）
- 售出按钮 -> legal-sell API
- 过户确认按钮 -> transfer API（含分层授权校验）
- 冻结按钮 -> freeze API

### 8.2 页面状态必须来源于正式状态

前端展示状态不得自行定义平行枚举。展示文案必须由正式状态映射而来。

### 8.3 异常展示必须区分两类

- **协议拒绝**：权限 / 状态 / 安全前提不满足
- **系统失败**：超时、上游失败、缓存不一致、网络问题

## 9. 本次整理的产品归并结论

本次整理后：

- B 端定位从"品牌 ERP"转为"品牌 ERP 的防伪插件"
- C 端 MVP 仍为基础验真 + 资产馆，长期愿景为"高净值人群数字身份证"
- SKU 管理从独立模块改为外部 SKU 映射
- 审批流从系统内功能改为品牌方自行处理
- 虚拟母卡过户为默认路径（2 步操作），物理母卡为高端可选
- H5 视觉规范、藏馆白皮书、预览码概念稿、Console 方案稿等历史材料退出当前产品真源
