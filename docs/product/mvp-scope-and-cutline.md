# MVP 范围与切线

> 文档类型：Product  
> 状态：Active  
> 权威级别：Execution Baseline

---

## 1. 一句话商业价值

用最小可上线系统，为高价值实物品牌提供“可验证真伪 + 可控激活 + 售后可追踪”的数字主权基础设施，先拿下**品牌接入费 + 激活服务费**两类收入。

---

## 2. 第一付费客户是谁

### 2.1 第一付费客户定义

第一付费客户不是消费者，也不是工厂，而是：

**有高客单价商品、强防伪诉求、愿意为品牌控制力和售后可追踪能力付费的品牌方。**

优先客户画像：

- 单件商品价值高
- 假货和窜货问题明显
- 售后和二级流转存在争议
- 有品牌总部可推动流程变更
- 能接受首批试点导入

### 2.2 为什么不是消费者

原因：

- 消费者不是当前系统的付费决策者
- 消费者价值成立，但变现不直接
- 早期应先证明品牌端 ROI，而不是先做 C 端增长故事

### 2.3 为什么不是工厂

原因：

- 工厂是流程参与者，不是预算主导者
- 工厂需求强，但通常不负责长期 SaaS 采购决策

### 2.4 第一付费动作

第一阶段预期的真实成交方式应是：

1. 品牌按年度支付接入与治理服务费
2. 品牌按激活件数或批次支付执行服务费
3. 平台提供后台、验真页与激活能力

---

## 3. 第一版只做什么

第一版不是完整的资产主权平台，而是为拿下第一批品牌试点而设计的最小成交版本。

### 3.1 MVP 必做范围

第一版只做以下 6 项核心能力：

| 模块 | 必做能力 | 商业理由 |
|---|---|---|
| Brand Console | 品牌 / 产品创建与管理 | 没有这个就无法接入品牌 |
| Factory Flow | 盲扫登记、批次 / 会话管理 | 没有资产入口就无法形成真源 |
| Activation | 资产认领、激活、绑定确认 | 没有激活就没有主权建立 |
| Verification | C 端验真页 | 没有用户可见价值就无法证明系统价值 |
| Legal Sell / Vault | 售出入馆、基础持有视图 | 没有售后关系就无法支撑品牌续费逻辑 |
| Audit | 基础审计查询 | 没有后台可追踪性就无法支撑运营交付 |

### 3.2 MVP 的最小用户路径

品牌侧：

1. 创建品牌与产品
2. 导入 / 关联标签资产
3. 工厂盲扫登记
4. 品牌认领并激活
5. 门店 / 渠道完成售出登记
6. 消费者扫码验真并进入基础 Vault

### 3.3 MVP 的最小页面集

B 端页面只保留：

- 登录页
- 品牌 / 产品管理页
- 批次 / 会话管理页
- 盲扫任务页
- 激活页
- 售出确认页
- 基础审计页

C 端页面只保留：

- 验真页
- 基础资产详情页
- 基础 Vault 页

## 4. 第一版绝对不做什么

以下内容全部明确排除出 MVP，统一进入 Backlog，不进入当前冲刺。

### 4.1 绝对不做的功能

| 功能 | 处理方式 | 原因 |
|---|---|---|
| 完整过户交易系统 | 延后 | 首批品牌接入不依赖二级流转完整闭环 |
| 荣誉态体系 | 延后 | 展示价值大于成交价值 |
| 高级 Preview 分享模型 | 延后 | 可由基础详情页替代 |
| 复杂委托授权体系 | 延后 | 首批客户不需要复杂治理深度 |
| 自动化恢复编排 | 延后 | 早期可人工审核处理 |
| 高级报表中心 | 延后 | 先导出数据即可 |
| 多品牌复杂策略中心 | 延后 | 先用人工配置或后台简单配置 |
| 消费者增长玩法 | 延后 | 当前重点不是拉新，而是成交品牌试点 |

### 4.2 非 MVP 的设计纪律

任何新增需求若不直接提升以下任一目标，默认砍掉：

- 首单成交概率
- 首批试点上线速度
- 激活成功率
- 验真体验闭环
- 品牌续费理由

---

## 5. 上线成功指标是什么

禁止使用“体验更好”“系统更完整”这类模糊目标。第一版上线必须绑定量化指标。

### 5.1 商业指标

| 指标 | 目标 |
|---|---:|
| 首批签约品牌数 | >= 1 |
| 首批试点 SKU 数 | >= 1 |
| 首批接入标签量 | >= 500 |
| 品牌接入收入 | 达成首单 |
| 激活计费链路 | 能按批次或件数结算 |

### 5.2 产品指标

| 指标 | 目标 |
|---|---:|
| 激活成功率 | >= 95% |
| 验真成功响应率 | >= 99% |
| 消费者首次验真完成率 | >= 85% |
| 合法售出后入馆率 | >= 60% |
| 后台关键页面任务完成率 | >= 90% |

### 5.3 运行指标

| 指标 | 目标 |
|---|---:|
| 验真接口 P95 响应时间 | <= 800ms |
| 激活操作成功写入率 | >= 99% |
| 关键写操作审计覆盖率 | 100% |
| Redis 丢失后快照可恢复 | 必须满足 |

### 5.4 失败判定标准

以下任一情况出现，视为 MVP 未达标：

- 没有签下首个品牌试点
- 激活链路频繁失败，无法稳定批量导入
- 验真页虽然能打开，但无法支撑真实门店 / 消费者使用
- 品牌后台无法完成最小运营闭环

## 6. 哪些流程先人工兜底

一人公司阶段，必须把系统能力与人工运营分开。能人工兜底的不要过早系统化。

### 6.1 可先人工兜底的流程

| 流程 | 当前策略 | 原因 |
|---|---|---|
| 品牌接入审核 | 人工审核 | 频次低，不值得先做复杂审批流 |
| 产品发布审批 | 人工确认 + 后台录入 | 首批客户量小 |
| 异常争议处理 | 人工工单 | 早期更需要判断，不需要自动化 |
| 恢复审批 | 人工审核后执行 | 高风险但低频 |
| 报表分析 | 导出 CSV / 手工汇总 | 先验证需求再做报表系统 |
| 策略配置 | 后台表单或配置文件 | 避免过早建设策略中心 |
| 客户培训与上线支持 | 人工服务 | 首批试点必须重服务 |

### 6.2 绝不能人工替代的流程

| 流程 | 原因 |
|---|---|
| 状态机推进 | 协议真相，不能靠人工维护 |
| 动态认证校验 | 防伪核心能力，不能人工判断 |
| 密钥派生 | 安全边界，不能人工操作 |
| 激活写入 | 必须有正式审计事实 |
| 售出登记真源 | 必须形成正式状态变化 |
| 审计记录 | 后续争议处理必须依赖 |

## 7. 验收标准（DoD）

### 7.1 商业验收

- 能向至少 1 个品牌完整演示从盲扫到验真的闭环
- 合同或试点协议中能清晰绑定“接入费 + 激活服务费”

### 7.2 产品验收

- 品牌管理员可独立完成品牌 / 产品录入
- 工厂操作员可完成批次与盲扫流程
- 品牌方可完成激活
- 消费者可扫码验真并进入基础资产页
- 售出后用户可在 Vault 看见资产

### 7.3 技术验收

- 核心写操作全部走 Rust 真源
- Go 层未绕开协议状态写入
- 前端未定义平行状态机
- 关键动作具备幂等与审计能力

---

## 8. 边缘场景（至少 3 个异常流）

### 8.1 激活异常流

1. 标签重复激活：必须拒绝并返回明确错误
2. 激活中途网络中断：必须支持重试或幂等返回
3. 密钥派生失败：必须中止，不允许半成功状态

### 8.2 验真异常流

1. 扫码时网络不可用：前端给出明确失败提示
2. 动态认证参数非法：返回无效认证，不进入正常页
3. 资产处于 `Disputed`：必须展示受限而不是正常结果

### 8.3 售出与入馆异常流

1. 售出记录成功但入馆快照延迟：允许短暂延迟，但必须最终可见
2. 同一资产重复售出提交：必须幂等或拒绝冲突
3. 用户无绑定身份：允许先验真，再引导后续入馆

---

## 9. 当前结论

MVP 的目标不是证明 RCProtocol 的全部终局能力，而是以最低研发成本证明三件事：

1. 品牌愿意为这套系统付费
2. 资产从工厂到激活再到验真可以稳定跑通
3. 售后关系能够形成品牌可持续运营价值

只要这三件事没被验证，其他高级能力一律不进入当前冲刺。
