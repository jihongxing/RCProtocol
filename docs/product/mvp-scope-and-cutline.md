# MVP 范围与切线

> 文档类型：Product  
> 状态：Active  
> 权威级别：Execution Baseline  
> 最后更新：2026-04-09（对齐协议术语、平台 MVP 与终局协议边界）

---

## 1. 一句话商业价值

用最小可上线系统，为高价值实物品牌提供“可验证真伪 + 可控激活 + 售后可追踪”的平台化交付能力，先拿下**品牌接入费 + 激活服务费**两类收入。

注意：

> 本文件定义的是 **MVP 可成交、可交付、可试点** 的产品边界，不定义终局协议是否已经成立。

---

## 2. 第一付费客户是谁

### 2.1 第一付费客户定义

第一付费客户不是消费者，也不是工厂，而是：

**有高客单价商品、强防伪诉求、愿意为品牌控制力和售后可追踪能力付费的品牌方。**

优先客户画像：

- 单件商品价值高
- 假货和窜货问题明显
- 售后和二级流转存在争议
- 有品牌总部可推动流程变更
- 能接受首批试点导入

### 2.2 为什么不是消费者

原因：

- 消费者不是当前系统的付费决策者
- 消费者价值成立，但变现不直接
- 早期应先证明品牌端 ROI，而不是先做 C 端增长故事

### 2.3 为什么不是工厂

原因：

- 工厂是流程参与者，不是预算主导者
- 工厂需求强，但通常不负责长期 SaaS 采购决策

### 2.4 第一付费动作

第一阶段预期的真实成交方式应是：

1. 品牌按年度支付接入与治理服务费
2. 品牌按激活件数或批次支付执行服务费
3. 平台提供后台、验真页与激活能力

---

## 3. MVP 要解决的真实问题

MVP 要解决的是：

- 品牌能不能接入
- 工厂能不能登记标签
- 品牌能不能激活资产
- 消费者能不能验真
- 用户能不能持有与转移资产
- 平台能不能完成审计与交付

MVP 不要求解决的是：

- 品牌与平台是否已经完成共同信任根落地
- 真品定义是否已经完全摆脱数据库依赖
- 协议终局是否已经成立

换句话说：

> MVP 解决的是平台产品闭环，不是终局协议闭环。

---

## 4. 第一版只做什么

第一版不是完整的终局协议网络，而是为拿下第一批品牌试点而设计的最小成交版本。

核心定位：

> 我们不是品牌的 ERP，我们是品牌 ERP 的防伪与协议插件。

### 4.1 MVP 必做范围

第一版只做以下 7 项核心能力：

| 模块 | 必做能力 | 商业理由 |
|---|---|---|
| Brand Registration | 极简品牌注册 + API Key 管理 | 没有这个就无法接入品牌 |
| Factory Flow | 盲扫登记、批次 / 会话管理 | 没有资产入口就无法形成平台闭环 |
| Activation | 资产激活（含虚拟母卡生成 + 外部 SKU 映射绑定） | 没有激活就没有可交付资产 |
| Verification | C 端验真页 | 没有用户可见价值就无法证明系统价值 |
| Legal Sell / Vault | 售出入馆、基础持有视图 | 没有售后关系就无法支撑品牌续费逻辑 |
| Transfer | 过户（虚拟母卡自动调用，用户 2 步完成） | 资产流转是核心平台能力 |
| Audit | 基础审计查询 | 没有后台可追踪性就无法支撑运营交付 |

### 4.2 当前 MVP 的正确表述

当前 MVP 中：

- 验真是当前可运行验真实现
- 激活是当前可运行激活动作
- 过户是当前可运行授权处置流程
- 平台承担当前主要工程真相承载职责

这些都不能被直接表述为：

- 终局真品定义已经落地
- 平台与品牌已经共同持密
- 数据库已经只剩投影层角色

### 4.3 MVP 的最小用户路径

品牌侧（API 对接模式）：

1. 极简注册，获得 API Key
2. 工厂调用盲扫 API
3. 品牌在自己的系统中审批后，调用激活 API
4. 配置 Webhook 接收事件通知

品牌侧（Web 后台模式）：

1. 极简注册，登录 Web 后台
2. 工厂用 App / 设备扫码
3. 品牌在 Web 后台选中资产并激活
4. 在 Web 后台查看资产状态和审计记录

消费者侧：

1. 扫码验真并进入基础 Vault
2. 查看资产归属和状态
3. 发起 / 接收过户

## 5. 第一版绝对不做什么

以下内容全部明确排除出 MVP，统一进入 Backlog，不进入当前冲刺。

### 5.1 绝对不做的功能

| 功能 | 处理方式 | 原因 |
|---|---|---|
| 完整 SKU / 产品管理系统 | 废弃 | 改为外部 SKU 映射，不维护 SKU 详情 |
| 系统内审批流 | 延后 | 品牌方自行审批，我们只执行 |
| 主权分数（Sovereignty Score） | 延后 | C 端增长玩法，不影响首单成交 |
| 身份认证与权益解锁 | 延后 | 需要用户量基础 |
| 社交层（动态、圈子、关注） | 延后 | 远期能力 |
| 资产估值服务 | 延后 | 需要市场数据支撑 |
| 物理母卡默认化 | 延后 | MVP 先跑通虚拟母卡 |
| 品牌 SDK | 延后 | 下一阶段 |
| Excel 批量导入 | 延后 | 下一阶段 |
| 会员订阅 | 延后 | 远期商业模式 |
| 金融服务佣金 | 延后 | 远期商业模式 |
| 荣誉态体系强化 | 延后 | 展示价值大于成交价值 |
| 高级 Preview 分享模型 | 延后 | 基础详情页可替代 |
| 复杂委托授权体系 | 延后 | 首批客户不需要复杂治理深度 |
| 自动化恢复编排 | 延后 | 早期可人工审核处理 |
| 高级报表中心 | 延后 | 先导出数据即可 |
| 多品牌复杂策略中心 | 延后 | 先用人工配置或后台简单配置 |
| Webhook 高级配置 | 延后 | MVP 只支持基础事件推送 |
| 共同信任根正式落地 | 延后到 Stage 8+ | 不属于 MVP 成交闭环 |
| 双签名真品定义 | 延后到 Stage 9+ | 不属于当前交付切线 |
| 数据库降级为投影层 | 延后到 Stage 10+ | 不属于当前交付切线 |

### 5.2 非 MVP 的设计纪律

任何新增需求若不直接提升以下任一目标，默认砍掉：

- 首单成交概率
- 首批试点上线速度
- 激活成功率
- 验真体验闭环
- 品牌续费理由

## 6. 上线成功指标是什么

禁止使用“体验更好”“系统更完整”这类模糊目标。第一版上线必须绑定量化指标。

### 6.1 商业指标

| 指标 | 目标 |
|---|---:|
| 首批签约品牌数 | >= 1 |
| 首批试点 SKU 数 | >= 1 |
| 首批接入标签量 | >= 500 |
| 品牌接入收入 | 达成首单 |
| 激活计费链路 | 能按批次或件数结算 |

### 6.2 产品指标

| 指标 | 目标 |
|---|---:|
| 激活成功率 | >= 95% |
| 验真成功响应率 | >= 99% |
| 消费者首次验真完成率 | >= 85% |
| 合法售出后入馆率 | >= 60% |
| 后台关键页面任务完成率 | >= 90% |
| 品牌 API 对接完成时间 | <= 2 天 |

### 6.3 运行指标

| 指标 | 目标 |
|---|---:|
| 验真接口 P95 响应时间 | <= 800ms |
| 激活操作成功写入率 | >= 99% |
| 关键写操作审计覆盖率 | 100% |
| Redis 丢失后快照可恢复 | 必须满足 |

### 6.4 失败判定标准

以下任一情况出现，视为 MVP 未达标：

- 没有签下首个品牌试点
- 激活链路频繁失败，无法稳定批量导入
- 验真页无法支撑真实门店 / 消费者使用
- 品牌后台无法完成最小运营闭环
- 品牌 API 对接无法跑通核心接口

注意：

- 这些指标达成，说明 **MVP 成功**
- 这些指标达成，不说明 **终局协议成功**

## 7. 哪些流程先人工兜底

一人公司阶段，必须把系统能力与人工运营分开。能人工兜底的不要过早系统化。

### 7.1 可先人工兜底的流程

| 流程 | 当前策略 | 原因 |
|---|---|---|
| 品牌接入审核 | 人工审核 | 频次低 |
| 品牌激活审批 | 品牌方自行审批 | 品牌已有自己的审批系统 |
| 异常争议处理 | 人工工单 | 早期更需要判断 |
| 恢复审批 | 人工审核后执行 | 高风险但低频 |
| 母卡遗失恢复 | 人工审核 | 低频 |
| 报表分析 | 导出 CSV / 手工汇总 | 先验证需求 |
| 策略配置 | 后台表单或配置文件 | 避免过早建设策略中心 |
| 客户培训与上线支持 | 人工服务 | 首批试点必须重服务 |
| 高价值商品物理母卡配发 | 人工兜底 | 低频 |

### 7.2 绝不能人工替代的流程

| 流程 | 原因 |
|---|---|
| 状态机推进 | 当前执行真源，不能靠人工维护 |
| 动态认证校验 | 当前防伪内核 |
| 密钥派生 | 当前安全边界 |
| 激活写入 | 必须有正式审计事实 |
| 售出登记真源 | 必须形成正式状态变化 |
| 审计记录 | 后续争议处理必须依赖 |
| 分层授权校验 | 过户安全不可跳过 |

## 8. MVP 验收标准（DoD）

### 8.1 商业验收

- 能向至少 1 个品牌完整演示从盲扫到验真的闭环
- 合同或试点协议中能清晰绑定“接入费 + 激活服务费”
- 品牌可通过 API 或 Web 后台两种模式完成接入

### 8.2 产品验收

- 品牌方可在 30 秒内完成极简注册
- 品牌方可通过 API 或 Web 后台完成批量激活
- 工厂操作员可完成批次与盲扫流程
- 消费者可扫码验真并进入基础资产页
- 消费者可完成虚拟母卡过户
- 售出后用户可在 Vault 看见资产

### 8.3 技术验收

- 核心写操作全部走 Rust Core
- Go 层未绕开协议状态写入
- 前端未定义平行状态机
- 关键动作具备幂等与审计能力
- 虚拟母卡链路可靠
- API Key 认证链路可靠

### 8.4 协议边界提醒

即使上述 DoD 全部通过，也只代表：

- MVP 产品可交付
- 平台闭环可演示
- 品牌可接入

不代表：

- `AssetCommitment` 已落地
- 品牌承诺与平台承诺已落地
- 双边共同信任根已落地
- 数据库已退出终局真相定义中心

---

## 9. Stage 7 之后才进入的协议演进事项

以下事项明确不属于 MVP 范围，而属于后续协议演进阶段：

1. `AssetCommitment` 正式落地
2. Brand Attestation 正式落地
3. Platform Attestation 正式落地
4. Verification V2（芯片证明 + 双承诺）
5. 真品定义从“当前实现”升级为“终局定义”
6. 数据库从真相承载层向投影层降级
