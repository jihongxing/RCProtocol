# RCProtocol 全球高性能与高安全改造实施路线图

> 产出时间：2026-04-07  
> 关联文档：`docs/audit/global-performance-security-architecture-audit-2026-04-07.md`  
> 文档目标：把审计意见转成可执行的技术改造计划，覆盖未来 12 周的架构、性能、安全、运维落地路径。

---

## 1. 路线图目标

本路线图不讨论“要不要重构”，而是直接定义：

1. **先把系统拉到生产可上线基线**
2. **再把系统升级到全球读优化 + 主区写收敛架构**
3. **最后把安全体系升级到零信任 + KMS + 不可抵赖审计**

### 最终目标状态

12 周后，RCProtocol 应具备以下能力：

- 核心写路径统一收敛到 `Rust API`
- 全球访问采用“**边缘读 + 主区写**”模式
- 所有服务具备**生产部署能力、健康检查、日志、指标、链路追踪**
- 身份体系完成**短期令牌、密钥轮换、最小权限治理**
- 数据层具备**读写分离、异步投影、缓存策略**
- 安全层具备**Secrets/KMS、mTLS/服务身份、审计归档**

---

## 2. 总体实施原则

### 2.1 不推倒重来

保留当前主技术路线：

- `Rust` 继续承担协议真源与核心安全逻辑
- `Go` 继续承担治理编排、BFF、IAM、审批、工单
- `PostgreSQL` 继续作为权威真源
- `Redis` 继续作为可重建加速层

### 2.2 先修基础设施，再做大规模功能扩展

顺序必须是：

1. 先修部署、观测、安全、数据边界
2. 再优化读写分层与全球接入
3. 最后再扩展高阶能力

### 2.3 所有改造以“验收标准”驱动

每一阶段必须输出：

- 可交付清单
- 可验证结果
- 监控指标
- 回滚方案

## 3. 组织分工建议

建议最少按 5 个责任域推进：

### A. 协议核心组（Rust）

负责：

- `rc-api`
- `rc-core`
- `rc-kms`
- 状态机、验真、幂等、审计事实

### B. 治理服务组（Go）

负责：

- `go-gateway`
- `go-iam`
- `go-bff`
- `go-approval`
- `go-workorder`

### C. 前端与接入组

负责：

- `b-console`
- `c-app`
- API SDK
- 前端安全接入与发布

### D. 平台基础设施组

负责：

- K8s / 部署编排

- 全球接入 / CDN / WAF

### E. 安全与合规组

负责：

## 4. 12 周分阶段实施图

---

## Phase 0：立即止血（第 1 周）

### 目标

把当前项目从“存在明显生产风险”拉到“可以继续安全迭代”的状态。

### 本阶段必须完成的事项

#### 4.0.1 关闭高风险默认配置

- 移除或禁止生产使用 `RC_AUTH_DISABLED=true`
- `.env.example` 中所有 Root Key / JWT Secret 改为占位符
- 启动时强制校验弱密钥、弱 secret、危险默认值

#### 4.0.2 修复已知关键鉴权链路问题

- Gateway 统一注入可信 `X-Claims-*`
- Gateway 剥离客户端伪造 `X-Claims-*`
- IAM 所有管理接口必须加鉴权与角色限制
- BFF 不允许只解 JWT payload，必须完整验签或强制只接受 Gateway 内部流量
- Approval / Workorder / BFF / IAM 统一拒绝绕过网关的伪造身份调用

#### 4.0.3 修复协议核心高风险安全缺陷

- `Recover` 必须只允许恢复到系统记录的冻结前状态
- 补齐 `Recover` 状态机注册
- 补齐 `Destructed` 合法动作入口
- 修复 `verify` 降级暴露敏感资产元数据问题
- 修复 CTR 持久化真源顺序，先 PG 后 Redis/内存

#### 4.0.4 前端止血

- C 端隐藏 UID-only 公共验真入口
- C 端写操作统一收敛到 BFF 语义接口，不允许直接打底层资产写接口
- B 端 / C 端减少长期持久化 token 风险，至少先缩短有效期并区分高权限端

### 交付物

- `critical-fixes` 代码分支合并
- 高风险项关闭清单
- 新的启动安全检查说明
- 风险回归测试报告

### 验收标准

- 所有已知 Critical 审计项关闭
- 没有任何生产默认弱密钥
- 关键写接口必须经过统一身份链路
- `verify` 在认证失败时不再泄露业务元数据

---

## Phase 1：生产化基线（第 2~3 周）

### 目标

把当前单机/compose 思维升级为最小生产基线。

### 重点事项

#### 4.1.1 建立生产部署骨架

- 从 `docker-compose` 升级到生产编排方案
- 建议优先：`Kubernetes`
- 为每个服务提供：
  - Deployment
  - Service
  - Config / Secret 注入
  - readiness/liveness 探针
  - resource request / limit

#### 4.1.2 服务无状态化确认

- 所有服务禁止依赖本地文件作为关键状态
- 本地缓存只能是临时可丢失层
- 重启后状态必须可恢复

#### 4.1.3 建立统一观测基线

- 全服务结构化日志
- 统一 `trace_id`
- 接入 Prometheus 指标
- 接入 OpenTelemetry Trace
- 打通网关 → Go 服务 → Rust API → DB 的链路追踪

#### 4.1.4 建立数据库与 Redis 运维基线

- PostgreSQL 自动备份
- Redis 持久化和故障恢复文档
- 数据库连接池统一配置
- 为核心 SQL 建立慢查询审查机制

### 交付物

- 生产部署目录结构
- 基础 Helm / K8s manifests
- 可观测性面板初版
- Postgres/Redis 备份恢复 SOP

### 验收标准

- 所有服务具备健康检查
- 所有服务启动配置不依赖人工临时环境
- 能看到每个接口的 QPS / 错误率 / P95
- 故障重启后服务可自动恢复

---

## Phase 2：全球入口与边缘接入（第 4~5 周）

### 目标

建立“全球访问”的真实入口层，而不是单区暴露。

### 重点事项

#### 4.2.1 接入全球流量入口

- 增加 `CDN + WAF + Global Load Balancer`
- 设计域名分层：
  - `verify` 公共流量
  - `b-console` 管理流量
  - `api` 系统流量

#### 4.2.2 区分公开流量与治理流量

- `verify` 走高缓存、高频防护策略
- 后台接口走更严格 WAF / MFA / 限流 / 地域策略
- 管理接口增加来源限制

#### 4.2.3 设计区域路由策略

建议采用：

- **主写区**：协议写入、审批、工单、身份核心
- **读区**：公共验证、只读查询、静态前端

#### 4.2.4 边缘缓存规划

- `verify` 公共可缓存响应拆分为弱敏感公开字段
- 静态资源全部上 CDN
- 前端资源版本化发布

### 交付物

- 全球入口拓扑图
- 域名与路由表
- CDN/WAF 策略初版
- 边缘缓存规则清单

### 验收标准

- 用户访问不再直打单区源站
- 前端静态资源走 CDN
- `verify` 流量可独立控流
- 管理接口与公开接口策略分离

---

## Phase 3：数据层升级与读写分层（第 6~7 周）

### 目标

让系统具备高并发访问能力，避免核心真表被列表/查询拖垮。

### 重点事项

#### 4.3.1 建立读写分层模型

- 核心写表只服务协议真相
- 列表 / 看板 / 查询改用 read model
- 建立只读副本策略
- 按接口区分回主库与回只读副本

#### 4.3.2 定义缓存策略

- 资产详情缓存
- 品牌 / 产品元数据缓存
- 热点 UID / 热点资产保护
- 缓存穿透 / 击穿 / 雪崩防护

#### 4.3.3 优化 `verify` 路径

- 拆分公开响应与深度回源响应
- 对无风险请求尽量命中缓存/投影
- 高风险请求回主链路进行完整校验

#### 4.3.4 数据库专项治理

- 为高频查询补索引
- 为慢 SQL 建优化清单
- 限制大表扫描
- 为热点维度（brand_id、asset_id、uid）做专项压测

### 交付物

- 读写分层设计文档
- cache key 规范
- SQL / 索引优化清单
- `verify` 两层响应方案

### 验收标准

- 后台列表页不直接压核心真表
- Redis 命中率达到可追踪标准
- 高热点接口有明确缓存策略
- 主库压力可控，读请求明显分流

---

## Phase 4：事件驱动与异步投影（第 8~9 周）

### 目标

把系统从同步串行聚合，升级为“写事实、读投影”的可扩展架构。

### 重点事项

#### 4.4.1 引入消息总线

建议优先候选：

- `Kafka`
- `Redpanda`
- `NATS JetStream`

根据团队复杂度承受能力选择其一。

#### 4.4.2 建立事件模型

至少定义以下事件：

- `asset_state_changed`
- `asset_verified`
- `approval_created`
- `approval_executed`
- `workorder_created`
- `risk_detected`
- `key_rotated`

#### 4.4.3 引入 Outbox Pattern

- 核心写事务先写业务事实 + outbox
- 异步投递到消息总线
- 下游消费者负责更新投影、缓存、搜索、报表

#### 4.4.4 改造 BFF 聚合逻辑

- 频繁查询页面优先读投影
- 减少串行调用 IAM / rc-api / approval / workorder
- 将 BFF 从“重编排”降为“轻聚合”

### 交付物

- 事件模型文档
- Outbox 实现
- 首批投影消费者
- BFF 改造说明

### 验收标准

- 关键页面读路径不依赖多服务串行 fan-out
- 写后异步投影链路可观测
- 缓存、投影、报表均由事件驱动更新

---

## Phase 5：安全体系升级（第 10~11 周）

### 目标

把当前“基础安全”升级为“生产级高安全”。

### 重点事项

#### 4.5.1 Secret 管理升级

- 从环境变量散落管理迁移到统一 Secret 平台
- 所有敏感配置最小权限读取
- Secret 访问纳入审计

#### 4.5.2 KMS 分层落地

- 接入云 KMS / HSM
- Root Key 不再以明文形式存在于普通业务配置
- 密钥支持版本号、轮换、吊销、再包裹

#### 4.5.3 服务间零信任化

- 服务到服务增加 `mTLS`
- 采用 workload identity / service identity
- 禁止仅凭“内网可达”信任调用

#### 4.5.4 管理后台高安全化

- B 端后台强制 MFA / WebAuthn
- 高风险操作加入二次验证
- 平台级和品牌级权限进一步细分

#### 4.5.5 安全审计升级

- 登录失败、权限失败、冻结、恢复、密钥轮换都进入独立安全审计流
- 审计写入独立归档存储
- 平台管理员无权直接篡改审计记录

### 交付物

- KMS 接入设计与实现
- Secret 管理平台接入结果
- mTLS / 内部身份方案
- 安全审计事件字典

### 验收标准

- Root Key 不再裸暴露在普通部署配置中
- 所有服务间调用具备身份可验证性
- 高风险后台操作必须二次确认
- 安全审计能按 trace_id 回放事件链路

## 7. 推荐上线门槛指标

### 接口性能

- `verify`：P95 < 150ms（缓存命中场景）
- `verify`：P95 < 400ms（回源场景）
- 管理后台列表：P95 < 300ms
- 核心写接口：P95 < 500ms

### 可用性

- 核心 API 成功率 ≥ 99.9%
- 验真接口成功率 ≥ 99.95%
- 单区服务异常不导致全局完全不可用

### 安全

- 所有高权限后台操作必须 MFA
- 所有服务间调用可验证身份
- Root Key 不以业务配置明文形式下发
- 审计日志具备独立归档与回放能力

### 运维

- PostgreSQL 恢复演练通过
- Redis 丢失后缓存可重建
- 单服务故障可自动恢复
- 发布失败可快速回滚

## 8. 风险提示

以下事项最容易导致路线图失败：

### 8.1 同时做太多功能开发

如果在基础设施还没稳住前继续大规模开发业务功能，会导致改造长期拖延。

### 8.2 把“全球化”误解成“全球多主写”

当前项目最适合的是“全球读优化 + 主区写收敛”，不要过早上全球多主写。

### 8.3 安全只停留在文档

必须把：

- 弱配置禁止
- KMS 接入
- MFA
- mTLS
- 审计归档

真正做成工程能力，而不是规范口号。

### 8.4 BFF 持续膨胀

如果 BFF 继续承担过重编排，将成为长期性能瓶颈。必须及早转向投影读模型。

## 9. 最终建议

这份路线图建议你们按“**三线并行、六阶段推进**”执行：

- **线 1：生产与全球接入**
- **线 2：数据与性能架构**
- **线 3：安全与合规升级**

### 最重要的执行顺序

1. **先止血**：修 Critical
2. **再生产化**：部署、观测、备份、限流
3. **再全球化**：CDN、WAF、读写分层、边缘缓存
4. **再高安全化**：KMS、mTLS、MFA、审计归档
5. **最后门槛化**：压测、故障演练、安全演练、上线 checklist
