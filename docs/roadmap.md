# RCProtocol Roadmap（阶段计划）

> 本文档用于描述 RCProtocol 从当前 MVP 工程承载形态，演进到可交付平台与目标协议形态的阶段路线。状态说明：
>
> - ⬜ 待开始
> - 🔧 进行中 / 部分完成
> - ✅ 基本完成 / 已完成
>
> 说明：Roadmap 反映的是“当前仓库真实实现状态 + 阶段准出标准 + 协议终局演进方向”，不是最初理想计划，也不等于终局协议已经落地。

---

## 路线总说明

RCProtocol 当前处于：

> **协议内核已形成、平台化 MVP 已基本成型、终局协议尚未落地**

因此本 Roadmap 同时描述两条并行但不可混淆的路线：

1. **MVP / 平台交付路线**：让品牌可接入、让平台可运营、让前后端可交付
2. **协议终局演进路线**：让系统从“平台承载协议”升级为“协议定义真相、平台只是参与者之一”

禁止将前者误判为后者已经完成。

## Stage 3：Go 边车与双通路服务化改造 🔧 进行中

**目标**：围绕“品牌开放接入 + 平台托管运营”双通路，补齐 Gateway、IAM、BFF、Workorder、Webhook 等 Go 边车能力，使不同信息化成熟度的品牌方都能接入 RCProtocol，同时共享统一的协议核心与业务主状态机。

对应 **Spec-11 衍生 + 服务化拆分**。

### 双通路定义

#### 通路 A：品牌开放接入通路
面向 **信息化能力较强** 的品牌方。

- 品牌方保留自己的 ERP / OMS / 商品系统
- 通过 `API Key` 直接调用 RCProtocol 核心接口
- 不强依赖平台建议后台
- 重点满足：盲扫、激活、查询、验真、过户记录等开放接入需求

#### 通路 B：平台托管运营通路
面向 **信息化能力较弱** 的品牌方。

- 平台提供建议后台
- 帮助品牌方管理品牌资料、SKU、人员、API Key、工单与运营动作
- 通过后台页面完成盲扫、激活、售出、审计、冻结恢复等运营闭环
- 重点满足：低门槛上线、可直接使用、可替代自建系统的最小经营能力

### 当前阶段统一原则

- 双通路是**产品层双通路**，不是重复建设两套核心业务
- Go 边车负责接入层、聚合层、运营层、治理层能力，不复制 Rust 核心状态机
- 后台管理 SKU 的定位是**托管商品主数据 / 外部 SKU 托管能力**，不是第二套协议信相源
- 本阶段解决的是 **平台交付能力**，不是 **共同信任根已经落地**

### 当前阶段目标

| 序号 | 目标 | 核心交付 | 状态 |
|------|------|----------|------|
| 3.1 | 统一接入入口 | `go-gateway` 支持 API Key / JWT 双认证入口、统一转发、日志、Trace、限流、错误外壳 | 🔧 部分完成 |
| 3.2 | 托管运营身份中心 | `go-iam` 支持用户 / 组织 / 成员 / 岗位 / 品牌后台管理能力 | 🔧 部分完成 |
| 3.3 | 双端聚合视图层 | `go-bff` 为 B/C 端提供稳定 ViewModel，聚合品牌、SKU、资产、API Key、Dashboard、工厂任务等视图 | 🔧 部分完成 |
| 3.4 | 运营动作编排层 | `go-workorder` 支持冻结 / 恢复 / 风险标记等运营动作编排，并稳定联动 `rc-api` | 🔧 部分完成 |
| 3.5 | 历史审批归档 | `go-approval` 明确归档，不再作为主线推进 | ✅ 已归档 |
| 3.6 | 事件扩展能力 | `go-webhook` 提供数据库队列、delivery worker、配置装配等事件推送能力 | 🔧 部分完成 |

### 当前阶段关键提醒

Stage 3 的完成意味着：

- 平台具备更完整的接入与托管能力
- 品牌方更容易接入
- 前后端联调边界更稳定

但 **不意味着**：

- 品牌已经成为协议共同信任根
- 平台数据库已经降级为投影层
- 真品成立已经脱离平台单点承载

## 阶段判断总规则

自本 Roadmap 生效起，阶段完成必须按以下规则解读：

1. **Stage 1~6 完成**：说明平台 MVP、品牌接入、前后端交付与商业试点能力已成型
2. **Stage 7~10 完成**：说明协议开始真正从平台承载形态走向独立真相形态
3. **Stage 11 完成**：说明高安全版本接近用户要求的终局目标

换言之：

> **MVP 成功不等于协议终局成功。**

## 关键依赖与风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| NTAG 424 DNA 实物标签供应 | 无法真机测试验真链路 | 优先用模拟数据跑通协议链路，硬件测试并行推进 |
| 单人开发瓶颈 | 交付周期拉长 | 严格 MVP 切线，分阶段推进 |
| 密码学实现正确性 | 安全边界失守 | `rc-crypto` 必须有完整测试向量覆盖 |
| 多服务契约漂移 | 前后端和网关/BFF 行为不一致 | 以契约测试和联调脚本持续收口 |
| 概念混用持续存在 | 团队继续按平台思维替代协议思维实现 | 以 `protocol-definition.md`、`protocol-gap-analysis.md`、`target-protocol-architecture.md` 为前置约束 |
| 品牌侧协作成本高 | 共同信任根落地难度高 | 先落地双签名承诺模型，再逐步升级到更强方案 |
