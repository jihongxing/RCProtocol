> 文档类型：Engineering  
> 状态：Active  
> 权威级别：Authoritative  
> 最后更新：2026-04-09（对齐协议差距分析与目标协议终局架构）

---

## 1. 工程定位

RCProtocol 的工程架构遵循以下原则：

- 协议核心收敛
- 治理编排外置
- 运维支撑独立
- 当前工程承载与终局协议目标明确区分

当前工程基线是：

> **以中心化平台承载协议内核的 MVP 系统**。

它不是纯 CRUD 平台；但它也不是已经完成协议终局的系统。

因此本文件必须同时描述：

1. **当前工程如何运行**
2. **当前工程不应被误认为已经实现了什么**
3. **后续如何从工程承载形态演进到目标协议形态**

---

## 2. 当前架构基线

### 2.1 核心方向

- **Rust**：协议核心、状态机、认证、安全、KMS 边界、虚拟母卡分层校验
- **Go**：治理编排、BFF、工单、报表、API Key 管理、Webhook 推送
- **PostgreSQL**：当前权威持久化承载层
- **Redis**：缓存、快照、计数类加速层

### 2.2 当前工程原则

- 协议核心在 Rust，不在治理层
- 查询可聚合，写入必须收敛
- Redis 不是权威源
- 故障后必须可以从 PostgreSQL 恢复缓存与投影
- Go 边车不能复制 Rust 核心状态机

### 2.3 必须明确的当前边界

当前基线中，PostgreSQL 仍然承载大量协议关键事实。

因此当前系统可以表述为：

- **协议内核已经存在**
- **工程承载仍然明显中心化**

但不能表述为：

- 协议已经脱离平台数据库独立成立
- 平台只是索引层而不再是真相承载者
- 平台与品牌方已经实现共同信任根

---

## 3. 当前架构与终局协议的关系

### 3.1 当前工程是什么

当前工程是：

- 一个可开发、可部署、可恢复的协议型 MVP 系统
- 一个以芯片密码学与状态机为内核的平台化实现
- 一个允许品牌开放接入与平台托管运营双通路并存的系统

### 3.2 当前工程不是什么

当前工程不是：

- 已经完成双边共同信任根的协议网络
- 已经去除数据库真相源依赖的终局协议
- 已经把平台降级为纯粹的查询 / 路由 / 审计节点

### 3.3 终局演进方向

终局演进方向应当是：

- 真品成立依赖协议证明对象而不是资产表行记录
- 平台与品牌方共同参与真相建立
- 数据库从“真相承载”逐步降级为“索引 / 投影 / 审计设施”

---

## 4. 分层模型

### 4.1 Foundation / Protocol 层

包含：

- 状态机
- 标签动态认证
- 权限校验
- 风险与冻结判定
- 分层授权校验
- 协议承诺对象（终局需收敛到 `AssetCommitment`）

### 4.2 Service 层

包含：

- 协议 API
- KMS 服务
- Gateway
- IAM
- BFF
- Workorder
- Webhook

### 4.3 Data / Infra 层

包含：

- PostgreSQL
- Redis
- 审计存储
- 监控与告警
- 故障恢复工具

### 4.4 当前与终局的关键差异

当前 Data 层里，PostgreSQL 仍然承载大量关键事实。

终局要求是：

- PostgreSQL 保存承诺缓存、索引、投影、审计与恢复材料
- PostgreSQL 不再成为“真品成立”的最终依据

---

## 5. 当前典型系统拓扑

```text
Client / Device / H5 / BConsole / Brand API
        |
        v
Go Gateway / BFF (API Key 认证 / JWT 认证)
        |
        v
Rust API / Rust Core / Rust KMS
        |
   +----+----+
   |         |
   v         v
PostgreSQL  Redis
```

说明：

- 前端、设备、后台入口和品牌 API 调用先进入接入层或治理层
- 品牌 API Key 认证在 Gateway 完成，映射为内部调用上下文后转发至 Rust Core
- 所有核心写动作最终必须进入 Rust 核心链路
- Redis 只作为加速与快照层
- PostgreSQL 当前仍是主要权威持久化承载层

注意：

> “PostgreSQL 当前是权威持久化承载层”不等于“终局协议中数据库就是最终真相源”。

---

## 6. 模块职责拆解

### 6.1 Rust Core

负责：

- 状态机合法性校验
- 角色权限最终判定
- 资产生命周期推进
- 动态消息认证
- 风险冻结与恢复判断
- 分层授权校验
- 外部 SKU 映射写入
- 协议级审计事实写入

当前边界：

- Rust Core 已经是协议执行核心
- 但当前执行仍依赖平台数据库中的资产上下文

终局要求：

- Rust Core 应逐步围绕 `AssetCommitment` 等协议承诺对象执行
- 不应把资产表行记录本身当成协议真相本体

### 6.2 Rust KMS

负责：

- 当前平台侧密钥派生
- 安全敏感运算
- 认证辅助能力
- 风险相关密钥逻辑

当前边界：

- 当前 KMS 仍然是平台主导派生

终局要求：

- 应升级到品牌与平台共同参与的安全模型
- 双签名承诺优先，联合派生 / MPC / HSM 协作后续演进

### 6.3 Go Gateway / IAM / BFF / Workorder / Webhook

负责：

- API Key / JWT 接入
- 用户与组织治理
- 前端聚合视图
- 风险动作编排
- 事件通知

正确定位：

- 它们是接入层、治理层、运营层、聚合层
- 它们不是协议真相源
- 它们不应复制 Rust Core 状态机

---

## 7. 数据存储分工

### 7.1 PostgreSQL 的当前角色

当前 PostgreSQL 承载：

- 资产主记录
- 品牌记录
- 母卡凭证记录
- 授权绑定关系记录
- 状态变化历史
- 虚拟母卡敏感材料（加密存储）
- 审计事实
- 恢复与冻结事实
- API Key 记录

### 7.2 PostgreSQL 的终局目标角色

终局中 PostgreSQL 应逐步收敛为：

- 承诺索引
- 协议事件存储
- 状态投影
- 查询缓存基础
- 审计与恢复材料

终局中 PostgreSQL 不应承担：

- 真品存在性的最终定义
- 品牌发行真相的唯一依据
- 所有权合法性的唯一依据

### 7.3 Redis

Redis 作为加速层，承载：

- 钱包快照
- 列表投影
- 读缓存
- 某些计数辅助视图

约束：

- Redis 不能单独作为真源
- Redis 丢失后允许从 PostgreSQL 重建
- 快照不一致以更高优先级真相恢复

## 8. 当前关键链路的运行顺序

### 8.1 盲扫登记

1. Factory 发起任务（通过品牌 API 或 Web 后台）
2. Go 治理层创建批次 / 会话
3. Rust Core 执行盲扫登记
4. PostgreSQL 记录资产与状态
5. Go 层汇总结果

### 8.2 激活

1. Brand 发起激活
2. Gateway 验证 API Key / JWT，转发至 Rust Core
3. Rust Core 调用 KMS 执行密钥派生
4. 生成虚拟母卡凭证或关联物理母卡
5. 绑定外部 SKU 映射
6. 推进状态
7. 写入审计
8. Webhook 通知品牌方（如已配置）

### 8.3 验真

1. 读取标签动态消息
2. Rust Core 根据 `uid` 查询当前资产上下文
3. 调用平台侧 KMS 派生芯片相关密钥
4. 验证 `CMAC`
5. 检查 `CTR` 与风险规则
6. 返回验真结果并写入审计

必须明确：

- 这是当前 MVP 的可运行链路
- 不是终局协议的最终验真定义

### 8.4 过户

1. Consumer 发起转移请求
2. Go 层组织交易上下文或确认流
3. Rust Core 进行角色、状态、安全前提检查
4. 分层授权校验
5. 状态推进到目标状态
6. 快照异步更新或延迟重建
7. Webhook 通知品牌方（如已配置）

### 8.5 冻结与恢复

1. 风险系统或人工工单发起冻结
2. Rust Core 推进到 `Disputed`
3. 审核完成后恢复到前状态或进入安全终态
4. 投影与缓存同步更新

---

## 9. 一致性与恢复规则

### 9.1 当前优先级

发生冲突时，当前工程优先级为：

1. Rust Core 产生的协议执行事实
2. PostgreSQL 持久化数据
3. Redis 快照 / 聚合投影
4. 前端缓存

### 9.2 终局目标优先级

终局应逐步过渡为：

1. 协议承诺与协议事件事实
2. 由协议事件重建得到的持久化投影
3. Redis 快照 / 聚合投影
4. 前端缓存

### 9.3 失败恢复

系统必须支持：

- Redis 快照重建
- CTR 校准
- PostgreSQL 故障切换
- 审计完整性校验
- 网络分区后重新校准
- 虚拟母卡凭证重新下发

---

## 10. 开发约束

### 10.1 禁止项

- 禁止治理层绕过 Core 直接写协议状态
- 禁止前端页面状态替代后端状态机
- 禁止把缓存结果当成状态推进依据
- 禁止并行维护两套不同状态定义
- 禁止把 PostgreSQL 当前承载角色误写成终局协议定义
- 禁止把平台接入能力误写成品牌共同信任根已经落地

### 10.2 建议项

- 所有状态推进接口统一输出 pre-state / post-state
- 所有关键写操作携带 trace id 和 idempotency key
- 所有投影读模型都保留回源重建能力
- 所有协议关键能力都逐步引入承诺对象视角，而不是仅围绕表行主键展开

---

## 11. 协议终局演进的工程路线

### 11.1 第一步：引入统一术语与承诺对象

工程上先统一：

- 真品定义
- 承诺对象定义
- 数据库角色定义
- 品牌共同信任根定义

### 11.2 第二步：引入 `AssetCommitment`

将激活、验真、审计、过户等关键链路逐步围绕 `AssetCommitment` 收敛。

### 11.3 第三步：引入品牌承诺证明

品牌方必须开始对 `AssetCommitment` 给出协议级承诺，而不仅是发起 API 调用。

### 11.4 第四步：引入平台共同承诺证明

平台方对同一 `AssetCommitment` 给出共同承诺，使“平台 + 品牌”共同信任根成为事实。

### 11.5 第五步：将 PostgreSQL 从真相承载层逐步降级为索引 / 投影 / 审计层

当承诺对象与协议事件能够独立表达真相后，再逐步弱化资产表行记录的“主定义”地位。

## 12. 运维与实验基线

当前仍继续有效的运维能力包括：

- 故障恢复
- CTR 校准
- 地理围栏配置
- 分片扩容
- 钱包快照重建

当前仍继续有效的硬件与实验价值包括：

- ACR122U
- NTAG 424 DNA
- 单卡稳定测试路径

这些内容属于工程支撑材料，不替代协议规范。

---

## 13. 历史路线清理结论

本次整理后确认：

- 早期历史路线不再作为当前工程基线
- 与当前 Rust + Go 基线冲突的历史技术路线统一失效
- 完整 SKU/产品管理方案继续收敛为“外部 SKU 映射 / 品牌商品主数据托管能力”
- 平台托管后台继续允许存在，但其定位是运营层，不是协议真相源
