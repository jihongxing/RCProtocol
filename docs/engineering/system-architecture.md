> 文档类型：Engineering

---

## 1. 工程定位

RCProtocol 的工程架构遵循“协议核心收敛、治理编排外置、运维支撑独立”的原则。

当前工程基线不是 Python MVP，也不是通用化 ALP 抽象平台，而是围绕 RCProtocol 本身构建的可开发、可部署、可恢复的系统架构。

---

## 2. 当前架构基线

### 2.1 核心方向

- **Rust**：协议核心、状态机、认证、安全、KMS 边界
- **Go**：治理编排、审批流、BFF、报表与工单
- **PostgreSQL**：权威持久化存储
- **Redis**：缓存、快照、计数类加速层

### 2.2 原则

- 主权真相在协议核心，不在治理层
- 关键状态以持久化与协议校验为准，不以缓存为准
- 查询可聚合，写入必须收敛
- 故障后必须可以从权威数据恢复缓存与投影

---

## 3. 分层模型

### 3.1 Foundation / Protocol 层

包含：

- 状态机
- 密钥派生
- 动态认证
- 权限校验
- 风险与冻结判定

### 3.2 Service 层

包含：

- 协议 API
- KMS 服务
- 治理服务
- BFF 聚合接口

### 3.3 Data / Infra 层

包含：

- PostgreSQL
- Redis
- 审计存储
- 监控与告警
- 故障恢复工具

---

## 4. 典型系统拓扑

```text
Client / Device / H5 / BConsole
        |
        v
Go Gateway / BFF / Governance
        |
        v
Rust API / Rust Core / Rust KMS
        |
   +----+----+
   |         |
   v         v
PostgreSQL  Redis
```

说明：

- 前端、设备与后台入口可以先进入治理层或接入层
- 所有协议核心写动作最终必须进入 Rust 核心链路
- Redis 只作为加速与快照层，不是权威源

---

## 5. 模块职责拆解

### 5.1 Rust Core

负责：

- 状态机合法性校验
- 角色权限最终判定
- 资产生命周期推进
- 动态消息认证
- 风险冻结与恢复判断

输入：

- 已认证的调用上下文
- 资产 / 品牌 / 会话等资源标识
- 当前请求动作

输出：

- 成功的状态变化结果
- 协议级错误
- 审计事实

### 5.2 Rust KMS

负责：

- 密钥派生
- 安全敏感运算
- 认证辅助能力
- 风险相关密钥逻辑

输入：

- 品牌标识
- UID / Epoch / Serial 等派生要素

输出：

- 派生结果
- 验证结果
- 安全失败语义

### 5.3 Go Governance

负责：

- 组织 / 岗位 / 策略 / 审批流
- 工单推进
- BFF 聚合
- 报表与运营视图

输入：

- 前端或后台操作请求
- 审批与治理上下文

输出：

- 是否允许进入协议核心执行
- 聚合后的查询与治理反馈

## 6. 开发落地最小模块清单

当前建议按以下最小模块推进落地。

### 6.1 Phase A：协议主链路

必须先完成：

- 资产状态模型
- 角色权限模型
- 认证与校验链路
- 盲扫登记
- 激活
- 合法售出
- 过户
- 冻结 / 恢复

### 6.2 Phase B：治理编排

在主链路可用后补齐：

- 审批流
- 工单流
- 品牌 / 产品治理后台
- 策略版本管理

### 6.3 Phase C：查询投影与运营能力

在写入主链路稳定后补齐：

- 钱包 / 资产快照
- 报表与看板
- 投影优化
- 快照重建与定时校验

## 7. 数据存储分工

### 7.1 PostgreSQL

作为权威存储，承载：

- 资产主记录
- 状态变化历史
- 品牌 / 产品正式数据
- 协议级审计事实
- 恢复与冻结相关事实

### 7.2 Redis

作为加速层，承载：

- 钱包快照
- 列表投影
- 读缓存
- 某些计数辅助视图

约束：

- Redis 不能单独作为真源
- Redis 丢失后允许从 PostgreSQL 重建
- 快照不一致以 PostgreSQL 为准

---

## 8. 关键链路的运行顺序

### 8.1 盲扫登记

1. Factory 发起任务
2. Go 治理层创建批次 / 会话
3. Rust Core 执行盲扫登记
4. PostgreSQL 记录资产与状态
5. Go 层汇总结果

### 8.2 激活

1. Brand 发起认领 / 激活
2. Go 治理层检查审批 / 策略上下文
3. Rust Core 调用 KMS 执行密钥相关逻辑
4. 状态从 `Unassigned` 推进到 `Activated`
5. 审计事件写入

### 8.3 过户

1. Consumer 发起转移请求
2. Go 层组织交易上下文或确认流
3. Rust Core 进行角色、状态、安全前提检查
4. 状态推进到 `Transferred`
5. 快照异步更新或延迟重建

### 8.4 冻结与恢复

1. 风险系统或人工工单发起冻结
2. Rust Core 推进到 `Disputed`
3. 审核完成后恢复到前状态或进入安全终态
4. 投影与缓存同步更新

---

## 9. 一致性与恢复规则

### 9.1 真源优先

发生冲突时，优先级为：

1. 协议核心状态事实
2. PostgreSQL 持久化数据
3. Redis 快照 / 聚合投影
4. 前端缓存

### 9.2 失败恢复

系统必须支持以下恢复：

- Redis 快照重建
- CTR 校准
- PostgreSQL 故障切换
- 审计完整性校验
- 网络分区后重新校准

### 9.3 异步容忍边界

可异步的内容：

- 钱包快照
- 报表投影
- 看板聚合

不可异步确认的内容：

- 状态推进结果
- 权限裁决结果
- 核心审计事实

## 10. 开发约束

### 10.1 禁止项

- 禁止治理层绕过 Core 直接写协议状态
- 禁止前端页面状态替代后端状态机
- 禁止把缓存结果当成状态推进依据
- 禁止并行维护两套不同状态定义

### 10.2 建议项

- 所有状态推进接口统一输出 pre-state / post-state
- 所有关键写操作携带 trace id 和 idempotency key
- 所有投影读模型都保留回源重建能力
