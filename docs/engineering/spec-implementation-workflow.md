# Spec-Driven 实施工作流

> 文档类型：Engineering  
> 状态：Active  
> 权威级别：Execution Guide

---

## 1. 目标

本文件定义 RCProtocol 的 spec-driven 实施方法，用于确保需求、规格、实现、联调、测试与运维遵循统一流程，避免规则漂移。

本文件回答以下问题：

- 什么叫 spec-driven
- 需求进入项目后先改什么
- Foundation / Product / Engineering 分别承担什么规格职责
- Rust / Go / uni-app 应如何按规格实施
- 什么时候可以进入开发
- 如何把规格绑定到测试与验收

如本文件与 Foundation 层文档冲突，以 Foundation 层为准。

---

## 2. 定义

RCProtocol 中的 spec-driven 指：

- 先定义协议真相
- 再定义产品流程与页面映射
- 再定义工程实现与服务拆分
- 最后才进入代码实现与运维落地

它不是“先写很多文档”，而是：

- 先固定状态
- 先固定权限
- 先固定安全规则
- 先固定接口语义
- 再让 Rust、Go、uni-app 分别去实现这些规格

## 3. 四层规格体系

### 3.1 L1：Protocol Spec（Foundation）

目录：`docs/foundation/`

职责：

- 定义项目边界
- 定义领域对象
- 定义状态机
- 定义权限矩阵
- 定义安全模型
- 定义服务与 API 边界

当前权威文件：

- `project-overview.md`
- `domain-model.md`
- `state-machine.md`
- `roles-and-permissions.md`
- `security-model.md`
- `api-and-service-boundaries.md`

这是整个项目的最高真源。

### 3.2 L2：Product Spec（Product）

目录：`docs/product/`

职责：

- 把协议规则翻译成用户旅程
- 定义页面流程
- 定义状态展示
- 定义角色可见性
- 定义页面动作与接口动作的映射

当前权威文件：

- `product-system.md`

### 3.3 L3：Engineering Spec（Engineering）

目录：`docs/engineering/`

职责：

- 定义工程分层
- 定义技术选型
- 定义服务拆分
- 定义仓库布局
- 定义实施节奏

当前权威文件：

- `system-architecture.md`
- `technical-solution.md`
- `service-split-and-repo-layout.md`
- `spec-implementation-workflow.md`

### 3.4 L4：Ops Spec（Ops）

目录：`docs/ops/`

职责：

- 定义恢复
- 定义校准
- 定义运维执行路径
- 定义运行期修复与检查流程

这一层不重写协议规则，只执行运行期规范。

## 4. spec-driven 核心原则

### 4.1 状态先于接口

先定义状态变化，再定义动作，再定义接口。

禁止做法：

- 先写接口，再回头推导状态
- 页面流程先上线，再补协议状态

### 4.2 权限先于页面

先定义谁能做什么，再定义按钮和页面可见性。

禁止做法：

- 前端先画按钮，后端再补权限
- Go 层自己发明越权通道

### 4.3 Rust 先于 Go

先由 Rust 落协议真相，再由 Go 做治理编排与聚合。

禁止做法：

- Go 先写完整业务逻辑，Rust 被动适配
- Go 层绕过 Rust 直接写协议状态

### 4.4 后端正式状态先于前端文案

前端展示语义必须来自正式状态与正式规则映射。

禁止做法：

- 前端发明平行状态枚举
- 页面文案替代协议规则

### 4.5 文档变更必须可追踪

每次规格变更都必须能回答：

- 为什么改
- 改了哪一层
- 是否影响状态
- 是否影响权限
- 是否影响接口
- 是否影响前端流程
- 是否影响运维手册

## 5. 需求进入后的标准流程

任何需求进入后，必须先判断是否影响以下内容：

- 状态
- 权限
- 安全
- 接口语义
- 页面流程
- 运行恢复路径

### 5.1 判断结果为“否”

如果只是：

- 纯展示文案优化
- 非关键报表字段补充
- 纯样式调整
- 不影响协议语义的聚合查询调整

可以直接进入产品或工程实现。

### 5.2 判断结果为“是”

如果影响上述任一项，则必须先走规格更新。

标准顺序：

1. 更新 `foundation/`
2. 更新 `product/`
3. 更新 `engineering/`
4. 必要时更新 `ops/`
5. 再进入代码开发

---

## 6. 每一层应该更新什么

### 6.1 更新 Foundation 的触发条件

以下场景必须先更新 `foundation/`：

- 新增状态
- 修改状态迁移
- 修改角色权限
- 修改认证规则
- 修改安全前置条件
- 修改接口语义
- 修改资源边界

### 6.2 更新 Product 的触发条件

以下场景必须更新 `product/`：

- 新增页面流程
- 页面按钮对应动作变化
- 状态展示变化
- 页面可见性变化
- 错误提示逻辑变化

### 6.3 更新 Engineering 的触发条件

以下场景必须更新 `engineering/`：

- 服务拆分变化
- Rust / Go / uni-app 分工变化
- 仓库结构变化
- 数据存储分工变化
- 开发阶段变化

### 6.4 更新 Ops 的触发条件

以下场景必须更新 `ops/`：

- 恢复路径变化
- 校准流程变化
- 快照重建变化
- 运行期诊断流程变化

---

## 7. 实现顺序

规格评审通过后，代码实现顺序固定如下：

### 7.1 第一步：Rust

优先实现：

- 状态机
- 权限裁决
- 安全规则
- 核心接口
- 审计事实

### 7.2 第二步：Go

在 Rust 语义稳定后实现：

- Gateway
- IAM
- Approval
- Workorder
- Policy
- BFF
- Reporting

### 7.3 第三步：uni-app

在接口与状态语义稳定后实现：

- 页面
- 组件
- 状态展示
- 动作触发
- 错误提示

### 7.4 第四步：Ops

最后补齐：

- 恢复手册
- 巡检项
- 校准流程
- 投影修复与重建

## 8. 规格模板要求

为了让规格可执行，所有关键规格都应采用固定模板。

### 8.1 状态规格

每个状态至少包含：

- 状态名
- 定义
- 可从哪些状态进入
- 可流向哪些状态
- 允许角色
- 展示语义
- 拒绝条件

### 8.2 动作规格

每个动作至少包含：

- 动作名
- 目标资源
- 调用角色
- 前置状态
- 前置权限
- 安全前置条件
- 成功后的状态变化
- 失败错误码

### 8.3 接口规格

每个接口至少包含：

- path
- method
- 所属服务
- 请求头要求
- 请求体
- 响应体
- 幂等要求
- 审计要求
- 状态影响
- 错误码

### 8.4 页面规格

每个关键页面至少包含：

- 页面名
- 目标角色
- 数据来源
- 可见字段
- 可操作按钮
- 按钮对应接口
- 成功后的页面变化
- 失败后的提示规则

## 9. 团队分工约束

### 9.1 产品 / 协议负责人

负责：

- 需求抽象
- Foundation 规格审查
- 状态、权限、安全边界确认

### 9.2 Rust 负责人

负责：

- Protocol Spec 的实现
- 协议级错误语义
- 状态机测试
- 安全实现一致性

### 9.3 Go 负责人

负责：

- 治理编排实现
- 聚合接口实现
- 不越界写协议真相
- 审批 / 工单 / 报表落地

### 9.4 Frontend 负责人

负责：

- 按 Product Spec 实现页面
- 不自定义协议状态
- 不绕过接口动作

### 9.5 运维负责人

负责：

- 保证运行期操作与规格一致
- 确保恢复手册和当前系统一致

## 10. 评审与准入门槛

进入开发前，至少满足以下条件：

### 10.1 Foundation Ready

必须明确：

- 状态变化
- 角色权限
- 安全前提
- 核心接口语义

### 10.2 Product Ready

必须明确：

- 页面入口
- 页面动作
- 页面显示状态
- 角色可见性

### 10.3 Engineering Ready

必须明确：

- 由 Rust 还是 Go 实现
- 存储落位
- 是否需要缓存 / 投影 / worker
- 是否影响现有服务拆分

缺少以上任一项，不应直接进入编码。

---

## 11. 测试与验收绑定规则

spec-driven 必须绑定到测试，而不是只停留在文档层。

### 11.1 Rust 测试

至少覆盖：

- 状态迁移测试
- 权限测试
- 安全前置条件测试
- 核心错误语义测试

### 11.2 Go 测试

至少覆盖：

- 审批流与工单流编排测试
- 幂等测试
- 错误映射测试
- 聚合响应测试

### 11.3 Frontend 测试

至少覆盖：

- 页面动作与接口映射测试
- 状态展示测试
- 禁用动作测试
- 错误提示测试

### 11.4 验收结论

验收必须回答：

- 是否实现了规格中的状态变化
- 是否遵守了权限边界
- 是否没有绕过 Rust 核心
- 是否前端展示与正式状态一致

---

## 12. 项目级执行节奏

推荐每个功能都按以下节奏推进：

```text
需求提出
  -> 规格影响判断
  -> 更新 Foundation
  -> 更新 Product
  -> 更新 Engineering
  -> Rust 实现
  -> Go 实现
  -> uni-app 实现
  -> 测试与联调
  -> 更新 Ops
```

## 14. 当前结论

RCProtocol 的 spec-driven 实施方法可以归纳为：

- 用 `foundation/` 固定协议真相
- 用 `product/` 固定页面和流程映射
- 用 `engineering/` 固定实现路径与拆分方式
- 用 `ops/` 固定运行与恢复方法
- 用 Rust、Go、uni-app 按顺序落实规格，而不是反向定义规格

这是当前项目避免规则漂移、接口失真、前后端不一致和治理越界的最佳实施方式。
