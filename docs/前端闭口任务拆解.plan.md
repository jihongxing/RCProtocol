## 3. 闭口原则

1. 页面层禁止新增临时直拼业务路径
2. typed API 是唯一正式业务入口
3. 错误模型统一收敛到：
   - `AUTH_REQUIRED`
   - `FORBIDDEN`
   - `ORG_SELECTION_REQUIRED`
   - `NOT_FOUND`
   - `CONFLICT`
   - `UNPROCESSABLE`
   - `NETWORK_ERROR`
4. 页面必须具备最小可感知反馈，不允许“点了没反应”
5. 每完成一个任务块，必须同步更新：
   - 页面实现
   - typed API
   - 共享类型
   - 最小测试或联调记录

- `frontend/packages/api/src/adapters.ts`
- `frontend/packages/api/src/interceptors.ts`

**任务**：
1. 统一 web / uni 端 request 抽象
2. 统一 token 注入与 `401` 认证失效处理
3. 统一 trace id 生成
4. 统一后端错误体解析与 error code 映射
5. 统一 `unwrapData` 行为

**结果**：
- 已形成 `createRequest + adapter + interceptors` 的统一入口
- 已统一 `X-Trace-Id` 与基础错误处理
- 已统一 `handleErrorResponse / unwrapData`

**验收**：页面不再关心底层 transport 与 unwrap 细节。✅

### A2. Auth / Console / App / Transfer typed API 收口
**涉及文件**：
- `frontend/packages/api/src/authApi.ts`
- `frontend/packages/api/src/consoleApi.ts`
- `frontend/packages/api/src/appApi.ts`
- `frontend/packages/api/src/transferApi.ts`
- `frontend/packages/api/src/index.ts`

**任务**：
1. 拆分按领域的 typed API
2. 页面层统一经 `useTypedApi()` 获取调用能力
3. 禁止页面直接拼接业务路径
4. 统一写操作 header 透传能力（如 `X-Idempotency-Key`）

**结果**：
- B / C 端页面主要业务能力已迁移到 typed client
- transfer 相关能力已独立到 `transferApi`

**验收**：typed API 成为页面唯一正式 API 入口。✅

### A3. 登录态恢复与全局认证失败处理
**涉及文件**：
- `frontend/packages/state/src/useAuth.ts`
- `frontend/packages/state/src/storage.ts`
- `frontend/apps/b-console/src/composables/useTypedApi.ts`
- `frontend/apps/c-app/src/composables/useTypedApi.ts`

**任务**：

**结果**：
