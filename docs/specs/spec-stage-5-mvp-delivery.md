# Spec：Stage 5 MVP 可交付闭环

> 文档类型：Spec  
> 状态：Draft  
> 权威级别：Implementation Proposal  
> 最后更新：2026-04-09

---

## 1. 目标

本 Spec 定义 RCProtocol 在当前阶段应如何完成 **Stage 5：集成联调与 MVP 验收**，使系统从“功能基本存在”升级为“可稳定演示、可对外接入、可部署交付”的 MVP。

本 Spec 的核心目标不是继续扩展协议终局能力，而是优先完成以下闭环：

1. 全链路主流程可重复跑通
2. 品牌 API 接入链路可独立验证
3. 异常流、回归基线与部署基线清晰
4. `/verify` 与 `/verify/v2` 能并行服务当前 MVP 与协议演进演示
5. MVP 具备技术验收、产品验收与演示交付能力

---

## 2. 背景

根据当前 `roadmap.md`，RCProtocol 已完成或基本完成：

- Stage 1：协议与密码学基础
- Stage 2：Rust 核心业务后端
- Stage 3：Go 边车与双通路服务化改造（部分完成）
- Stage 4：uni-app 前端双端上线（部分完成）

同时，协议演进方向上已形成：

- `AssetCommitment`
- `Brand Attestation`
- `Platform Attestation`
- `Verification V2`

当前主要问题不再是“是否有方向”，而是：

> 系统已经具备较完整的协议型实现雏形，但尚未形成稳定、可回归、可部署、可试点的 MVP 交付闭环。

因此 Stage 5 必须优先解决 **联调、回归、部署、性能、验收** 五类问题，而不是继续扩大协议设计面。

---

## 3. Stage 5 的定义

Stage 5 的目标是让 RCProtocol 达到以下状态：

### 3.1 可交付

- 项目能在约定环境中稳定启动
- 核心依赖可初始化
- 基础联调脚本可直接运行
- 演示不依赖大量临时人工补数据

### 3.2 可回归

- 核心主链路有固定测试入口
- 异常流有最小覆盖矩阵
- 新改动后能快速判断是否破坏 MVP 主流程

### 3.3 可对接

- 品牌方可通过 API Key 调用关键接口
- 平台托管通路与品牌开放接入通路至少各有一条最小闭环
- 前后端、Gateway/BFF、Rust API 的接口语义稳定

### 3.4 可验收

- 技术侧知道验收什么
- 产品侧知道演示什么
- 商业侧知道对外试点能交付什么

---

## 4. 非目标

本 Spec 当前不包含：

- Stage 7~10 的终局协议定义继续扩展
- 双边共同信任根的进一步密码学升级
- 数据库降级为投影层的架构重构
- 完整生产级高可用部署体系
- 全量前端体验打磨与视觉优化

换言之，Stage 5 的重点是：

> 先把“当前 MVP 形态”做成一个可交付产品，而不是把“终局协议形态”继续前推。

---

## 5. MVP 验收对象

Stage 5 必须围绕以下五类对象收口。

### 5.1 主链路闭环

必须至少稳定打通一条黄金主链路：

1. 盲扫
2. 激活（生成资产承诺、品牌声明、平台声明）
3. 激活绑定（生成虚拟母卡并建立母子绑定）
4. 售出
5. 验真
6. 入馆 / 查看资产详情
7. 发起过户
8. 确认 / 拒绝过户

### 5.2 品牌接入闭环

必须至少稳定打通一条品牌侧开放接入链路：

1. 平台注册品牌
2. 签发 API Key
3. 品牌通过 API Key 调用核心接口
4. 品牌完成至少一次资产激活或验真调用
5. 品牌能够查询品牌自身相关数据

### 5.3 协议演进兼容闭环

必须验证当前 MVP 与协议演进能力可以并存：

- `/verify` 继续承担 V1 当前 MVP 验真语义
- `/verify/v2` 可用于展示承诺状态与协议升级能力
- V2 不破坏当前 C 端或品牌接入主流程

### 5.4 异常流闭环

必须覆盖以下异常类型：

- 激活前置状态不合法
- 验真认证失败
- replay suspected
- 承诺缺失 / 不完整
- 售出冲突
- 过户冲突
- 冻结 / 恢复后的受限行为
- API Key 无效或已撤销

### 5.5 环境与交付闭环

必须明确：

- 如何启动依赖
- 如何初始化数据库
- 如何运行服务
- 如何执行脚本与测试
- 如何验证部署环境达到可演示状态

---

## 6. 核心验收原则

### 6.1 主链路优先于扩功能

如果一个新增能力不能提高 Stage 5 验收通过率，则不应优先进入当前阶段。

### 6.2 回归入口优先于手工演示

任何可以手工演示但无法稳定重复执行的链路，都不能视为 Stage 5 完成。

### 6.3 双接口并存优先于默认切换

在 MVP 阶段：

- 继续保留 `/verify`
- 将 `/verify/v2` 作为灰度联调与协议演进展示入口
- 不要求当前阶段默认切换到 V2

### 6.4 品牌开放接入与平台托管运营并行验证

Stage 5 不要求双通路都做到完整产品化，但必须至少证明：

- 平台托管运营通路可完成主链路演示
- 品牌开放接入通路可完成最小 API 闭环

### 6.5 以验收清单作为阶段完成标准

Stage 5 是否完成，不以“代码很多”作为判断，而以是否满足正式验收清单作为判断。

---

## 7. 交付范围

### 7.1 联调范围

至少包含以下模块：

- `rc-api`
- `go-gateway`
- `go-bff`（如当前主链路依赖）
- 前端 B 端 / C 端最小页面
- PostgreSQL
- Redis（若当前视图、缓存或脚本已依赖）

### 7.2 测试范围

至少包含以下测试类型：

- 核心单元测试
- 核心集成测试
- 端到端联调脚本
- 异常流回归脚本

### 7.3 部署范围

至少形成以下基线：

- Docker Compose 或单机启动方案可运行
- 必需环境变量有清单
- 数据初始化步骤明确
- 常见启动失败点可排查

---

## 8. 黄金主链路定义

推荐以如下顺序作为 Stage 5 标准主链路：

### 8.1 链路 A：平台托管运营通路

1. 平台登录后台
2. 注册品牌
3. 导入或准备可激活资产
4. 完成盲扫 / 激活（生成资产承诺、品牌/平台声明，进入 `RotatingKeys`）
5. 完成激活绑定（生成虚拟母卡并建立母子绑定，进入 `EntangledPending`）
6. 完成售出
7. C 端执行验真
8. C 端查看资产馆 / 资产详情
9. 发起过户
10. 确认过户
11. 查看最终状态与审计记录

### 8.2 链路 B：品牌开放接入通路

1. 平台注册品牌并签发 API Key
2. 品牌持 API Key 调用激活或查询接口
3. 品牌调用 `/verify` 或相关查询接口确认结果
4. 平台侧与品牌侧看到一致状态

### 8.3 链路 C：协议演进演示通路

1. 使用同一资产跑通 `/verify`
2. 再跑 `/verify/v2`
3. 对比：标签真实性、承诺完整性、协议状态、最终结论

---

## 9. 错误与异常语义要求

Stage 5 期间，所有核心链路必须尽量满足以下要求：

### 9.1 错误可区分

不能继续出现“所有错误都表现为泛化失败”的情况，至少应区分：

- 认证失败
- 权限失败
- 状态冲突
- 数据缺失
- 协议承诺不完整
- 环境依赖未就绪

### 9.2 错误可追踪

关键失败场景必须能通过以下至少一项快速定位：

- 审计事件
- 结构化日志
- Trace ID
- 联调脚本输出

### 9.3 错误可复现

对 P0 主链路中的关键失败场景，应能通过固定脚本或固定测试数据重放。

---

## 10. 性能与稳定性基线

Stage 5 不要求做到最终生产指标，但至少应建立一版可被记录和复测的基线。

### 10.1 推荐基线

- 验真接口 P95 ≤ 800ms
- 激活写入成功率 ≥ 99%
- 主链路脚本连续运行时无明显随机失败

### 10.2 基线要求

必须记录：

- 测试环境配置
- 测试数据规模
- 测试方法
- 结果摘要
- 当前瓶颈点

---

## 11. 部署与运行要求

### 11.1 最小运行环境

至少应能在一套标准开发 / 演示环境中完成：

- PostgreSQL 启动
- Redis 启动
- 核心服务启动
- 必需 migration / 初始化完成
- 联调脚本可执行

### 11.2 运行文档要求

必须明确：

- 环境变量模板
- 启动顺序
- 初始化命令
- 常见报错处理
- 如何验证“系统已可演示”

---

## 12. Stage 5 输出物

Stage 5 完成后，至少应交付以下输出：

1. 一份正式任务拆解文档
2. 一份黄金主链路验收清单
3. 一组联调 / 回归脚本
4. 一份最小部署与运行说明
5. 一份性能基线记录
6. 一份 MVP 验收结论

---

## 13. DoD（Definition of Done）

Stage 5 满足以下条件后，方可判定“可交付 MVP 已形成”：

- [ ] 黄金主链路可在约定环境中重复跑通
- [ ] 品牌 API 接入闭环可重复验证
- [ ] `/verify` 与 `/verify/v2` 可并行验证且语义清晰
- [ ] 核心异常流有最小覆盖矩阵
- [ ] 部署方案可在单机 / Compose 环境拉起
- [ ] 至少有一版性能基线结果
- [ ] 技术、产品、演示三类验收清单明确

---

## 14. 推荐实施顺序

推荐按以下顺序推进：

1. 环境与依赖初始化基线
2. 黄金主链路联调
3. 品牌 API 接入联调
4. 异常流补测
5. 部署基线固定
6. 性能基线测试
7. MVP 验收与结论归档

不建议顺序：

- 继续先扩协议终局设计，再回头补联调
- 先做大规模 UI 改造，再补核心测试
- 先切换 `/verify/v2` 为默认接口，再验证现有前端兼容性
