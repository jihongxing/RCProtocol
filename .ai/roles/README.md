# RCProtocol 文档使用说明：角色体系改造版

本目录下的角色文档已经不再是通用“一人公司”模板，而是针对 `RCProtocol` 当前项目基线重写后的项目专用规则。

## 使用原则

1. 先读 `基础宪章.md`
2. 再根据任务类型读取对应角色文档
3. 若角色文档与当前 `docs/` 权威文档冲突，以 `docs/` 当前基线事实为准
4. 若角色文档与历史归档叙事冲突，忽略归档叙事

## 角色文档改造方向

本次改造统一围绕以下事实：

- 项目是高价值实物资产数字主权协议，不是泛化 SaaS 模板
- 当前基线围绕 `NTAG 424 DNA`、14 态状态机、5 角色权限、三级密钥派生链展开
- 当前 MVP 目标是拿下品牌试点，而不是做完整终局平台
- 工程基线是 `Rust Core + Go Governance + PostgreSQL + Redis`

## 当前适用场景

- 需求裁剪
- MVP 产品定义
- 架构设计与实现约束
- 全栈字段 / 接口 / 页面打通
- 测试、审计、运维、安全、数据和增长协作
- 协议实现落地与规格映射
- B 端治理后台产品拆解
- NFC / 标签认证与近场安全分析
- 文档分层、SSOT 维护与归档判断

## 新增的项目专用角色

- `协议实现官宪法.md`：负责把 Foundation 规格稳定落到 Rust 协议实现
- `B端治理产品官宪法.md`：负责平台 / 品牌 / 工厂 / 审核员治理后台产品化
- `NFC-标签安全官宪法.md`：负责 `NTAG 424 DNA`、认证链路、读卡器与标签安全
- `文档总编-SSOT守门员宪法.md`：负责文档分层、单一定义与归档边界

## 不再适用的旧倾向

以下旧模板倾向已被弱化或移除：

- 泛创业项目视角
- 与项目无关的夸张口号
- 不贴当前协议基线的抽象工程要求
- 把历史归档内容当现行规范
